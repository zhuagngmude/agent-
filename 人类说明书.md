# agent蜂群 人类说明书

## 1. 这是什么

agent蜂群是一个多 AI 智能体协作开发平台。

它的核心作用是把“我想做一个项目”变成一套可以由多个 AI 分工完成的协作流程。

它不是普通聊天工具，也不是单个 AI 写代码工具，而是一个项目指挥台：

```text
项目想法
→ 架构师 Agent 出总体方案
→ 调度器 Agent 拆任务
→ 前端 / 后端 / 测试 / 文档 Agent 分头处理
→ 审查 Agent 汇总检查
→ 用户确认
→ 本地 Runner 安全改代码和跑测试
```

## 2. 这个项目解决什么问题

单个 AI 做复杂项目时，容易出现这些问题：

- 一次上下文太长，容易忘事。
- 架构、代码、测试、文档混在一起，输出不稳定。
- 多个 AI 一起做时，没有任务分配和状态管理。
- 不知道哪个 AI 做了什么，也不知道为什么这么改。
- 如果多个 AI 同时改代码，可能互相覆盖或误删。

agent蜂群要解决的是：

- 自动拆分任务。
- 自动分配不同 AI 角色。
- 保存每个 Agent 的输出和历史。
- 支持团队协作和权限控制。
- 后续支持用户确认后的自动代码修改。
- 防止多个 AI 乱改同一份代码。

## 3. 第一版产品形态

第一版不是纯本地网页，也不是纯云端平台，而是：

```text
云端 Web App + 本地 Python Runner
```

云端 Web App 负责：

- 登录
- 团队协作
- 项目管理
- Agent 配置
- 任务看板
- 多模型调用
- 历史记录
- 权限系统

本地 Python Runner 负责：

- 连接云端任务
- 读取本地代码仓库
- 展示将要修改的文件
- 等用户确认
- 创建 Git checkpoint
- 修改代码
- 跑测试
- 上传执行结果和日志

## 4. 谁会用

第一版主要给：

```text
你自己 + 少量朋友 / 团队成员
```

第一版不做公开平台，不做任何人都能注册的 SaaS。

推荐采用邀请制。

## 5. 核心用户流程

### 5.1 创建项目

用户创建一个项目，填写：

- 项目名称
- 项目想法
- 技术偏好
- 约束条件
- 是否允许子 Agent
- 是否允许代码执行

### 5.2 自动生成方案

架构师 Agent 自动调用模型，生成：

- 产品目标
- 技术栈建议
- 页面和模块
- 数据模型
- MVP 范围
- 风险和不做事项

### 5.3 自动拆分任务

调度器 Agent 根据架构方案拆出任务：

- 前端任务
- 后端任务
- 测试任务
- 文档任务
- 审查任务
- 执行任务

### 5.4 多 Agent 分头执行

不同 Agent 调用不同模型执行任务。

例如：

```text
架构师 Agent：GPT
前端 Agent：Claude
测试 Agent：Gemini
审查 Agent：GPT
```

### 5.5 子 Agent

主 Agent 可以创建受控子 Agent。

例如：

```text
前端 Agent
→ UI 子 Agent
→ 移动端适配子 Agent
→ 交互检查子 Agent
```

第一版限制：

- 最多 2 层：主 Agent → 子 Agent。
- 每个主 Agent 最多 3 个子 Agent。
- 子 Agent 不能继续创建子 Agent。
- 子 Agent 输出必须回到父 Agent 汇总。

### 5.6 用户确认后改代码

第一版允许主执行 Agent 改代码，但必须先让用户确认。

流程：

```text
Agent 生成修改计划
→ 系统展示将修改哪些文件
→ 用户确认
→ 本地 Runner 创建 Git checkpoint
→ Runner 修改代码
→ Runner 跑测试
→ 审查 Agent 检查结果
```

第一版不允许多个 Agent 同时改代码。

## 6. 权限系统

项目采用高级权限系统。

权限可以细到：

- 谁能看项目
- 谁能运行 Agent
- 谁能创建子 Agent
- 谁能管理 API Key
- 谁能导出项目
- 谁能审查结果
- 谁能确认代码执行

第一版提供角色模板：

- 项目所有者
- 项目管理员
- 开发者
- 审查者
- 只读成员
- 自定义权限

高风险权限默认关闭：

- 管理 API Key
- 自动执行代码
- 删除项目
- 导出项目
- 管理权限

## 7. API Key 管理

支持两种方式：

- 平台默认 Key
- 用户自己的 Key

规则：

- API Key 必须加密保存。
- API Key 不写进项目日志。
- 导出项目时不包含 API Key。
- 调用失败时不能显示完整 Key。

## 8. 第一版必须做

- 登录和邀请制团队使用
- 项目管理
- Agent 配置
- 自动多 Agent 流程
- 受控子 Agent
- 任务看板
- 完整历史记录
- 高级权限系统
- API Key 加密管理
- Python Runner
- 用户确认后自动改代码
- Git checkpoint
- 审查 Agent 汇总结果

## 9. 第一版暂时不做

- 公开注册
- 计费系统
- 无限子 Agent
- 多个 Agent 同时改代码
- 完全无人确认的自动写文件
- 移动 App
- 复杂插件市场

## 10. 后续升级方向

- 多 Agent 并行改代码
- 文件所有权
- 任务锁
- Git 分支隔离
- 自动合并前审查
- 成本统计和预算限制
- 项目模板市场
- Agent 能力市场
- 更完整的审计日志

## 11. UI 设计原则

旧的 4 套 UI 静态原型已删除，因为用户反馈“不好看”。

下一轮 UI 不直接凭空画页面，先做设计上下文确认：

- 是否有参考产品或截图。
- 是否有喜欢的设计风格。
- 是否有不喜欢的风格。
- 是否要偏工作台、IDE、流程编排，还是团队管理平台。
- 是否使用成熟设计系统作为起点。

下一轮 UI 必须避免：

- 只堆卡片。
- 假数据装饰。
- 装饰性图标滥用。
- 紫蓝渐变、霓虹科技感、通用 AI 味界面。
- 没有真实使用路径的演示页面。

推荐方向：

- 主界面应该像专业工作台，而不是宣传页。
- 信息密度要高，但层级要清楚。
- Runner 和代码执行页可以参考 IDE/审查工具。
- Agent 流程页要清楚表达状态、依赖、成本和权限。

当前已创建新的 UI 方向稿：

```text
design/index.html
```

方向稿包含：

- Linear 气质：执行队列和任务状态。
- Notion 气质：项目文档和运行笔记。
- Raycast 气质：命令中心和快捷操作。
- GitHub 气质：Issue、检查项和审查流程。
- Vercel 气质：部署流水线和运行日志。
- Cursor 气质：Agent 输出、文件和代码计划。
- 飞书气质：团队、权限和协作。
- IDE 气质：Runner 执行确认和 diff。

待用户选择方向后，再进入下一轮细化。

## 12. 怎么判断这个项目做好了

第一版完成时，应该能做到：

- 用户能登录。
- 用户能创建项目。
- 用户能配置多个 Agent 和模型。
- 系统能自动跑完整 Agent 流程。
- 系统能保存每个 Agent 输出。
- 用户能看到任务状态和历史。
- 用户能确认代码修改。
- 本地 Runner 能安全修改代码和跑测试。
- 系统能记录 Git checkpoint 和执行日志。
- 审查 Agent 能给出总结。
