项目已经完成：

2. 文档体系确认：人类说明书 + AI 开发维护手册。

## 下一阶段目标

下一阶段目标不是直接把所有功能做完，而是把前端页面变成“可接数据、可接后端、可接 Runner”的工程雏形。

阶段名称：

```text
MVP-0.2：前端工程化 + Mock 状态机 + Runner 审批原型
```

## 推荐开发顺序

### 1. 修前端编码和文案

优先级：最高

状态：进行中，已完成第一轮清理。

当前前端文件中出现中文乱码，需要先统一处理。

任务：

- 将 `frontend/index.html` 保存为 UTF-8。
- 将 `frontend/README.md` 保存为 UTF-8。
- 检查 `frontend/app.js`、`frontend/styles.css` 是否存在乱码。
- 把页面示例项目改回 `agent蜂群 MVP`。
- 减少装饰性 emoji 和模板后台感文案。
- 保留中文 UI 文案，但代码变量、文件名、路径尽量英文 ASCII。

完成标准：

- 直接打开 `frontend/index.html`，页面中文全部正常。
- 导航、标题、表格、按钮、卡片说明都能读懂。
- 不出现伪中文、乱码、异常符号。

已处理：

- `frontend/app.js` 已替换为干净导航脚本。
- 新增 `frontend/data.js`，作为导航标题和后续 mock 数据入口。
- `frontend/index.html` 已移除内联脚本，改为加载 `data.js` 和 `app.js`。
- 首页示例项目已从通用模板项目收敛到 `agent蜂群 MVP`。
- 审批卡片已避免使用“批量审批”暗示，改为逐项审查。
- API Key 展示已改为加密/脱敏说明，不展示尾号。

### 2. 固定 12 个一级模块

优先级：高

当前前端已经形成 12 个一级模块，下一步先固定，不再频繁增删。

固定模块：

```text
overview
agents
tasks
workflow
knowledge
code
runtime
approval
monitor
billing
integrations
settings
```

完成标准：

- 每个模块都有稳定 id。
- 导航和页面标题使用同一份配置。
- 后续数据模型和 API 按这些模块映射。

### 3. 抽出前端 Mock 数据模型

优先级：高

不要继续把所有数据写死在 HTML 里。

建议新增：

```text
frontend/data.js
```

先定义这些 mock 对象：

```text
project
dashboardMetrics
agents
tasks
workflows
approvalRequests
gitCheckpoints
knowledgeUpdates
usageSummary
integrations
settings
```

完成标准：

- 页面渲染从 mock 数据读取。
- 不同页面复用同一份数据。
- 后续接 API 时只替换数据来源，不重写 UI。

### 4. 建立核心状态机

优先级：高

先在前端 mock 层模拟状态，不急着接数据库。

至少实现：

```text
TaskStatus
AgentStatus
WorkflowRunStatus
ApprovalStatus
RunnerJobStatus
```

重点是 `ApprovalStatus`：

```text
draft
pending
approved
rejected
patch_only
expired
executed
rolled_back
```

完成标准：

- 审批列表能区分待审批、已拒绝、只生成补丁、已执行。
- 高风险审批不能一键批量通过。
- Runner 执行前必须先有 Git checkpoint 状态。

### 5. 打磨审批与确认页面

优先级：最高

这是本系统能不能让人放心使用的关键页面。

必须展示：

- 申请来源 Agent。
- 操作类型。
- 影响文件。
- 修改原因。
- 风险等级。
- diff 预览。
- Git checkpoint。
- 审批按钮。
- 二次确认。
- 拒绝原因。
- 只生成补丁。

禁止：

- 高风险操作默认批量批准。
- 把“允许执行”做成普通主按钮。
- 隐藏完整影响范围。

完成标准：

- 用户不需要问 AI，也能知道这次执行会发生什么。
- 用户可以拒绝、查看差异、只生成补丁、允许执行。
- 高风险允许执行必须二次确认。

### 6. 写 API 草案

优先级：中

前端 mock 稳定后，写 API 草案，不急着实现。

建议新增：

```text
docs/api-draft.md
```

至少包括：

```text
GET /api/projects/:projectId/dashboard
GET /api/projects/:projectId/agents
GET /api/projects/:projectId/tasks
GET /api/projects/:projectId/workflows
GET /api/projects/:projectId/approvals
POST /api/approvals/:approvalId/approve
POST /api/approvals/:approvalId/reject
POST /api/approvals/:approvalId/patch-only
GET /api/projects/:projectId/git/checkpoints
GET /api/projects/:projectId/usage
```

完成标准：

- 前端每个模块都能找到对应 API。
- Approval 和 Runner 分开。
- Dashboard 使用聚合接口。

### 7. 再决定技术栈落地

优先级：中

等前端数据结构和 API 草案稳定后，再开始真实工程。

候选方案：

```text
Next.js + Supabase PostgreSQL + Local Python Runner
```

不要现在就直接上复杂后端，因为前端模块边界还在稳定。
