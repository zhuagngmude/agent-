# 前端交互反推架构调整

日期：2026-06-08

参考前端：

- `frontend/index.html`
- `frontend/styles.css`
- `design/image2/swarm-command-overview-20260608-182854.png`
- `design/image2/swarm-light-admin-vuexy-20260608-183631.png`

## 结论

当前前端已经从最初的 8 页原型，扩展为更完整的 12 个模块控制台：

1. 项目概览
2. 智能体管理
3. 任务管理
4. 工作流编排
5. 知识库管理
6. 代码与知识
7. 运行与调度
8. 审批与确认
9. 监控与日志
10. 费用与用量
11. 集成与扩展
12. 系统设置

因此系统架构需要从“多 Agent 任务调度工具”升级为：

```text
项目控制台 + Agent 调度中台 + 本地 Runner 安全网关 + 知识库与审计系统
```

前端已经体现出更像“AI 项目操作系统”的形态，不再只是任务拆解器。

## 架构需要调整的地方

### 1. 前端导航决定后端模块边界

后端不应只按 `Project / Agent / Task / Runner` 四个大对象粗分。

建议拆成以下业务服务：

```text
Project Service
Agent Service
Task Service
Workflow Service
Knowledge Service
Code Context Service
Runner Service
Approval Service
Audit Log Service
Usage Billing Service
Integration Service
Settings Service
```

其中 `Runner Service` 和 `Approval Service` 必须拆开：

- Runner Service 负责本地执行能力。
- Approval Service 负责审批流、风险等级、确认记录、二次确认。

这样可以避免“能执行代码的模块自己决定自己能不能执行”，安全边界更清楚。

### 2. 项目概览页需要一个聚合接口

前端首页展示了：

- 当前项目状态
- 活跃智能体数
- 待确认事项
- 活跃任务
- Git 检查点
- Token 消耗
- 模型使用
- Runner 待审批
- 任务队列
- Agent 状态
- 知识库更新
- API Key 状态

这些不应该由前端分别请求十几个接口后自己拼。

建议新增：

```text
GET /api/projects/:projectId/dashboard
```

返回一个聚合视图：

```text
project
metrics
workflow_summary
pending_approvals
task_queue
agent_status
git_checkpoints
knowledge_updates
usage_summary
integration_health
```

这个接口只服务首页，不等同于底层数据模型。

### 3. 智能体管理需要从“角色列表”升级为“Agent 资源池”

前端已经有“智能体资源池”“模型分配”“版本号”“运行状态”。

架构上 Agent 不再只是一个固定角色，而应该是可配置资源：

```text
AgentTemplate
AgentInstance
AgentCapability
AgentModelBinding
AgentRuntimeState
SubAgentPolicy
```

关键字段：

- Agent 名称
- 角色类型
- 能力范围
- 使用模型
- 版本
- 当前状态
- 权限范围
- 是否允许创建子 Agent
- 最大子 Agent 数
- 父子 Agent 关系

### 4. 任务管理需要支持队列和状态机

前端出现了：

- 进行中
- 排队中
- 已完成
- 待确认
- 高优先级
- 剩余时间

因此 Task 不能只是普通待办，应该有状态机：

```text
created
planned
queued
running
blocked
waiting_user
waiting_review
completed
failed
cancelled
archived
```

并且需要任务调度字段：

```text
priority
assigned_agent_id
depends_on_task_ids
risk_level
estimated_duration
deadline
related_files
approval_required
```

### 5. 工作流编排需要单独建模

前端有“流程视图 / 拓扑视图 / 依赖视图”，这说明工作流不能只是任务列表。

建议新增：

```text
Workflow
WorkflowNode
WorkflowEdge
WorkflowRun
WorkflowRunStep
```

每个节点可以绑定：

- Agent
- Task
- Approval
- Runner action
- Knowledge update
- Git checkpoint

这样后续才能做流程回放、失败重试、节点暂停、人工确认。

### 6. 知识库管理需要支持文档片段

前端展示“新增多少文档片段 / 更新多少文档片段”，说明知识库不能只保存整篇文档。

建议模型：

```text
KnowledgeDocument
KnowledgeSection
KnowledgeChunk
KnowledgeIndexJob
KnowledgeLink
DocSyncStatus
```

人类说明书和 AI 开发维护手册要能互相关联：

```text
human_doc_section_id
ai_doc_section_id
feature_id
last_synced_at
sync_status
```

### 7. 代码与知识需要一个 Code Context Service

前端单独出现“代码与知识”，说明系统需要把代码文件、文档、任务、Agent 输出连起来。

建议新增：

```text
CodeRepository
CodeFile
CodeSymbol
CodeContextIndex
FileOwnership
ChangePlan
```

后续自动改代码时，Agent 不能只拿一段聊天上下文，而要拿：

- 相关文件
- 相关文档
- 相关任务
- 最近 Git checkpoint
- 文件所有权
- 允许修改范围

### 8. 运行与调度需要 Runtime Queue

前端有运行队列、调度成功率、平均响应。

建议新增：

```text
RuntimeQueue
RuntimeJob
SchedulerDecision
ModelCall
RetryPolicy
```

用于记录：

- 哪个 Agent 在跑
- 调用了哪个模型
- 花了多少 token
- 是否失败重试
- 是否被用户暂停
- 是否等待 Runner 审批

### 9. 审批与确认必须成为核心模块

前端有独立的“审批与确认”，这是正确方向。

建议模型：

```text
ApprovalRequest
ApprovalAction
RiskAssessment
DiffPreview
ExecutionPlan
UserDecision
```

审批类型：

```text
file_read
file_write
file_delete
command_execute
network_request
git_operation
model_key_access
knowledge_publish
```

审批状态：

```text
draft
pending
approved
rejected
patch_only
expired
executed
rolled_back
```

高风险操作必须二次确认。

### 10. 监控与日志需要从第一版就留接口

前端有 CPU、内存、网络、日志入口。

第一版不一定要真实监控所有系统指标，但架构要预留：

```text
RunnerHeartbeat
SystemMetric
AgentLog
AuditEvent
ErrorEvent
```

日志要分级：

- 用户可见日志
- AI 执行日志
- Runner 本地日志
- 审计日志
- 脱敏后的模型调用日志

### 11. 费用与用量需要独立统计

前端有 Token 消耗、今日成本、模型使用。

建议新增：

```text
UsageRecord
CostEstimate
ModelPriceSnapshot
BudgetLimit
```

统计维度：

- workspace
- project
- agent
- task
- workflow_run
- model_provider
- model_name

注意：费用统计不能记录完整 API Key，日志必须脱敏。

### 12. 集成与扩展需要插件边界

前端已有 GitHub、本地 Runner 等集成入口。

建议模型：

```text
IntegrationProvider
IntegrationConnection
IntegrationSecret
WebhookEvent
ExtensionManifest
```

第一版优先支持：

- Local Runner
- Git
- GitHub
- OpenAI / Claude / Gemini API

其他集成后续再做。

## 第一版开发顺序调整

根据前端交互，后续不要一次性做完整系统。

建议顺序：

1. 修复前端中文编码和页面结构。
2. 固定前端导航模块，不再频繁增删一级菜单。
3. 抽象 mock 数据模型。
4. 实现 Dashboard 聚合数据结构。
5. 实现 Agent / Task / Workflow 的本地 mock 状态机。
6. 重点实现 Approval / Runner 安全确认静态流程。
7. 再接本地 JSON 或 SQLite。
8. 再接云端数据库。
9. 最后接真实多模型调用和本地 Runner 自动执行。

## 需要修改的前端交互建议

### 保留

- 12 个模块的大导航方向可以保留。
- 首页作为控制台聚合视图可以保留。
- Runner 待审批卡片必须保留。
- Git 检查点、知识库更新、费用用量可以保留。

### 收敛

- 首页指标不要太像通用后台模板，要更贴近 `agent蜂群`。
- “智能客服系统升级项目”这类示例应改回 `agent蜂群 MVP` 或可配置项目。
- 减少装饰性 emoji 和伪图标，避免显得像模板后台。
- 审批按钮不能有“批量批准高风险操作”的默认暗示。
- API Key 卡片不要展示类似真实 key 的尾号，统一展示脱敏状态。

### 必须修

- 当前前端文件存在明显中文编码异常，需要先统一为 UTF-8。
- 导航中文、标题中文、表格中文必须恢复为真实文本。
- 与代码/脚本相关的路径尽量使用英文 ASCII。
- 中文可用于 UI 文案和说明正文。
