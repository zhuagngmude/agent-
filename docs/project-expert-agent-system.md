# 项目专家 Agent 系统设计

## 定位

`agent-swarm` 不应该只是一组固定 AI 员工。它应该支持“核心员工 + 项目专家 Agent”的动态队伍：

- 核心员工负责大多数项目的基础交付链路。
- 项目专家 Agent 根据项目类型、技术栈、风险和任务阶段临时加入。
- 专家 Agent 可以参与写代码、审查、测试、文档、部署方案等工作，但必须进入项目任务链路和安全边界。

这是一项核心能力，后续 UI、任务拆解、Runner、审批和产物归档都要围绕它设计。

## 目标

1. 让普通项目默认可由核心员工完成。
2. 让复杂项目自动补充合适专家，而不是让用户手动想角色。
3. 让专家 Agent 真的进入项目执行：能被分配任务、产生产物、参与代码改动和验收。
4. 让所有专家行为可追踪：任务、输入、输出、文件改动、运行记录、审批记录都归属当前项目。
5. 保持安全：专家 Agent 不能绕过 Runner、审批、保护路径、密钥保护和高风险确认。

## 核心员工池

核心员工是所有项目默认可用的稳定角色。

| 角色 | 主要职责 |
| --- | --- |
| 总控 Agent | 理解目标、识别项目类型、拆解任务、调度员工、汇总进度 |
| 架构师 Agent | 技术路线、模块边界、数据流、风险设计 |
| 前端 Agent | 页面、组件、交互、前端状态和样式 |
| 后端 Agent | API、数据库、服务逻辑、权限和集成 |
| 文档 Agent | 使用说明、接口文档、维护文档、交付说明 |
| 质量保证 Agent | 测试计划、验收、回归检查、问题复现 |
| 审查 Agent | 安全、权限、改动范围、发布前风险复核 |

这些角色能覆盖大部分 Web App、后台系统、本地工具、AI 工作流工具和常见自动化项目。

## 项目专家 Agent

项目专家 Agent 是按项目需要加入的临时或半固定角色。

常见专家包括：

| 专家 | 触发场景 |
| --- | --- |
| UI/UX 设计师 | 页面复杂、需要美观、流程体验重要 |
| 桌面端专家 | Tauri、Electron、本地文件、系统托盘、安装包 |
| 数据库专家 | 数据模型复杂、迁移、性能、索引、备份 |
| 运维部署专家 | Docker、服务器、CI/CD、域名、HTTPS、监控 |
| 安全专家 | 登录、权限、敏感数据、支付、外部 API、文件写入 |
| 数据分析专家 | 看板、指标、报表、数据清洗、可视化 |
| 移动端专家 | App、小程序、H5 适配、触屏交互 |
| 爬虫/采集专家 | 数据采集、反爬风险、合规边界 |
| 支付/商业化专家 | 订阅、订单、退款、发票、计费安全 |
| AI 提示词专家 | 多模型提示词、评测、工具调用、输出格式约束 |

专家不是越多越好。默认策略应是：核心员工常驻，专家按需加入，每个阶段只激活必要专家。

## 推荐流程

项目启动时，总控 Agent 应执行专家推荐流程：

1. 读取用户目标和约束。
2. 判断项目类型：网站、后台、桌面端、AI 工具、数据产品、移动端、自动化脚本等。
3. 判断技术栈和风险：是否涉及本地文件、账号权限、支付、部署、数据库、第三方 API、模型调用。
4. 输出推荐队伍：
   - 必选核心员工
   - 建议加入专家
   - 暂不需要专家
5. 请求用户确认或允许自动加入低风险专家。
6. 将专家加入当前项目成员，而不是只加入全局员工池。

推荐结果应该出现在项目中，例如：

```text
当前项目建议加入：
- UI/UX 设计师：负责主页画布、员工配置页、交互体验
- 桌面端专家：负责 Tauri、本地文件、安装包
- 安全专家：负责 Runner 权限、保护路径、审批风险
```

## 项目成员与全局员工池

系统应区分两个概念：

- `AI 员工`：全局可用员工池，包含核心角色和可用专家模板。
- `项目成员`：当前项目实际参与的 Agent，可以从员工池中选择，也可以由总控临时推荐。

一个专家可以存在于员工池，但只有加入项目成员后，才会参与该项目的任务拆解、执行和记录。

## 总控调度与职责边界

所有 Agent 都必须由总控 Agent 调度。固定员工和项目专家不能自行抢任务、跨模块决策或越权执行。

核心规则：

1. 总控读取用户目标，判断项目类型、技术栈、风险和当前阶段。
2. 总控决定哪些角色能帮到当前项目。
3. 总控把任务拆成模块内任务，再分配给对应 Agent。
4. Agent 只能在自己的职责模块内产出方案、代码、测试、文档或审查意见。
5. 跨模块任务必须退回总控，由总控拆分或转派。
6. 冲突任务必须由总控召回架构师、审查 Agent 或安全 Agent 协调。
7. 高风险动作必须进入审批，任何 Agent 都不能单独绕过。

职责边界示例：

| 角色 | 只能负责 | 不能负责 |
| --- | --- | --- |
| 产品经理 Agent | 需求、范围、优先级、验收口径 | 直接写代码、决定数据库结构 |
| 架构师 Agent | 技术路线、模块边界、系统风险 | 直接绕过总控改实现 |
| UI/UX Agent | 界面体验、交互、视觉一致性 | 改业务逻辑、改数据库 |
| 前端 Agent | 页面、组件、状态、样式 | 决定 API 契约、改数据库迁移 |
| 后端 Agent | API、服务逻辑、权限集成 | 改 UI 体验、发布部署 |
| 桌面端 Agent | Tauri、本地文件、窗口、安装包 | 改业务策略、越权读写保护路径 |
| 数据库 Agent | 表结构、迁移、索引、数据一致性 | 改产品流程、改 UI |
| AI 提示词 Agent | 提示词、工具调用、输出格式 | 直接调用真实模型、写项目文件 |
| 测试 Agent | 测试、验收、复现问题 | 直接修复实现代码 |
| 安全 Agent | 权限、密钥、保护路径、审批风险 | 放行未经审批的高风险动作 |
| 部署运维 Agent | Docker、CI/CD、环境、监控 | 擅自 push、发布或改业务代码 |
| 文档 Agent | README、接口文档、维护说明 | 改产品行为 |
| 审查 Agent | 最终审查、变更范围、发布风险 | 承担具体实现任务 |

可以把这条规则视为系统宪法之一：

```text
Agent 只能在自己的职责模块内工作；跨模块任务必须由总控拆分或转派；高风险动作必须进入审批。
```

## 专家 Agent 写代码的边界

专家 Agent 可以参与写代码，但必须遵守以下边界：

1. 专家只能通过任务链路获得工作，不允许自由修改项目。
2. 写代码必须绑定到具体任务、项目和输出目录。
3. 文件改动必须进入 Runner 或受控写入服务。
4. 高风险动作必须进入审批：
   - 删除文件
   - 修改保护路径
   - 运行命令
   - Git commit/push
   - 网络请求
   - 涉及密钥、Token、私钥、`.env`
5. 专家输出必须记录：
   - 输入目标
   - 分配任务
   - 使用模型/执行器
   - 产出摘要
   - 文件改动
   - 错误和重试
   - 审查结论

专家 Agent 可以写代码，但不能拥有越权通道。

## UI 设计要求

后续界面建议拆成三个层次：

1. `AI 员工`
   - 全局员工池
   - 每个员工可选执行器和模型
   - 显示核心员工和专家模板

2. `项目成员`
   - 当前项目已加入哪些 Agent
   - 谁是常驻，谁是临时专家
   - 每个成员当前任务、状态、模型和产物

3. `项目专家推荐`
   - 总控根据项目自动推荐专家
   - 用户可一键加入、忽略、替换模型
   - 推荐理由必须可见

主页画布也应体现专家加入后的项目队伍：

```text
总控 Agent
  -> 架构师 / 前端 / 后端 / 测试 / 文档 / 审查
  -> UI/UX 专家 / 桌面端专家 / 安全专家
  -> Runner / 审批 / 运行输出
```

## 数据模型草案

后续可新增或扩展以下概念：

```text

- id
- name
- role
- category: core | expert
- default_executor
- default_model

- risk_level

project_agents
- id
```

第一阶段可以先只做前端展示和本地选择；第二阶段再接入 SQLite 和审批。

## MVP 实现顺序

建议按以下顺序实现：

1. 在 `AI 员工` 页区分核心员工和专家模板。
2. 给每个员工增加执行器/模型选择 UI。
3. 增加 `项目专家推荐` 面板，先用前端规则推荐。
4. 增加 `项目成员` 视图，显示当前项目已启用 Agent。
5. 将专家加入任务拆解结果，让任务能分配给专家。
6. 将专家执行记录接入 `运行输出`。
7. 将高风险专家行为接入 `审批与安全`。
8. 最后再做持久化和真实 Runner 写入。

## 不做的事

当前阶段不要做：

- 不要让专家 Agent 绕过审批直接写文件。
- 不要让 UI 选择模型后立即修改真实执行配置。
- 不要让专家自动运行 shell、Git 或网络请求。
- 不要把 API Key、Token、私钥、`.env` 内容写入数据库、日志或文档。
- 不要为了角色数量好看而默认激活大量专家。

## 成功标准

这个能力完成后，用户应该能做到：

1. 输入一个项目目标。
2. 系统自动推荐适合这个项目的专家 Agent。
3. 用户确认后，专家出现在当前项目中。
4. 总控能把任务分配给专家。
5. 专家能在安全边界内写代码、产出文件、参与审查。
6. 用户能在运行输出和审批页面看见专家做了什么。
7. 项目结束后，专家的贡献能被归档到该项目。

这会让 `agent-swarm` 从“固定几个 AI 员工的控制台”，升级成“按项目自动组建专业团队的本地 AI 公司”。
