# 多模型 Agent 调度功能说明

## 一句话说明

`agent-swarm` 的核心能力是：用户只需要说项目目标，系统会由总控 Agent 拆解任务，再把任务分配给不同岗位的 Agent；每个岗位背后可以使用 Codex、Claude、DeepSeek、Gemini、Cursor、OpenCode 等不同模型或执行器。

这些 Agent 不是各家官方产品里的固定员工，而是 `agent-swarm` 自己定义的岗位。模型只是岗位背后的“大脑”。

## 为什么不同模型也能一起工作

不同模型本身不会互相认识，也不会天然组成团队。

`agent-swarm` 在中间做了统一调度层：

```text
用户目标
  ↓
总控 Agent
  ↓
任务拆解和角色选择
  ↓
岗位配置：前端 Agent / 后端 Agent / 安全 Agent / 文档 Agent ...
  ↓
模型网关：Codex / Claude / DeepSeek / Gemini / Cursor / OpenCode ...
  ↓
各 Agent 返回结果
  ↓
总控汇总、审查、进入 Runner 或审批
```

也就是说，系统不是要求 Codex、Claude、DeepSeek 互相协作，而是由 `agent-swarm` 把它们包装成统一的岗位，并统一收发任务。

## Agent 是什么

在本系统里，Agent 是一个岗位配置，通常包含：

- 名称：例如“前端 Agent”
- 职责：例如“只负责页面、组件、状态和样式”
- 边界：例如“不能决定数据库结构”
- 默认模型：例如 Claude、Codex、DeepSeek 或 Gemini
- 执行器：例如 Codex 执行器、Claude 执行器、OpenClaw 执行器
- 权限：是否只能读、是否能提出写入、是否需要审批
- 记录：任务、输入、输出、文件改动和运行结果

所以 Agent 不是单纯的聊天窗口，而是“岗位 + 模型 + 规则 + 记录”。

## 总控 Agent 怎么调度

总控 Agent 是调度核心。

用户输入一个目标后，总控会做这些事：

1. 理解用户要做什么。
2. 判断项目类型，比如网站、后台、桌面端、AI 工具、数据产品。
3. 判断需要哪些角色，比如前端、后端、数据库、安全、文档、部署。
4. 把目标拆成多个任务。
5. 给每个任务选择合适的 Agent。
6. 给每个 Agent 附上岗位提示词和任务上下文。
7. 收集每个 Agent 的结果。
8. 发现冲突时重新分派或要求审查。
9. 需要写代码或执行高风险动作时进入 Runner 和审批。

示例：

```text
用户：我要做一个本地 AI 蜂群管理工具。

总控判断：
- 需要产品经理 Agent 澄清范围
- 需要架构师 Agent 设计模块
- 需要 UI/UX Agent 设计页面
- 需要前端 Agent 写 React 页面
- 需要后端 Agent 接 API
- 需要桌面端 Agent 接 Tauri
- 需要安全 Agent 审查 Runner 权限
- 需要文档 Agent 更新说明
```

## 岗位提示词怎么约束模型

不同模型接到任务前，系统会给它一段岗位提示词。

例如前端 Agent 的提示词可以是：

```text
你现在是前端 Agent。
你的职责只限于 React 页面、组件、CSS、前端状态。
不能决定数据库结构。
遇到后端接口问题必须交回总控。
当前任务是：修改 AI 员工页面。
```

后端 Agent 会收到不同的提示词：

```text
你现在是后端 Agent。
你的职责只限于 API、服务逻辑、权限集成。
不能改 UI 体验。
遇到页面设计问题必须交回总控。
当前任务是：补充 Agent 配置接口草案。
```

这就是为什么不同模型可以扮演不同角色：角色规则由 `agent-swarm` 提供，模型负责在规则内完成任务。

## 模型网关是什么

模型网关是统一调用不同模型的入口。

它负责：

- 根据 Agent 配置选择模型。
- 把统一任务格式转成对应模型能理解的请求。
- 屏蔽不同模型 API 的差异。
- 控制超时、错误、重试和响应大小。
- 避免把密钥、Token、`.env` 等敏感信息暴露给页面或日志。
- 把模型调用结果记录到运行链路中。

未来一个项目里可以这样配置：

| Agent | 执行器 / 模型 |
| --- | --- |
| 总控 Agent | OpenClaw / DeepSeek |
| 架构师 Agent | Codex |
| UI/UX Agent | Claude |
| 前端 Agent | Cursor 或 Claude |
| 后端 Agent | Codex |
| 安全 Agent | Gemini |
| 文档 Agent | Claude 或 DeepSeek |

用户看到的是 Agent 队伍，系统内部再决定调用哪个模型。

## 写代码是怎么发生的

写代码不是让某个模型直接乱改项目。

正确流程是：

1. 总控生成任务。
2. 任务分配给对应 Agent。
3. Agent 产出方案或补丁建议。
4. 系统判断是否涉及写文件、运行命令、删除文件、Git 操作等高风险动作。
5. 低风险动作进入受控 Runner。
6. 高风险动作进入审批。
7. Runner 执行后记录改了什么文件、产出了什么结果。
8. 总控汇总结果给用户。

也就是说：

```text
Agent 可以参与写代码，但不能绕过 Runner 和审批直接乱写。
```

## 为什么要有职责边界

如果所有 Agent 都能做所有事，系统会变乱。

所以每个 Agent 只能在自己的固定模块内发挥：

- 产品经理 Agent：需求和范围，不写代码。
- 架构师 Agent：模块和技术路线，不直接绕过总控改实现。
- 前端 Agent：页面和组件，不决定数据库结构。
- 后端 Agent：API 和服务逻辑，不改 UI。
- 数据库 Agent：表结构和迁移，不改产品流程。
- 安全 Agent：风险和审批，不放行未经确认的高风险动作。
- 测试 Agent：验证问题，不直接修代码。
- 文档 Agent：说明和文档，不改变产品行为。

跨模块问题必须回到总控，由总控拆分或转派。

## 项目专家 Agent 怎么加入

除了固定员工池，系统还可以按项目加入专家。

例如用户要做桌面端 AI 工具，总控可能推荐：

- UI/UX 专家：负责画布、员工配置和交互体验。
- 桌面端专家：负责 Tauri、本地文件和安装包。
- 安全专家：负责 Runner 权限、保护路径和密钥风险。

专家加入后，会成为当前项目成员，只参与当前项目的任务。

## 用户能看到什么

用户不需要理解所有内部细节，但应该能看到这些信息：

- 当前项目有哪些 Agent。
- 每个 Agent 是核心员工还是项目专家。
- 每个 Agent 使用哪个模型或执行器。
- 每个 Agent 当前负责什么任务。
- 每个 Agent 产出了什么结果。
- 哪些动作等待审批。
- Runner 最终改了哪些文件。

这能让用户知道系统不是黑箱。

## 安全边界

系统必须坚持以下规则：

- API Key、Token、私钥、`.env` 内容不能写入文档、数据库或日志。
- Agent 不能自由运行 shell。
- Agent 不能自由删除文件。
- Agent 不能自由执行 Git commit、push。
- Agent 不能绕过保护路径。
- 高风险动作必须进入审批。
- 所有写入都必须有任务归属、Agent 归属和运行记录。

- Runner 记录每个 Agent 的产物。
- 审批页显示高风险动作来源。

## 用户版总结

`agent-swarm` 的实现方式不是让多个官方 AI 产品直接互相聊天，而是：

```text
我们自己定义岗位，
给岗位绑定不同模型，
由总控统一拆任务和分派，
由模型网关统一调用，
由 Runner 和审批保证安全，
最后把结果统一回到项目。
```

这就是多模型 Agent 团队能协作的原因。
