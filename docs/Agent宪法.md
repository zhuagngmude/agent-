# Agent 宪法（项目 AI 协作规则）

> 本文件为 agent-swarm 项目长期有效的 AI 行为准则。每次进入项目时，必须先读本文件，再读 `dev-docs/` 下的开发和交接文档。

## 一、边界与禁区

### 受保护路径
以下目录和文件不要修改、不要删除、不要作为自动化写入目标：

- `design/image2/`
- `_internal/`
- `data/mock/runtime-state.json`
- `data/local/`
- `logs/`
- `.playwright-cli/`

### 禁止操作
1. **禁止擅自添加新依赖**：Rust 侧（`Cargo.toml`）、前端侧（`package.json`）引入新库必须先说明理由并获批准。
2. **禁止破坏公共接口**：Tauri commands（Rust → 前端）、共享 UI 组件（`packages/ui/`）、数据库 schema（Migration 文件）的签名和行为一旦改动，必须同步更新所有调用方。
3. **禁止绕过设计文件直接改库**：任何数据库结构变更，必须先修改 Migration 文件并说明原因，再执行迁移。
4. **禁止绕过 Model Gateway**：UI、Agent、Runner 或任务状态机不得绕过 Model Gateway 直接调用 provider。

## 二、当前阶段边界

### Runner 安全边界
- Runner 不得自动执行命令、写文件、删文件、发网络请求或修改 Git。
- 项目计划审批只能创建只读 Runner request queue 记录。
- 高风险动作必须先有二次确认和 Git checkpoint。
- 在真实模型链、Agent Run 记录链和执行计划审查链稳定前，不开放真实 Runner 执行。

### 真实模型边界
- 真实模型调用默认关闭。
- 禁止导入 provider SDK、读取 raw key、返回 key fragment、记录 raw prompt 或 raw provider response。
- 后续如需开启真实模型调用，必须先更新准入规格、验证脚本、数据模型和脱敏策略。

### 明确不做
- 不开放真实 Runner 执行
- 不调用真实模型
- 不做云同步
- 不做完整权限系统
- 不写本地项目文件（第一版骨架阶段）

## 三、行为准则

5. **先理解，再动手**：修改任何文件前，必须先阅读相关模块的现有代码和文档，明确其设计意图。不允许为眼前功能瓦解已有抽象。
6. **只用最稳妥的方案**：遇到 Bug 或需求时，优先选择对现有结构影响最小的实现路径。禁止用临时补丁掩盖底层问题。
7. **无证据不修改**：如果无法从代码、文档或对话记录中找到确凿依据，不得凭猜测修改业务逻辑。
8. **所有操作保留后路**：任何有风险的操作（如数据库结构变更、目录重组）必须先说明回滚方案，否则不得执行。

## 四、前端开发规则

9. **框架规范优先**：React + TypeScript + Vite + Ant Design，严格遵循各框架的官方推荐目录结构和最佳实践。
10. **组件复用强制**：同类 UI 结构出现超过 2 次，必须封装为通用组件放入 `packages/ui/`，禁止复制粘贴。
11. **样式 Token 强制**：颜色、字号、间距、圆角、阴影必须从设计 Token 中引用，严禁在组件内硬编码具体数值。
12. **文案统一管理**：在多处重复出现的按钮文字、标题、提示信息等文案，必须提取到独立文案文件统一引用，不在组件内分散硬编码。

## 五、数据库规则

13. **业务先行**：先理清业务对象和关系，再设计表结构，禁止跳过讨论直接建表。
14. **Migration 即真相源**：所有表结构变更必须从 Migration 文件发起，文件纳入 Git 管理。禁止绕过 Migration 直接改库。
15. **命名与字段规范**：表名、字段名统一 `snake_case`；每张业务表包含 `id`、`created_at`、`updated_at`；金额用 `DECIMAL` 或整数分，密码统一存哈希。

## 六、版本管理规则

16. **阶段式提交**：每完成一个可验证的代码变更批次（功能模块、文档收口、架构调整），必须执行 Git 提交，提交信息格式为 `<type>: <简述>`（如 `feat:`、`fix:`、`chore:`、`docs:`）。仅改文档或仅做讨论的阶段，提交节奏可以放宽，但不得长期不提交导致仓库脏状态。
17. **大改动前先存档**：大范围重构、删文件或数据库结构变更前，必须确认当前工作区干净且已有最近提交。
18. **推送前检查**：推送到远程前，必须列出新增/修改文件清单，确认无敏感内容（密钥、内部笔记、本地路径）。

## 七、决策优先级

当不同目标发生冲突时，按以下顺序决策：

- **架构一致性 > 功能实现速度**
- **数据安全与完整性 > 用户体验便利**
- **代码可维护性 > 代码简洁性**
- **明确的已有规范 > 模型自由偏好**

## 八、完成标准

声称"完成"一项任务时，必须满足以下所有条件：

19. **给出可验证的证明**：提供测试步骤、终端输出或截图，证明功能在正常、边界和异常情况下均符合预期。
20. **不留下已知隐患**：无新增警告、错误日志或性能明显下降；处理了空数据、权限不足等边界情况。
21. **交付物完整**：涉及接口变更，已更新对应的类型定义；涉及 UI 改动，已说明在不同状态下的效果；涉及数据库变更，Migration 文件已同步。
22. **主动声明未完成项**：任何暂未处理或需人工确认的部分，必须在回复开头以"⚠️ 待确认"明确列出，不得隐藏。

## 附则：规则运作

- 本宪法规则文件由 AI 工具在每次会话启动时自动加载。
- 项目过程中如果发现规则缺失导致系统性错误，会以"补充案"形式追加条目。
- 所有公开计划、交接说明和阶段记录统一放到 `dev-docs/`，不放在项目根目录。
