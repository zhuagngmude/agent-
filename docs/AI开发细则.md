# AI 开发细则

> 本文件承接《Agent 宪法》的执行细则。宪法管原则，细则管日常开发怎么落地；如果两者冲突，以《Agent 宪法》为准。

## 一、当前阶段边界

- 桌面端是主入口，Web 只作为辅助预览或后续扩展入口。
- 当前允许的本地写入只限已确认的工程文件和 Tauri app data 下的 SQLite 运行库。
- 不写入受保护路径，不绕过审批写用户项目文件。
- 阶段 34 已开放极小范围真实 Runner（仅临时沙箱路径+固定 Git 只读命令+人工二次确认），所有 Runner 执行必须经完整闸门链（preflight→gate→dry-run→exec_lock→minimal_run）；不开放批量/自动 Runner、自由命令、Git commit/push、文件删除、网络请求、云同步、完整权限系统。
- 阶段 35 已开放受控真实模型调用（仅经 Model Gateway、受控模型目录、脱敏审计链路的人工二次确认 project_plan_generation）；不开放前端自由模型配置、key、base URL、prompt、header、body。
- 阶段 35.1 已完成前端中文化收口：用户可见文案默认中文；后端枚举、数据库字段名、内部状态不得直接裸露；英文只允许保留在模型 ID、命令、环境变量、Provider、路径、数据库 ID 等技术标识中。
- Runner 不得自动/越权执行命令、写文件、删文件、发网络请求或修改 Git。阶段 34 仅允许人工二次确认后的临时沙箱写入和固定白名单 Git 只读命令。
- 真实模型调用已在阶段 25/35 边界内开放；禁止导入 provider SDK、读取 raw key、返回 key fragment。禁止把 raw key、raw base URL、raw prompt、raw response、raw provider error 写入前端、日志、SQLite。
- 阶段 36 已定义全自动模式宪法与权限边界：系统支持 `manual`（默认，人工确认）和 `full_auto`（用户显式开启）两种运行模式；总控智能体在 `full_auto` 模式下可按自动授权策略为角色智能体授予 L0-L3 权限，但不开放自动 L4 执行。
- 自动授权矩阵（阶段 36 已定义）：L0 只读观察（允许）、L1 模型草案（允许受控自动，须审计）、L2 计划和队列（允许受控自动，须审计）、L3 执行准备（允许受控自动，须审计）、L4 最小真实执行（不自动开放，永远需人工二次确认）。Git commit/push、文件删除、网络请求、自由命令、保护路径写入：永不自动。
- 全自动模式审计要求：所有自动授权、模型调用、执行准备和执行尝试必须写入审计记录；审计字段禁止保存 raw key、raw base URL、raw prompt、raw provider response、raw provider error、用户项目文件完整内容、未脱敏敏感值。
- 低级智能体实现后续阶段前必须检查阶段 36 边界：阶段 37-43 的每个实现阶段必须在设计文档中声明其权限级别（L0-L4）和是否需要人工确认；不得在阶段 36 边界未明确的情况下自行开放自动执行或绕过审计。

## 二、前端开发细则

- 技术栈固定为 React + TypeScript + Vite + Ant Design。
- `packages/ui/` 是唯一共享 UI 源码，桌面端和 Web 预览必须复用它。
- UI 组件优先使用 Ant Design 和已确认的图标/组件库，禁止重新手搓已有成熟组件。
- 同类 UI 结构出现超过 2 次，必须封装为通用组件放入 `packages/ui/`。
- 颜色、字号、间距、圆角、阴影必须从设计 Token 中引用，避免在组件内分散硬编码。
- 多处重复出现的按钮文字、标题、提示信息等文案，应提取到统一文案文件。
- 用户可见文案默认中文；后端枚举值、数据库字段名、内部状态不得直接裸露给用户。英文只允许保留在模型 ID、命令、环境变量、Provider、路径、数据库 ID 等技术标识中。

## 三、数据库开发细则

- 先理清业务对象和关系，再设计表结构，禁止跳过讨论直接建表。
- Migration 文件是数据库结构真相源，所有结构变更必须纳入 Git。
- 表名、字段名统一 `snake_case`。
- 每张业务表包含 `id`、`created_at`、`updated_at`。
- 当前本地数据库使用 SQLite + rusqlite；运行库放 Tauri app data 目录，不放 `data/local/`。
- SQL 查询必须使用参数化查询，禁止拼接用户输入。

## 四、Tauri/Rust 后端细则

- Rust 侧分层遵守 `commands/`、`services/`、`db/`：
  - `commands/` 只作为 Tauri invoke 入口，负责参数接收和响应封装。
  - `services/` 放业务规则、状态流转和审批判断。
  - `db/` 只做 SQLite 连接、迁移、seed 和数据读写。
- 第一阶段优先补齐只读 commands，再讨论写入 commands。
- 写入、变更、Runner、Git checkpoint、模型相关能力必须进入审批链，不得直接开放。

## 五、版本管理细则

- 每完成一个可验证的代码变更批次，必须执行 Git 提交，提交信息格式为 `<type>: <简述>`。
- 大范围重构、删文件或数据库结构变更前，必须确认当前工作区干净且已有最近提交。
- 暂存文件时优先指定路径，避免使用 `git add .` 误纳入保护路径、构建产物或本地运行数据。
- 推送到远程前，必须列出新增/修改文件清单，确认无敏感内容。

## 六、文档同步细则

- 改 API 同步 `docs/api-draft.md`。
- 改数据结构同步 `docs/data-model-draft.md`。
- 改演示或验收流程同步 `docs/demo-checklist.md` 和 `scripts/README.md`。
- 改阶段状态同步 `dev-docs/下一步开发路线.md`。
- 改 AI 操作边界同步 `docs/Agent宪法.md`、本文件和 `dev-docs/AI开发维护手册.md`。
- 公开计划、交接说明和阶段记录统一放到 `dev-docs/`。

## 七、推理闸

编码前至少回答以下问题，未闭合不开始实现：

1. 实际要解决的问题是什么？
2. 谁创建这个概念、谁调用它、谁消费它？
3. 当前真源在哪里？是否已有同职责模块、表、schema、service？
4. 唯一 owner 是哪一层？哪些层禁止成为 owner？
5. 更简单、更保守的设计是什么？为什么不够？
6. 最大回归风险是什么？用什么测试阻断？
7. 本轮触达的调用链是否闭合到 owner、测试和文档？

## 八、错误分级

审查、实现和验收时必须把问题分级：

- **阻断问题（P0）**：破坏核心功能、owner 边界、安全/隐私、数据真相、构建测试或当前目标链路；当轮必须收掉。
- **设计风险（P1）**：不一定立即破坏，但会导致架构漂移、运行面失控或后续难维护；必须说明取舍和验收入口。
- **可记录债务（P2）**：不影响本轮目标链路，可暂缓；必须说明不处理原因和后续入口。
- **无关优化（P3）**：不进入本轮范围，禁止借机扩大改造。

禁止用情绪强度替代工程优先级。没有证据的担忧不能写成阻断；有证据的阻断也不能降级成"后面再说"。

## 九、多 Agent 协作细则

- 任务可拆成互不干扰的独立域时，才考虑使用多 Agent 模式；小范围任务不应拆分。
- 每个子 agent 必须有明确范围、输入、禁止事项和期望输出。
- 子 agent 修改代码时必须分配不重叠的文件或模块 owner。
- 子 agent 的结论不是最终真源；最终结论仍要由主 agent 对照当前代码、测试和文档验收。
- 不把当前关键路径上的阻塞任务交给子 agent 后原地等待；主 agent 必须继续做不重叠的工作。

## 十、生成文件与不可手改文件

- 标记为 `DO NOT EDIT`、生成目录、schema/codegen 输出、bundle artifact、以及由工具（如 ORM diff 工具）自动生成的 migration 文件只读不手改。手写的 SQL migration 文件是数据库结构真相源，不在本条禁止之列。
- 需要变更时必须找到源文件和生成命令，改源并重新生成。
- 不手改生成文件后跳过生成器。
