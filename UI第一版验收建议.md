## 总体判断

这版 UI 的方向是可以继续往下做的。

它已经避开了普通 AI 聊天页和营销页，整体更像一个开发控制台 / 项目调度台。左侧导航、顶部项目状态、中间工作区、右侧上下文面板这个大结构是正确的，适合 `agent蜂群` 这种多 Agent 协作系统。

但它还不能直接作为最终前端实现，需要先做一轮“产品化打磨”：把信息层级、执行确认、Agent 编排、文档联动和真实状态机制做得更清楚。

## 优先修改项

### P0：工程路径尽量使用英文 ASCII

这次审查时，Python 工具在 `C:/Users/zmd/Documents/项目立项/...` 这类中文路径下出现路径解析问题。

后续规则：

- 代码项目、临时工作区、脚本、构建产物、测试目录尽量使用英文 ASCII 路径。
- 中文可以继续用于 Obsidian 文档标题、说明正文、产品 UI 文案。
- 接收到中文路径资源时，先复制到英文临时目录再运行自动化工具。
- 推荐临时工作目录：`F:/codex-work/`。

### P0：不要使用生成图里的小字作为正式 UI 文案

`agent-swarm-ui-v1-20260608-092448.png` 里的中文存在明显乱码和伪字问题，只能作为布局气质参考，不能作为正式页面依据。

正式 UI 应以 `agent-swarm-prototype.html` 里的真实文本和后续手写文案为准。

### P0：Runner 确认页需要更强的安全表达

当前已经有“查看差异 / 只生成补丁 / 允许执行 / 拒绝”，方向正确，但还需要补足：

- 明确显示本次操作类型：读文件、写文件、执行命令、删除文件、网络请求。
- 明确显示 Git checkpoint 是否已经创建。
- 明确显示允许执行后的后果。
- `允许执行` 不应和普通按钮同级，需要二次确认。
- 高风险文件要有更醒目的解释，例如“会影响本地 Runner 权限逻辑”。

### P1：Agent 编排页需要更像“关系图 + 队列”，不要只是卡片列表

当前能看到父 Agent 和子 Agent，但还不够直观。

建议改成：

- 左侧：Agent 树 / 拓扑关系。
- 中间：当前 Agent 的任务队列。
- 右侧：该 Agent 的职责、模型、权限、上下文、输出摘要。
- 子 Agent 必须显示来源、目的、回传状态。

### P1：任务拆解页需要支持筛选和批量操作

现在表格方向对，但需要补充：

- 按状态筛选：待分配、进行中、待审查、失败、等待用户确认。
- 按 Agent 筛选。
- 按风险等级筛选。
- 支持任务详情抽屉。
- 支持“重新分配 Agent”“标记阻塞”“进入审查”。

### P1：执行流水线页要能回放每一步

现在流程线是对的，但需要每一步可展开：

- 输入内容。
- Agent 输出摘要。
- 失败原因。
- 使用的模型。
- 耗时与 token / 成本估计。
- 关联任务和文件。

### P1：右侧上下文面板要跟随页面变化

右侧面板目前像固定信息。

建议根据当前页面变化：

- 总览页：显示当前项目风险、待确认事项、最近文档。
- Agent 页：显示选中 Agent。
- Runner 页：显示风险和文件差异。
- Git 页：显示选中 commit 的关联任务和文档。
- 文档页：显示人类文档与 AI 手册的对应关系。

### P2：视觉细节继续打磨

建议保留当前冷静、专业、高密度方向，但优化：

- 减少大面积卡片边框，改成更清楚的分栏和表格。
- 统一 badge 颜色含义。
- 左侧导航图标可以加入，但不要装饰化。
- 表格需要 hover、选中、空状态、加载状态。
- 高风险按钮和普通主按钮要拉开视觉差距。
- 需要 1440px 和 1920px 两种桌面宽度检查。

## 可以保留的部分

- 左侧导航 + 顶部项目状态 + 中间主工作区 + 右侧上下文面板。
- 页面范围完整，覆盖了总览、Agent、任务、流水线、Runner、Git、知识库、设置。
- 颜色克制，没有走廉价 AI 大屏路线。
- Runner 页面已经体现了“用户确认优先”的产品方向。
- Git 保存点和文档联动的概念已经放进 UI 里。

## 下一步建议

第二版不建议再做 8 套风格，而是基于这一版做 3 个核心页面的高质量版本：

1. 项目总览工作台
2. Agent 编排与任务分配页
3. Runner 执行确认页

这三个页面决定产品是否“看起来能用、敢用、可控”。等这三页确定后，再扩展 Git 历史、知识库和设置页。
