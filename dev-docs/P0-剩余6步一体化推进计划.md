## 1. 最终目标

把 AI 员工、任务分派、Runner 边界校验和运行输出串成一条真实本地闭环：

```text
AI 员工真实配置
-> 总控确定性选择项目 Agent
-> 任务绑定 project_agents.id
-> Runner 执行前做后端边界检查
-> 运行输出能追踪 Agent / 执行器 / 模型 / 产物
-> 重启桌面端后配置仍在 SQLite 中
```

最终数据口径：

- `project_agents.id` 是任务分配真源。
- `tasks.assigned_agent_id` 逐步绑定 `project_agents.id`，不再绑定旧 `agents` 表。
- 专家推荐复用 `agent_templates(category=expert)` 和 `project_agents(source=recommended/manual)`。
- `model_catalog` 继续按 `(project_id, executor_key, provider, model_id, purpose)` 唯一。
- `agent_boundary_checks` 是 Runner 执行前边界检查的后端记录，只读展示给前端。

## 2. 公共接口目标

新增或补齐这些 Tauri commands：

```text
recommend_project_agents
assign_project_agents_to_task
check_agent_boundary
list_agent_boundary_checks
```

继续使用这些既有写入口：

```text
upsert_project_agent
remove_project_agent
upsert_executor_config
```

接口边界：

- `recommend_project_agents` 第一版只做确定性规则，不调用模型。
- `assign_project_agents_to_task` 必须校验被分配对象存在于 `project_agents`，并且属于当前项目或允许的全局项目口径。
- `check_agent_boundary` 结果只能是 `allowed`、`denied`、`needs_approval`。
- `list_agent_boundary_checks` 只读，不允许前端写入或伪造检查记录。

### Checkpoint 4：Runner 执行前边界强校验

目标：

- 新增 `check_agent_boundary` service/command。
- Runner preflight 前检查 `agent_id`、`task_type`、`module_scope`、`target_path`、`forbidden_actions`。
- 每次检查写入 `agent_boundary_checks`。

主要文件：

```text
apps/desktop/src-tauri/src/services/agent_config.rs
apps/desktop/src-tauri/src/services/runner_preflight.rs
apps/desktop/src-tauri/src/commands/agent_config.rs
apps/desktop/src-tauri/src/commands/runner_preflight.rs
```

判定规则：

- Agent 不存在、已移除或不属于项目：`denied`。
- `task_type` 不在模板允许范围：`denied` 或 `needs_approval`。
- `target_path` 不在允许路径内：`denied`。
- 命中 `forbidden_actions`：`denied`。
- 风险不确定但可能允许：`needs_approval`。
- 只有明确满足边界时：`allowed`。

验收：

- Runner preflight 不能绕过后端边界检查。
- `agent_boundary_checks` 能看到 allowed / denied / needs_approval 三类结果。
- denied 的任务不能继续进入执行链路。
