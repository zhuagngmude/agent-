# 阶段 21：真实模型接入新架构适配设计

日期：2026-06-14

本文把旧项目中已有的 Model Gateway 准入规则（阶段 2 已收口的 helper-only scaffold）适配到新 Tauri/Rust 架构口径。本阶段只做设计文档，不写代码、不建表、不发 provider 请求。

## 一、背景

### 1.1 旧架构当前状态

旧 Node.js API（`services/api/`）中有 6 个 Model Gateway 模块，全部处于 `feature_disabled` / helper-only 模式：

| 旧模块 | 文件 | 当前状态 |
|--------|------|----------|
| 主网关入口 | `model-gateway.js` | 返回全局 disabled 状态 + provider 元数据 |
| Provider Adapter 注册表 | `model-gateway-adapters.js` | 4 个 provider 全部 `mode: "disabled"` |
| Provider 配置解析器 | `model-gateway-provider-config.js` | 只解析 env，不读取 raw secret |
| 项目计划请求构建器 | `model-gateway-project-plan.js` | 永远返回 `feature_disabled` |
| 脱敏/响应限制器 | `model-gateway-redaction.js` | `canWrite=false` |
| Model Calls 写入/迁移草案 | `model-gateway-model-calls.js` | helper-only scaffold，`canWrite=false` |

4 个旧 API 路由全部返回 `blocked / feature_disabled`：
- `GET /api/model-gateway/status`
- `POST /api/model-gateway/dry-run`
- `POST /api/model-gateway/connectivity-test`
- `POST /api/projects/:projectId/project-plan-model-requests`

### 1.2 准入规格状态

旧项目定义了 17 项准入检查项（`dev-docs/真实模型接入准入规格.md`），步骤 1-9 已完成（禁用态验证），步骤 10-12 未开始（真实 SDK 接入、一次真实调用、结果入审批草案）。

### 1.3 新架构目标

把 Model Gateway 从旧 Node.js HTTP 路由架构迁移到新 Tauri/Rust 架构。本阶段（阶段 21）只做设计，阶段 22-25 逐步实现。

## 二、总体架构适配

### 2.1 旧架构 vs 新架构对照

```
旧架构（Node.js HTTP）              新架构（Tauri/Rust）
─────────────────────────          ─────────────────────────
前端 fetch()                       前端 invoke()
  ↓                                  ↓
POST /api/.../project-plan-        #[tauri::command]
  model-requests                     request_project_plan_model_draft
  ↓                                  ↓
server.js 路由分发                  commands/model_gateway.rs（薄封装）
  ↓                                  ↓
model-gateway.js 主入口             services/model_gateway.rs（业务编排）
  ↓                                  ↓
model-gateway-project-plan.js       services/model_gateway/ 子模块：
model-gateway-provider-config.js      project_plan.rs（请求构建+验证）
model-gateway-adapters.js             provider_config.rs（配置解析）
model-gateway-redaction.js            redaction.rs（脱敏+限长）
model-gateway-model-calls.js          model_calls.rs（记录草案）
  ↓                                  ↓
Mock runtime state                  SQLite model_calls 表（阶段 23 建）
```

### 2.2 入口定义

**唯一 Tauri command：**

```rust
#[derive(Deserialize)]
#[serde(deny_unknown_fields)]  // 拒绝前端传入 idea/constraints 以外的任何字段
struct ProjectPlanModelDraftInput {
    idea: String,                          // 必填，项目想法描述
    constraints: Option<String>,           // 选填，约束条件
    second_confirm: Option<bool>,          // 阶段 22 暂不启用，阶段 24 真实调用前必须为 true
    confirm_text: Option<String>,          // 阶段 22 暂不启用，阶段 24 真实调用前必须匹配 "我确认发起真实模型调用"
}

#[tauri::command]
fn request_project_plan_model_draft(
    state: tauri::State<'_, DbState>,
    input: ProjectPlanModelDraftInput,
) -> Result<ProjectPlanModelDraftResponse, String>
```

**`#[serde(deny_unknown_fields)]` 的作用：**
- 前端只能传 `idea`、`constraints`、`second_confirm`、`confirm_text` 四个字段
- 传入 apiKey、baseUrl、headers、provider、model、prompt、stream、runnerJobId 等字段会被 serde 直接拒绝
- 不需要额外写 forbidden field 检查逻辑，结构体定义本身即为白名单

**触发方式：** 用户在前端页面点击按钮触发，禁止页面加载自动触发、禁止 Agent 触发、禁止 Runner 触发。

**二次确认字段（阶段 24/25 启用）：**
- 阶段 22（helper-only scaffold）暂不校验 `second_confirm` 和 `confirm_text`
- 阶段 24 真实调用前，`second_confirm` 必须为 `true`，`confirm_text` 必须等于 `"我确认发起真实模型调用"`
- 这些字段对齐旧准入规格中要求的人工确认环节

### 2.3 调用链

```
UI 按钮点击（用户主动）
  ↓
Tauri invoke("request_project_plan_model_draft", { idea, constraints? })
  ↓
commands/model_gateway.rs
  → 提取 DbState
  → 调用 services::model_gateway::create_project_plan_draft()
  ↓
services/model_gateway/mod.rs（业务编排）
  → 1. 检查 feature flag（AGENT_SWARM_ENABLE_REAL_MODEL_PROJECT_PLAN）
  → 2. 验证输入（禁止客户端传 provider/key/prompt/headers）
  → 3. 解析 provider 配置（env var → configured/missing/invalid）
  → 4. 检查 provider/model/用途白名单
  → 5. 构建脱敏请求信封（不包含 raw key）
  → 6a. feature flag 关闭 → 返回 feature_disabled 草案
  → 6b. feature flag 开启 → 通过 adapter 发请求 → 脱敏响应
  → 7. 生成 project_plan 审批草案（不创建 task/approval/runner job）
  ↓
返回 ProjectPlanModelDraftResponse
```

## 三、配置与密钥

### 3.1 环境变量

第一版只读两个环境变量：

```
AGENT_SWARM_OPENAI_COMPAT_API_KEY    # provider API key
AGENT_SWARM_OPENAI_COMPAT_BASE_URL   # provider base URL（OpenAI-compatible endpoint）
AGENT_SWARM_ENABLE_REAL_MODEL_PROJECT_PLAN  # feature flag，默认 "false"
```

### 3.2 密钥存储原则

- Key 只在 Rust 服务端通过 `std::env::var()` 读取
- 不进前端（不通过 invoke 返回值暴露）
- 不进 SQLite（不建 `api_keys` 表）
- 不进日志、不进 runtime_events、不进 model_calls
- 不进错误消息（错误只返回 `feature_disabled` / `provider_config_missing` 等枚举，不返回 key fragment）
- 后续如需持久化密钥，必须独立设计 Secret Store（OS keychain 或加密文件），不在本阶段范围内

### 3.3 Feature Flag

`AGENT_SWARM_ENABLE_REAL_MODEL_PROJECT_PLAN` 默认 `"false"`：

- 等于 `"false"` 或未设置 → 返回 `feature_disabled`，不发起任何网络请求
- 等于 `"true"` 且 Key/URL 均有效 → 走真实调用链路
- 等于 `"true"` 但 Key 缺失 → 返回 `provider_config_missing`
- 等于 `"true"` 但 URL 无效 → 返回 `provider_config_invalid`

此 flag 只控制 `project_plan_generation` 用途，不能复用为其他用途的开关。

## 四、Provider 策略

### 4.1 第一版只选 openai_compat

不做多 provider 选择，不做前端 provider 下拉框。

- Provider ID：`openai_compat`
- 默认模型：`gpt-5.4-mini`（与旧项目 `provider-config.js` 对齐）
- 允许的模型列表：`["gpt-5.4-mini"]`（第一版只允许一个）
- 允许的用途：`["project_plan_generation"]`（第一版只允许一个）

### 4.2 Provider 配置解析（Rust 版）

`services/model_gateway/provider_config.rs`：

```rust
pub struct ProviderConfig {
    pub provider_id: String,
    pub status: ProviderConfigStatus,  // Configured | MissingKey | MissingBaseUrl | InvalidBaseUrl
    pub default_model: String,
    pub allowed_models: Vec<String>,
    pub allowed_purposes: Vec<String>,
}

pub enum ProviderConfigStatus {
    Configured,
    MissingKey,
    MissingBaseUrl,
    InvalidBaseUrl,
}

pub fn resolve_provider_config() -> ProviderConfig {
    // 只读 env，不返回 raw key/base URL
    let key = std::env::var("AGENT_SWARM_OPENAI_COMPAT_API_KEY").ok();
    let base_url = std::env::var("AGENT_SWARM_OPENAI_COMPAT_BASE_URL").ok();

    let status = match (&key, &base_url) {
        (None, _) => ProviderConfigStatus::MissingKey,
        (_, None) => ProviderConfigStatus::MissingBaseUrl,
        (Some(_), Some(url)) if !is_valid_base_url(url) =>
            ProviderConfigStatus::InvalidBaseUrl,
        _ => ProviderConfigStatus::Configured,
    };

    ProviderConfig {
        provider_id: "openai_compat".into(),
        status,
        default_model: "gpt-5.4-mini".into(),
        allowed_models: vec!["gpt-5.4-mini".into()],
        allowed_purposes: vec!["project_plan_generation".into()],
    }
}
```

注意：`resolve_provider_config()` 不返回 key、key suffix、masked fragment 或 base URL 原文。调用方只能得到 `Configured | MissingKey | MissingBaseUrl | InvalidBaseUrl` 四种粗粒度状态。

## 五、唯一用途：project_plan_generation

### 5.1 输入验证（对齐旧 forbiddenClientFieldTokens）

请求构建器 `services/model_gateway/project_plan.rs` 在构建请求信封前必须验证：

**接受的字段：** `idea`（必填）、`constraints`（选填）

**拒绝的字段（共 23 项，与旧 `forbiddenClientFieldTokens` 对齐）：**
```
apiKey, authorization, baseUrl, headers, systemPrompt,
prompt, stream, tools, files, runnerJobId,
temperature, topP, maxTokens, provider, model,
key, secret, token, password, bearer,
api_key, sk-, token=
```

**拒绝的值模式（与旧 `forbiddenValuePatterns` 对齐）：**
- `/sk-[a-zA-Z0-9]{20,}/`（OpenAI 风格的 key）
- `/api_key=[a-zA-Z0-9_-]+/i`
- `/authorization:\s*bearer\s+[a-zA-Z0-9._-]+/i`
- `/token=[a-zA-Z0-9._-]+/i`
- `/password=[a-zA-Z0-9._-]+/i`

### 5.2 输出约束

`project_plan_generation` 返回的结果只能是一个"项目计划审批草案"：

```rust
pub struct ProjectPlanModelDraftResponse {
    pub status: String,             // "feature_disabled" | "draft_ready" | "provider_error"
    pub draft_id: Option<String>,   // 草案 ID（如有）
    pub summary: Option<String>,    // 脱敏后的计划摘要
    pub sections: Option<Vec<PlanSection>>,  // 计划分节
    pub warnings: Vec<String>,      // 安全警告（如 "redaction_applied"）
}
```

**禁止：**
- 不能直接创建 Task
- 不能直接创建 Approval（`target_service = "project_plan"` 目前不在 Rust approvals 允许列表中，需后续独立扩展审批设计）
- 不能创建 Runner job
- 不能触发 Agent Run
- 不能写文件
- 不能修改 Git
- 不能把 raw provider response 返回给前端

### 5.3 草案性质

阶段 21-25 的 `ProjectPlanModelDraftResponse` 是**内存中的只读草案**：

- 不写入 `approvals` 表（不是 Approval 记录）
- 不写入 `projects` 表（不修改项目状态）
- 阶段 25 真实调用成功后，草案仅在 Rust 返回值中携带，前端展示后即丢弃
- 后续如要落地为正式 Approval，必须先扩展 Rust approvals 的 `target_service` 白名单，新增 `project_plan`，并设计 `project_plan` 审批通过后的任务实例化流程

### 5.4 结果后续处理

`project_plan_generation` 的结果是只读草案。用户在前端查看草案后，可以选择：
- 确认 → 进入 project_plan 审批流程（阶段 21 不做，且需先扩展 Rust approvals）
- 拒绝 → 丢弃草案
- 修改 constraints → 重新请求

草案确认后的流程（后续阶段设计，需先扩展 Rust `target_service` 白名单）：
```
用户确认草案
  → POST create_approval（target_service = "project_plan"，需先扩展 Rust）
  → 审批通过
  → 实例化 project plan（创建 tasks + 只读 runner request 记录）
```

## 六、数据层设计

### 6.1 model_calls 表

阶段 21 不建表。阶段 23 建 `model_calls` 表，字段对齐旧 `model-gateway-model-calls.js` 中定义的 `modelCallColumns`（20+ 列）：

**核心字段：** `id`, `project_id`, `purpose`, `provider`, `model`, `status`, `request_hash`, `structured_summary`, `token_usage`, `cost_estimate`, `error_category`, `error_message`, `redaction_applied`, `duration_ms`, `related_approval_id`, `runtime_event_id`, `created_at`, `updated_at`

**关键约束（从 `data-model-draft.md` 对齐）：**
- `request_hash` 基于脱敏后的固定请求信封生成，不可逆推出原始输入
- `structured_summary` 只保存结构化、脱敏、限长后的摘要
- `token_usage` 和 `cost_estimate` 只保存粗粒度结果，不保存账单凭据或原始 provider 响应
- 不保存 raw prompt、raw provider response、raw key
- 每次状态变化可选地写入 `runtime_events`，通过 `runtime_event_id` 关联
- 写入 model_calls 和对应审计事件必须在同一事务中

### 6.2 脱敏规则（对齐旧 redaction.js）

```
sk-...                    → [REDACTED_SECRET]
Authorization: Bearer ... → [REDACTED_SECRET]
api_key=...               → [REDACTED_SECRET]
token=...                 → [REDACTED_SECRET]
password=...              → [REDACTED_SECRET]
URL（含 query token）     → [REDACTED_URL]
```

响应体大小限制：4096 字节（与旧项目对齐）。

## 七、错误分类

与旧 `allowedErrorCategories`（15 项）对齐：

| 错误类别 | 含义 |
|----------|------|
| `feature_disabled` | Feature flag 关闭 |
| `missing_key` | 环境变量中无 API key |
| `missing_base_url` | 环境变量中无 base URL |
| `invalid_base_url` | base URL 不合法（非 https、localhost、私有 IP 等） |
| `unsupported_provider` | 请求了不支持的 provider |
| `unsupported_model` | 请求了不在白名单中的模型 |
| `invalid_purpose` | 请求了 `project_plan_generation` 以外的用途 |
| `forbidden_field` | 请求包含了禁止的客户端字段 |
| `timeout` | Provider 请求超时 |
| `provider_error` | Provider 返回错误 |
| `response_too_large` | 响应超过大小限制 |
| `redaction_failed` | 脱敏过程失败 |
| `unknown` | 未分类错误 |

## 八、阶段规划

### 阶段 21（本文）：真实模型接入新架构适配设计

### 阶段 22：Model Gateway Rust helper-only scaffold
- 新建 `services/model_gateway/mod.rs`、`provider_config.rs`、`project_plan.rs`、`redaction.rs`

- 不建表
- Rust 单测：feature flag 关闭、key 缺失、base URL 无效、forbidden field 拒绝

### 阶段 23：model_calls migration + helper-only 写入草案
- 新建 `data/migrations/003_add_model_calls.sql`
- 在 `db/mod.rs` 中执行 migration
- `services/model_gateway/model_calls.rs`：helper-only 写入草案，`canWrite=false`（与旧 `model-gateway-model-calls.js` 对齐）

### 阶段 24：单 provider 真实调用，默认关闭
- 导入 Rust HTTP 客户端（候选 `reqwest`）—— 新增 Rust crate 依赖必须另行说明理由并获批准后方可执行
- 实现 `openai_compat` adapter
- `request_project_plan_model_draft` 在 feature flag 开启且配置有效时发起真实请求
- 响应脱敏、限长
- 结果写入 `model_calls` + `runtime_events`
- 默认 `AGENT_SWARM_ENABLE_REAL_MODEL_PROJECT_PLAN=false`
- 新增 Rust 单测：成功/失败/超时/脱敏

### 阶段 25：一次人工触发真实 project_plan_generation
- 用户配置 env var 后，在 UI 中点击按钮触发一次真实调用
- 结果以只读草案形式展示
- 验证全链路：前端 invoke → Tauri command → Rust service → provider → 脱敏 → 返回草案
- 全验证：typecheck / build / cargo check / cargo test / verify-model-gateway.ps1

## 九、边界约束（全阶段通用）

- 不导入 provider SDK（只使用标准 HTTP 客户端）
- 不把 raw key 写入 SQLite、日志、runtime_events、错误消息
- 不把 raw prompt 或 raw provider response 保存到任何存储
- 不把 raw key 返回给前端
- 不创建 Task（除非用户后续确认审批草案）
- 不触发 Agent Run
- 不启用 Runner
- 不写用户文件
- 不操作 Git
- `AGENT_SWARM_ENABLE_MODEL_CONNECTIVITY_TEST` 不复用为真实业务开关
- 所有真实调用默认关闭（feature flag 控制）
