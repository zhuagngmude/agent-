# 阶段 28：可配置任务角色模板

目标：阶段 27 已经让真实模型草案在人工审批后生成 5 个固定任务和 5 条只读 `runner_requests`。阶段 28 要把这 5 个固定角色模板变成“本地可配置模板”，允许用户在安全边界内增加或停用角色，例如 `security`、`devops`、`ux`、`data`。

这一阶段只配置任务模板，不执行 Runner、不执行命令、不写用户项目文件、不改 Git、不让模型自由决定任务数量。

## 一、阶段边界

允许做：

- 新增本地 SQLite 表保存项目级任务角色模板。
- 提供 Tauri command 读取模板列表。
- 提供 Tauri command 保存模板配置。
- `approve_project_plan` 按已启用模板生成任务和只读 `runner_requests`。
- 前端在 `ProjectPlanPage` 或独立配置区域展示模板，允许用户启用/停用内置扩展角色。
- 允许角色数量从固定 5 个变为“启用中的模板数量”。

禁止做：

- 不执行 Runner。
- 不执行命令。
- 不写用户项目文件。
- 不改 Git。
- 不创建可执行 Runner job。
- 不调用真实模型。
- 不让模型输出直接新增模板。
- 不让前端传 raw key / base URL / provider / model / prompt / headers / body。
- 不允许用户填写真实文件路径，只允许 `virtual/...` 形式的虚拟 affected file。
- 不允许用户填写任意 operation type，只允许后端白名单。
- 不允许用户填写任意 agent id，只允许当前项目已有且后端允许的 agent。
- 不触碰保护路径：`design/image2/`、`_internal/`、`data/mock/runtime-state.json`、`data/local/`、`logs/`、`.playwright-cli/`。

## 二、为什么阶段 28 做模板配置

当前固定 5 个角色：

```text
frontend
backend
qa
docs
reviewer
```

这足够完成最小闭环，但真实项目经常还需要：

```text
security
devops
ux
data
```

如果直接让模型决定任务数量，会把风险扩大到模型输出解析、Agent 分配、权限、文件范围和 Runner 边界。阶段 28 更稳的路线是：先由本地模板决定任务结构，模型只能提供草案摘要，用户审批后按模板生成任务。

## 三、推荐数据模型

新增 migration：

```text
006_add_project_plan_task_templates.sql
```

新增表：

```sql
CREATE TABLE IF NOT EXISTS project_plan_task_templates (
  id TEXT PRIMARY KEY,
  project_id TEXT NOT NULL,
  role TEXT NOT NULL,
  agent_id TEXT NOT NULL,
  title TEXT NOT NULL,
  description TEXT NOT NULL,
  priority TEXT NOT NULL,
  risk_level TEXT NOT NULL,
  affected_file TEXT NOT NULL,
  operation_type TEXT NOT NULL,
  enabled INTEGER NOT NULL DEFAULT 1,
  sort_order INTEGER NOT NULL,
  is_builtin INTEGER NOT NULL DEFAULT 1,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  FOREIGN KEY (project_id) REFERENCES projects(id),
  FOREIGN KEY (agent_id) REFERENCES agents(id)
);

CREATE INDEX IF NOT EXISTS idx_project_plan_task_templates_project_id
  ON project_plan_task_templates(project_id);

CREATE INDEX IF NOT EXISTS idx_project_plan_task_templates_enabled
  ON project_plan_task_templates(project_id, enabled, sort_order);

CREATE UNIQUE INDEX IF NOT EXISTS idx_project_plan_task_templates_project_role
  ON project_plan_task_templates(project_id, role);
```

说明：

- `affected_file` 第一版只允许 `virtual/...`，不要保存真实路径。
- `operation_type` 必须来自后端白名单。
- `enabled` 控制是否生成任务。
- `sort_order` 控制任务顺序。
- `is_builtin` 标记内置模板，第一版可以只允许启用/停用内置模板，不允许用户自由新增。

## 四、内置模板

第一版建议提供 9 个内置角色，其中前 5 个默认启用，后 4 个默认停用。

默认启用：

| role | agent_id | priority | risk_level | operation_type | affected_file |
| --- | --- | --- | --- | --- | --- |
| frontend | agent_frontend | high | medium | frontend_plan | virtual/frontend-plan.md |
| backend | agent_backend | high | medium | backend_plan | virtual/backend-plan.md |
| qa | agent_qa | medium | low | qa_plan | virtual/qa-plan.md |
| docs | agent_docs | medium | low | docs_plan | virtual/docs-plan.md |
| reviewer | agent_reviewer | high | medium | review_plan | virtual/review-plan.md |

默认停用：

| role | agent_id | priority | risk_level | operation_type | affected_file |
| --- | --- | --- | --- | --- | --- |
| security | agent_reviewer | high | high | security_review_plan | virtual/security-plan.md |
| devops | agent_backend | medium | medium | devops_plan | virtual/devops-plan.md |
| ux | agent_frontend | medium | low | ux_plan | virtual/ux-plan.md |
| data | agent_backend | medium | medium | data_plan | virtual/data-plan.md |

如果当前 seed 没有专门的 `agent_security`、`agent_devops`、`agent_ux`、`agent_data`，不要新增 agent。第一版可以复用已有 agent：

- security -> `agent_reviewer`
- devops -> `agent_backend`
- ux -> `agent_frontend`
- data -> `agent_backend`

### 5.1 migration 加载

在 `db/mod.rs` 中加载 migration 006。

因为当前 migration runner 每次启动都会 `execute_batch`，migration SQL 必须全部可重复执行。

除了建表和索引，还需要 seed 内置模板。推荐不要在纯 SQL 里写复杂 upsert 逻辑；可以在 Rust migration helper 中：

1. 确保表存在。
2. 读取当前项目。
3. 对每个内置模板 `INSERT OR IGNORE`。
4. 不覆盖用户已经修改过的 enabled 状态。

如果实现更简单，也可以在 `project_plan` service 第一次读取模板时懒初始化，但必须有测试覆盖。

### 5.2 新增类型

建议在 `services/project_plan.rs` 增加：

```rust
pub struct ProjectPlanTaskTemplateSummary {
    pub id: String,
    pub project_id: String,
    pub role: String,
    pub agent_id: String,
    pub title: String,
    pub description: String,
    pub priority: String,
    pub risk_level: String,
    pub affected_file: String,
    pub operation_type: String,
    pub enabled: bool,
    pub sort_order: i64,
    pub is_builtin: bool,
    pub created_at: String,
    pub updated_at: String,
}

pub struct UpdateProjectPlanTaskTemplateInput {
    pub role: String,
    pub enabled: bool,
}
```

第一版只允许按 `role` 启用/停用，不允许编辑 title、description、agent_id、operation_type、affected_file。这样更稳。

### 5.3 新增 service 函数

建议新增：

```rust
pub fn list_project_plan_task_templates(
    connection: &Connection,
) -> Result<Vec<ProjectPlanTaskTemplateSummary>, String>

pub fn update_project_plan_task_template(
    connection: &mut Connection,
    input: UpdateProjectPlanTaskTemplateInput,
) -> Result<Vec<ProjectPlanTaskTemplateSummary>, String>
```

`list` 行为：

- 获取当前项目。
- 确保内置模板已 seed。
- 按 `sort_order, role` 返回全部模板。

`update` 行为：

- 获取当前项目。
- 校验 role 在内置角色白名单内。
- 只更新 `enabled` 和 `updated_at`。
- 更新后返回模板列表。
- 不写 tasks。
- 不写 runner_requests。
- 不写 runtime_events。
- 不执行 Runner。

### 5.4 approve_project_plan 改造

把当前固定常量：

```rust
const PROJECT_PLAN_ASSIGNMENTS: [ProjectPlanAssignment; 5]
```

改成：

- 内置模板 seed 源仍可用常量。
- `approve_project_plan` 实例化时从 SQLite 读取 `enabled=1` 的模板。
- 按启用模板生成 tasks 和只读 runner_requests。

要求：

- 至少要有 1 个启用模板，否则拒绝审批：

```text
invalid_input: at least one project plan task template must be enabled
```

- 默认 seed 后仍然生成 5 个任务，确保阶段 24/27 行为不破。
- 启用 `security` 后应生成 6 个任务和 6 条只读 runner_requests。
- 停用 `docs` 后应生成 4 个任务和 4 条只读 runner_requests。
- 模板变化只影响之后新审批的草案，不回改已实例化草案。

### 5.5 依赖关系规则

当前非 frontend 任务依赖 frontend。阶段 28 可继续保持简单规则：

- 如果启用了 `frontend`，其他任务依赖 frontend。
- 如果没有启用 `frontend`，所有任务无依赖。

不要引入复杂 DAG 配置。

## 七、测试要求

Rust 必须新增测试。

### 7.1 migration 和 seed

- migration 006 后表存在。
- 默认内置 9 个模板。
- 默认启用 5 个。
- 默认停用 4 个。
- 重复初始化不重复插入。

### 7.2 list templates

- `list_project_plan_task_templates` 返回当前项目模板。
- 按 `sort_order` 排序。
- 不返回其他项目模板。

### 7.3 update enabled

- 启用 `security` 成功。
- 停用 `docs` 成功。
- 未知 role 拒绝。
- update 不写 tasks。
- update 不写 runner_requests。
- update 不写 runtime_events。

### 7.4 approve 默认兼容

默认配置下，本地草案审批仍创建 5 个 tasks 和 5 条只读 runner_requests。

真实模型草案审批默认仍创建 5 个 tasks 和 5 条只读 runner_requests。

### 7.5 approve 使用启用模板数量

- 启用 `security` 后审批创建 6 个 tasks 和 6 条只读 runner_requests。
- 停用 `docs` 后审批创建 4 个 tasks 和 4 条只读 runner_requests。
- 全部停用后审批拒绝，不写 tasks/runner_requests/runtime_events。

### 7.6 安全污染拒绝

手动污染数据库后应拒绝审批：

- `affected_file = ../secret`
- `affected_file = C:\Users\...`
- `operation_type = file_write`
- `agent_id = unknown_agent`

断言：

- 不创建 tasks。
- 不创建 runner_requests。
- 不写 runtime_events。

### 7.7 Runner 边界

- 生成的 `runner_requests.safety_note` 仍是只读说明。
- `side_effects.executes_runner = false`。
- `side_effects.writes_project_files = false`。
- `side_effects.modifies_git = false`。
- `side_effects.calls_real_model = false`。

前端验证：

- `npm run typecheck`
- `npm run build`

Rust 验证：

- `cargo fmt --check`
- `cargo check`
- `cargo test`
