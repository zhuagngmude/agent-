# 阶段 35.2-43 全自动 Agent 蜂群总方案

> 更新提示（2026-06-16）：本文件是早期全自动总方案，阶段编号已被 `dev-docs/当前项目导航.md` 修正。当前下一步是阶段 38“项目类型分流与通用想法入口”；旧阶段 38“自动工作流模板”和旧阶段 41“多 Provider / 多 Key 管理”仅作历史参考，不作为当前执行顺序。

## 1. 总目标

把当前 `agent-swarm` 从“人工审批驱动的本地 AI 协作控制台”，升级为“用户给出想法后，由总控智能体自动扩展想法、编排工作流、授权角色智能体、调用多模型、执行任务、测试返修和交付结果”的全自动本地 Agent 蜂群。

用户最终只负责：

- 给出初始想法。
- 选择是否开启全自动模式。
- 必要时按一键停止或查看结果。

系统负责：

- 引导和扩展想法。
- 生成项目种子和执行计划。
- 选择角色、模型和 Skill。
- 自动授权和审计。
- 自动执行、测试、返修、验收和回滚。

## 2. 当前基础不推倒

现有阶段 25-35.1 不是废弃代码，而是全自动系统的安全底座。

继续保留：

- `model_gateway`：所有模型调用仍必须从这里走。
- `model_calls`：模型调用安全审计。
- `project_plan_drafts`：想法、方案、模型草案和项目计划草案载体。
- `approvals`：后续从人工审批扩展为“人工审批 + 总控智能体自动审批记录”。
- `tasks`：角色智能体任务实例。
- `runner_requests`：Runner 请求入口。
- `runner_preflight_reviews`、`runner_execution_gates`、`runner_dry_runs`、`runner_execution_locks`、`runner_minimal_runs`：Runner 安全闸门和执行底座。
- `model_catalog`：受控模型目录。
- `project_plan_task_templates`：角色任务模板。

需要改变的是定位：

```text
旧：人手动审批 -> 人手动创建 gate/dry-run/lock -> 人手动执行
新：总控智能体按策略自动审批 -> 自动创建 gate/dry-run/lock -> 自动执行 -> 自动审计
```

## 8. 阶段 36：全自动模式宪法与权限边界

### 目标

引入全自动模式，但不能变成无审计的自由执行。

### 核心定义

默认模式仍是人工安全模式。

全自动模式必须由用户显式开启：

```text
用户一次性开启全自动模式
  -> 总控智能体获得最高项目权限
  -> 总控智能体按自动授权策略给其他角色智能体授权
  -> 所有授权和执行必须审计
```

### 必须具备的底线

- 一键停止。
- Git checkpoint / 回滚点。
- 自动授权记录。
- 执行记录。
- 文件改动记录。
- 命令执行记录。
- 模型调用审计。
- 不泄露 raw key、raw prompt、raw response。
- 总控智能体可以授权，但不能绕过审计。

### 建议新增概念

- `automation_modes`
- `auto_approval_policies`
- `controller_agents`
- `agent_permissions`
- `agent_delegations`

本阶段设计已完成。治理文档已同步，详见 `dev-docs/阶段36-全自动模式宪法与权限边界详细方案.md`。下一阶段进入阶段 37（想法引导官）。

## 9. 阶段 37：想法引导官

### 目标

新增“想法引导官”角色。用户只输入粗想法，它负责追问、扩展、补全边界，把想法变成可执行项目种子。

### 输出结构

```text
产品目标
目标用户
MVP 范围
明确不做什么
核心功能
页面 / 模块
技术约束
验收标准
风险点
```

### 要求

- 不能直接写文件。
- 不能执行 Runner。
- 不能自由调用 provider，必须走 Model Gateway。
- 输出可保存为项目种子或项目计划草案。
- 必须写入脱敏审计。

## 11. 阶段 39：总控智能体

### 目标

实现最高权限的“总控智能体”，它不是绕过规则，而是自动执行规则。

### 职责

- 选择工作流。
- 分配角色。
- 选择模型和 Skill。
- 给角色智能体发权限。
- 调用模型。
- 派发任务。
- 触发 Runner。
- 查看测试结果。
- 要求返修。
- 决定是否进入下一步。

### 禁止

- 不得无审计写文件。
- 不得无审计执行命令。

## 13. 阶段 41：多 Provider / 多 Key 管理

### 目标

支持 OpenAI、Claude、DeepSeek、Qwen、Gemini、中转站、本地模型等多 Provider。

### 分层

```text
Provider 服务商
  -> Credential 凭据引用
  -> Model 模型目录
  -> Role / Workflow 使用模型记录
```

### 安全要求

- 前端不能直接传 key。
- 前端不能自由传 raw base URL。
- 前端只能选择后端已启用的 provider/model/role/skill。
- 后端通过 `credential_ref` 找 key。
- SQLite 不明文存 raw key。
- 日志、审计表、前端响应不能出现 raw key、raw prompt、raw response。

### Provider 示例

- OpenAI
- Anthropic Claude
- DeepSeek
- Qwen
- Gemini
- OpenAI-compatible 中转站
- Local model

### 启动脚本定位

当前启动脚本里的 key 只作为开发兜底方式。正式能力应进入 Provider 凭据管理，启动脚本只负责打开桌面端。

## 14. 阶段 42：自动执行闭环

### 目标

让系统真正自动完成任务。

### 流程

```text
总控智能体授权
  -> 角色智能体生成方案 / 代码
  -> Runner 写文件
  -> Runner 跑测试
  -> 失败自动返修
  -> 再测试
  -> 通过后进入下一步
```

### 必须保留

- 自动授权记录。
- Runner 执行记录。
- 文件改动清单。
- 命令结果。
- 测试结果。
- 失败原因。
- 回滚入口。

### 权限扩大原则

只能从现有阶段 34 极小 Runner 逐步扩大：

```text
沙箱文件
  -> 白名单项目文件
  -> 单任务执行
  -> 单工作流执行
  -> 多步骤自动执行
```

不得一次性开放自由命令、批量改文件、Git commit/push。

## 15. 阶段 43：自动验收、回滚和交付

### 目标

让系统自己判断是否完成，并给用户可读交付结果。

### 必须输出

- 完成了什么。
- 改了哪些文件。
- 跑了哪些测试。
- 哪些测试通过 / 失败。
- 有哪些风险。
- 是否创建 checkpoint。
- 如何回滚。
- 下一步建议。

### 必须支持

- 自动验收报告。
- 失败自动返修次数上限。
- 达到上限后停止并报告。
- 一键停止。
- 一键回滚。

## 16. 前端原则

本项目是本地桌面 AI 协作控制台，不是大众 SaaS，也不是移动 App。

前端原则：

- 桌面端主控台优先。
- 浏览器只做预览。
- 视觉风格稳、清楚、可控。
- 不做花哨营销页。
- 用户可见文案默认中文。
- 英文只保留在模型 ID、Provider、命令、环境变量、路径、数据库 ID 等技术标识中。
- 不得把后端枚举、数据库字段名、内部状态裸露给用户。
