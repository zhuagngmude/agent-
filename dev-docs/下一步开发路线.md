## 老师反馈后的调整结论

本轮不再继续在旧手搓架构上叠功能。需要先重新确认项目工程方案，再决定如何初始化新工程和迁移旧原型价值。

已确认的调整方向：

- Agent 宪法、维护手册、交接说明和阶段路线默认使用中文。
- 前端不继续手搓，后续使用成熟 UI 技术栈。
- Web 和桌面端必须共用同一套 UI 源码。
- 桌面端使用 Rust/Tauri 作为宿主，Rust 负责本地能力，不重新手画 UI。
- 根目录重复前端、`apps/web` 前端、`packages/ui` 预留等目录需要重新规划，不能继续混乱叠加。
- 旧原型先归档和参考，不直接删除；等新架构确认后再清理。

## 当前最重要的边界

- 不写业务代码。
- 不继续手搓前端。
- 不继续扩展 Node.js 原生 HTTP 后端。
- 不初始化新工程，直到技术栈和目录结构确认。
- 不开放真实模型。
- 不开放真实 Runner。
- 不做云同步。
- 不做完整权限系统。
- 不修改保护路径：`design/image2/`、`data/mock/runtime-state.json`、`data/local/`、`logs/`、`.playwright-cli/`、`_internal/`。

## 候选技术栈口径

当前只是候选方案，尚未进入代码实现。

```text
前端 UI：
React + TypeScript + Vite + Ant Design

桌面端：
Tauri + Rust

后端：
Fastify 或 NestJS，替代 Node.js 原生 HTTP

本地数据库：
SQLite，配合 Prisma 或 Drizzle 这类成熟数据访问工具

后续云端数据库：
PostgreSQL / Supabase，等单机本地版稳定后再讨论

共享 UI：
packages/ui 作为唯一 UI 源码
apps/web 和 apps/desktop 只作为运行入口
```

## 下一步应该做什么

1. 先写清楚重新立项说明，明确旧 MVP 是验证原型，不作为正式工程架构继续扩展。
2. 对比并确认前端方案：React + Ant Design 是否作为最终 UI 技术栈。
3. 对比并确认桌面方案：Tauri + Rust 是否作为桌面宿主方案。
4. 对比并确认后端方案：Fastify / NestJS / 其他框架怎么选。
5. 确认数据库路线：本地 SQLite 继续保留，ORM / 查询层是否使用 Prisma 或 Drizzle。
6. 确认新目录结构，尤其是 `packages/ui` 作为唯一 UI 源码，`apps/web` / `apps/desktop` 作为运行入口。
7. 确认旧原型哪些内容归档、哪些迁移、哪些删除。

## 仍然不要做的事

- 不要直接开始 React 重写。
- 不要直接接 Tauri。
- 不要直接删除旧前端目录。
- 不要继续在原生 HTML / CSS / JavaScript 上加新功能。
- 不要继续扩展 Node.js 原生 HTTP 路由。
- 不要把真实模型调用、真实 Runner、云同步或完整权限系统混入重新立项阶段。
