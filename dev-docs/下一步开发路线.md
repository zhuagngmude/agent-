## 应用形态路线

当前看到的是电脑端 Web App，也就是面向电脑大屏使用的网页控制台，不是传统安装版桌面软件。

产品形态分三步：

1. **电脑端 Web App**
   - 当前阶段。
   - 重点是把页面、数据结构、状态机、Runner 审批流程做稳。
   - 直接打开 `frontend/index.html` 或后续通过本地 Web 服务访问。

2. **Web App + 本地 Runner**
   - 推荐的 MVP 正式形态。
   - Web App 负责界面、项目管理、Agent 调度、审批确认、知识库和记录。
   - 本地 Runner 负责读写本地文件、创建 Git checkpoint、生成补丁、跑测试。
   - Runner 不能绕过 Approval Service 自己执行。

3. **桌面应用封装**
   - 后续阶段。
   - 当前 Web App 稳定后，可以用 Tauri 或 Electron 封装为 `agent蜂群.exe`。
   - 桌面应用本质上仍复用 Web App 界面，但启动体验更像普通电脑软件。
   - 不建议现在立刻封装，因为核心流程和数据结构还在稳定。

结论：先做稳电脑端 Web App，再接本地 Runner，最后封装成桌面应用。

## 2026-06-10 Model Gateway dry-run 规格

- 已实现 `POST /api/model-gateway/dry-run` 的禁用态只读接口，并把 `scripts/verify-local-ui.ps1` 扩展到检查该接口。
- dry-run 只验证请求体、provider 支持情况、服务端环境变量是否存在和当前安全开关；它不会发起真实 provider 请求。
- dry-run 验收禁止写 SQLite、创建任务/审批/Runner job、触发 Agent、调用真实模型、记录 prompt/result。
- 这些禁止项只适用于 dry-run 阶段，不代表长期产品不做这些能力；后续必须按 manual connectivity test、logged model call、Agent orchestration、Runner integration 分阶段开放。
