- MVP-0.4 已验收；阶段 2 真实模型调用准入设计已收口；阶段 3 Agent Run 记录链已收口为本地 Mock / SQLite 流程。
- 已完成 MVP-0.3 主线：项目想法 -> `project_plan` 审批 -> 5 个 queued 任务 -> 5 条只读 Runner request 记录。
- 已完成 MVP-0.4 主线：execution request 审查视图、生命周期流转和 runtime events 审计闭环。
- 阶段 2 已完成 helper-only scaffold：`model_calls` 结构草案、Model Gateway 未来正式入口、provider 配置边界、redaction helper、禁用态 route、helper-only 写入 / 迁移草案。
- 阶段 3 已完成本地记录链：`agent_runs` Mock / SQLite API、失败注入、runtime event 审计、Web UI 记录页和 Mock / SQLite / UI 验证。
- 旧实现使用原生 HTML / CSS / JavaScript 和 Node.js 原生 HTTP，用于 MVP 验证；它不作为后续正式工程架构继续扩展。
- 技术栈已确认：桌面端主入口，Tauri 2 + Rust + React + TypeScript + Vite + Ant Design + SQLite + rusqlite，不单独起 Node/Fastify/NestJS/FastAPI 后端。

## 老师反馈后的调整结论

本轮不再继续在旧手搓架构上叠功能。需要先重新确认项目工程方案，再决定如何初始化新工程和迁移旧原型价值。

已确认的调整方向：

- Agent 宪法、维护手册、交接说明和阶段路线默认使用中文。
- 前端不继续手搓，后续使用成熟 UI 技术栈。
- 桌面端是主入口，Web 只作为辅助预览或后续扩展入口。
- Web 和桌面端必须共用同一套 UI 源码。
- 桌面端使用 Rust/Tauri 作为宿主，Rust 负责本地能力，不重新手画 UI。
- 根目录重复前端、`apps/web` 前端、`packages/ui` 预留等目录需要重新规划，不能继续混乱叠加。
- 旧原型归档方案已确认：旧原型冻结为参考资产，不直接删除；等新架构覆盖后再讨论清理。
- 新项目目录架构已确认：`packages/ui` 为唯一 UI 源，`apps/desktop` 为主入口，`apps/web` 为辅助预览入口。
- `packages/ui` 已从预留 README 升级为 React + TypeScript + Vite + Ant Design 的最小可运行骨架。

## 当前最重要的边界

- 不写完整真实业务功能；当前只允许已验收的本地 SQLite 读链路和写入 commands 记录层。
- 不继续手搓前端。
- 不继续扩展 Node.js 原生 HTTP 后端。
- 不绕过已确认方案扩大桌面宿主、数据库和旧原型迁移范围。
- 不开放真实模型。
- 不开放真实 Runner。
- 不做云同步。
- 不做完整权限系统。
- 不修改保护路径：`design/image2/`、`data/mock/runtime-state.json`、`data/local/`、`logs/`、`.playwright-cli/`、`_internal/`。

## 候选技术栈口径

已确认方案，并已进入最小工程实现。

```text
桌面主入口：
Tauri 2 + Rust

前端 UI：
React + TypeScript + Vite + Ant Design

本地数据库：
SQLite + rusqlite

后续云端数据库：
PostgreSQL / Supabase，等单机本地版稳定后再讨论

共享 UI：
packages/ui 作为唯一 UI 源码
apps/desktop 作为主入口，apps/web 作为辅助预览入口
```
