当前主线已经从“只读演示/审批链路”转为“本地多模型 Agent 调度系统”。

- AI 员工页已开始承载全技术栈固定员工池、项目专家推荐、职责边界、执行器配置、模型目录、绑定智能体和 Skill 配置预览。

## 近期优先级

### P0：多模型 Agent 调度主线

- 把执行器配置、模型目录、全技术栈固定员工池、项目专家推荐从前端模板落到 SQLite 数据模型。
- 建立 `executor_configs` / `model_catalog` / `agent_templates` / `project_agents` / `expert_recommendations` / `executor_skills` 或等价结构。
- 总控根据项目目标、技术栈、风险和阶段选择固定员工和项目专家。
- 任务拆解必须绑定 Agent 职责模块；跨模块任务必须由总控拆分或转派。
- 每个 Agent 绑定执行器和模型，但真实调用必须经过模型网关。
- Runner 执行记录必须能追踪任务、Agent、模型/执行器、文件改动和产物。

### P0：职责边界和安全

- 固定员工和专家 Agent 只能在自己的职责模块内工作。
- 前端、后端、数据库、桌面端、安全、测试、文档等边界必须进入任务分派规则。
- 高风险动作继续进入审批；不能因为多 Agent 调度绕过 Runner 或审批。
- UI 选择模型不能直接修改真实执行配置，必须走受控配置和审批。
- 禁止 Agent 自由 shell、Git commit/push、文件删除、保护路径写入和未受控网络请求。
- 越界防护必须落到后端强校验：agent_id、task_type、module_scope、target_path、forbidden_actions 都要参与判定，不能只靠前端提示或 prompt。

### P1：产品体验

- 主控台应能展示总控如何选择角色和专家。
- AI 员工页应区分“已接入员工”和“待接入模板”。
- 项目成员视图应展示当前项目真正参与的 Agent。
- 运行输出页应展示每个 Agent 的输入、输出、模型、状态和产物。
- 用户可见文案默认中文，内部 ID 不直接裸露给用户。

### P1：模型体验

- 系统设置里清楚展示当前模型是否可用。
- key 过期、余额不足、Base URL 错误、模型 ID 错误要区分提示。
- 支持不同 Agent 绑定不同模型或执行器。
- 不得把 API Key、Token、私钥、raw prompt、raw response 写入文档、数据库或日志。

### P2：质量体检和旧入口清理

- 检查文档、页面、Rust service、Tauri command、共享类型是否一致。
- 补齐最小验证：UI typecheck、Rust check、关键 Runner 路径手测。
- 把历史问题单改成“历史记录 + 当前状态”，避免下一位智能体重复修已修问题。

## 不要做

- 不要把真实 API Key 写进任何文档、代码、日志或数据库。
- 不要做与“本地多模型 Agent 调度系统”无关的旁支功能。
- 不要绕开总控、职责边界、模型网关、Runner、审批和运行记录。
- 不要把历史“只读/必须人工确认”阶段边界当成当前产品限制。
- 不要碰 `design/image2/`、`data/local/`、`logs/`、`_internal/`。
- 不要擅自 Git push、Git commit 或删除用户文件。

## 推荐下一轮执行顺序

1. 盘点现有 SQLite schema、migration、Rust commands 和模型目录实现。
2. 设计 SQLite 表：执行器配置、模型目录、Agent 模板、项目成员、专家推荐、Executor Skill。
3. 明确密钥策略：API Key 不落普通数据库、不进日志、不显示给前端。
4. 为 AI 员工页补 Tauri commands：读取/保存执行器非敏感配置、模型目录、员工绑定和 Skill 配置。
5. 将任务拆解结果绑定到具体 Agent，而不是只绑定少数旧角色。
6. 设计总控分派规则：项目类型 -> 角色集合 -> 任务职责边界。
7. 建立后端越界校验：agent_id、task_type、module_scope、target_path、forbidden_actions、审批状态参与 Runner 判定。
8. 将运行输出按 Agent 维度展示模型、输入、输出、产物和错误。
9. 给高风险 Agent 行为接入审批来源展示。
