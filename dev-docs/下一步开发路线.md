# 下一步开发路线

日期：2026-06-11

## 当前状态

- 当前阶段：MVP-0.4 已验收，阶段 2 真实模型调用准入设计已开始。
- 已完成 MVP-0.3 主线：项目想法 -> `project_plan` 审批 -> 5 个 queued 任务 -> 5 条只读 Runner request 记录。
- 已完成 MVP-0.4 主线：execution request 审查视图、生命周期流转和 runtime events 审计闭环。
- MVP-0.4 已有验收快照：见 [MVP-0.4验收记录.md](./MVP-0.4验收记录.md)。
- 阶段 2 已完成第一小步：`docs/data-model-draft.md` 中的 `model_calls` 结构草案和 `dev-docs/真实模型接入准入规格.md` 中的记录边界。
- 阶段 2 已完成第二小步：`dev-docs/Model Gateway正式入口设计.md` 中的未来入口、provider 配置来源和 Key 存储边界。
- 阶段 2 已完成第三小步：provider config resolver helper、redaction / response limiter helper、安全 `model_calls` 记录草稿，以及 `project-plan-model-requests` 禁用态 route 草案验证。
- Agent config 的安全闭环已经单独收口：apply / version history / rollback dry-run / rollback review 都保持只读或特性关闭。
- Model Gateway 仍是禁用态，真实模型调用和真实 Runner 执行都不开放。

## 当前最重要的边界

- 不做云同步。
- 不做完整权限系统。
- 不把只读 Runner 请求变成可执行命令。
- 不直接启用真实回滚写入，除非先补齐 feature flag、二次确认、Git checkpoint、rollback acceptance 和无副作用验收。

## 下一步应该做什么

1. 保持 MVP-0.3 / MVP-0.4 已完成链路稳定，并持续跑 Mock / SQLite 回归验证。
2. 阶段 2 下一小步：补 `model_calls` Mock / SQLite 记录写入设计和迁移草案，仍然不导入 SDK、不发起 provider 请求、不把结果写入项目计划审批。
3. 按 [真实模型接入准入规格](./真实模型接入准入规格.md) 继续收口 feature flag、脱敏、日志、成本和错误分类；真实 provider 调用必须单独提交和单独验收。
4. 按 [应用真正可用落地计划](./应用真正可用落地计划.md) 分阶段推进真实可用能力。
5. Runner 永远最后开；在模型调用链、Agent Run 记录链和执行计划审查链稳定前，不开放真实执行。
6. 继续把 Model Gateway 的禁用态、relay 说明和验证脚本分开维护，直到进入单独批准的真实模型调用阶段。

## 仍然不要做的事

- 把 0.5 / 0.6 路线重新写回当前主线。
- 真实 Agent 触发。
- 真实 Runner 执行。
- 本地项目文件写入。
- Git mutation。
- 云同步。
- 完整权限系统落地。
