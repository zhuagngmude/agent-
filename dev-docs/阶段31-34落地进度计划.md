# 阶段 31-34 落地进度计划

日期：2026-06-15

本文用于把 `agent蜂群` 从“项目计划和审查链路”推进到“最小真实 Runner 执行”。它是阶段节奏表，不替代每个阶段的详细设计文档。

当前准确状态：

- 阶段 31 已完成：Runner 执行许可 gate 已落库。
- gate 显式 `can_execute=false`、`stage_boundary_locked=true`。
- 真实 Runner 仍关闭。
- 仍不执行命令、不写用户项目文件、不改 Git、不调用模型、不创建可执行 Runner job。

## 总体节奏

| 阶段 | 名称 | 预计用时 | 核心作用 | 是否执行 Runner |
| --- | --- | --- | --- | --- |
| 31 | 执行许可 gate | 0.5-1 天 | 把“审查通过后允许进入执行准备”的状态落库 | 否 |
| 32 | 执行沙箱设计和只读 dry-run | 1-2 天 | 模拟 Runner 会做什么，只生成计划、命令清单、影响文件清单 | 否 |
| 33 | Git checkpoint + 文件范围锁 | 1-2 天 | 真正写文件前必须有 Git 检查点，并锁定白名单文件范围 | 否，仍只准备 |
| 34 | 最小真实 Runner 执行 | 2-4 天 | 极小范围、手动确认后执行一个任务 | 是，极小范围 |

保守总计：5-9 天可以到“最小可落地执行”。前提是每阶段实现后都审查、修问题、验证通过。

## 阶段 31：执行许可 gate

预计：0.5-1 天。

当前状态：已完成。

作用：

- 把“执行前审查已经通过，可以进入执行准备”的状态落库。
- 创建 `runner_execution_gates`。
- 支持撤销 gate。
- gate 仍被阶段边界锁定。

必须保持：

- `can_execute=false`
- `stage_boundary_locked=true`
- `status=blocked_by_stage_boundary` 或 `revoked`
- 不执行 Runner。
- 不执行命令。
- 不写文件。
- 不改 Git。

完成判断：

- `cargo test` 通过。
- 前端可以看到 gate 状态。
- 未批准 preflight 审批时不能创建 gate。
- 数据库层禁止 `can_execute=1` 和 `stage_boundary_locked=0`。

## 阶段 32：执行沙箱设计和只读 dry-run

预计：1-2 天。

作用：

- 读取阶段 31 的 gate。
- 生成 dry-run 计划。
- 展示 Runner 如果未来执行会做什么。
- 只产生结构化计划，不执行任何动作。

建议新增对象：

```text
runner_dry_runs
```

建议字段：

```text
id
project_id
gate_id
runner_request_id
task_id
status
planned_operations
planned_commands
planned_file_changes
allowed_files
blocked_reasons
can_execute
stage_boundary_locked
created_at
updated_at
```

阶段 32 的 `can_execute` 仍必须为 false。

允许做：

- 创建 dry-run 记录。
- 生成命令清单草案。
- 生成影响文件清单草案。
- 展示风险、阻断原因和安全说明。

禁止做：

- 不执行命令。
- 不写文件。
- 不创建 Git checkpoint。
- 不改 Git。
- 不运行测试。
- 不创建真实 Runner job。
- 不把 dry-run 状态改成 running / completed。

验收重点：

- 没有 gate 不能创建 dry-run。
- gate 被撤销不能创建 dry-run。
- gate 不是 locked 状态不能继续。
- dry-run 只读，不改变 task / runner_request / gate。
- 前端按钮叫“生成 dry-run 预演”，不能叫“执行”。

## 阶段 33：Git checkpoint + 文件范围锁

预计：1-2 天。

作用：

- 为真实执行前增加 Git checkpoint 规则和文件范围锁。
- 仍不执行真实写入。
- 仍不跑命令。
- 只验证：如果未来执行，必须满足哪些本地 Git 和文件范围条件。

建议新增对象：

```text
runner_execution_locks
```

建议字段：

```text
id
project_id
dry_run_id
gate_id
allowed_files
denied_paths
requires_git_checkpoint
checkpoint_strategy
workspace_status
blocked_reasons
created_at
updated_at
```

阶段 33 是否创建真实 Git checkpoint：

- 第一版建议不创建真实 checkpoint，只记录 `checkpoint_required` 和 `checkpoint_strategy`。
- 如果确实要创建真实 checkpoint，必须单独拆一个子阶段并先人工确认。

允许做：

- 计算允许文件集合。
- 计算禁止路径集合。
- 检查计划文件是否全部在白名单内。
- 展示 checkpoint 要求。
- 展示工作区状态要求。

禁止做：

- 不执行 `git` 命令修改仓库。
- 不创建 commit。
- 不 stash。
- 不 reset。
- 不写项目文件。
- 不运行测试。
- 不打开真实 Runner。

验收重点：

- 计划外文件被拒绝。
- 保护路径被拒绝。
- 空文件范围被拒绝。
- denied_paths 包含项目保护路径。
- 前端只显示“准备要求”，不显示执行按钮。

## 阶段 34：最小真实 Runner 执行

预计：2-4 天。

作用：

- 第一次开放极小范围真实执行。
- 只允许一个任务。
- 只允许手动二次确认。
- 只允许沙箱目录或测试目录。
- 只允许白名单命令。

进入条件：

- 阶段 31 gate 完成并通过审查。
- 阶段 32 dry-run 完成并通过审查。
- 阶段 33 文件范围锁完成并通过审查。
- 工作区状态安全。
- 用户明确确认可以进入最小真实执行阶段。

第一版建议范围：

```text
只允许修改 sandbox/ 或 tests/sandbox/ 之类明确目录。
只允许一个 runner_request。
只允许一个 task。
只允许一个执行批次。
必须二次确认。
执行前必须有 checkpoint 或等价可回退方案。
```

允许命令建议：

```text
git status --short
git diff --stat
npm run typecheck
npm run build
cargo fmt --check
cargo check
cargo test
```

第一版仍不建议允许：

```text
git commit
git push
git reset
git clean
删除文件
任意 shell
网络请求
写保护路径
```

验收重点：

- 执行前显示完整计划。
- 执行前二次确认。
- 执行后只改白名单路径。
- 执行后有 diff 摘要。
- 执行失败立即停止。
- 不自动提交 Git。
- 不自动继续下一个任务。

## 推荐推进方式

每个阶段都按这个节奏走：

1. 先写阶段设计文档。
2. 交给低级智能体实现。
3. 实现后让高级审查。
4. 发现问题直接修。
5. 跑完整验证。
6. 再进入下一阶段。

不要跳过阶段 32 和 33 直接做阶段 34。
