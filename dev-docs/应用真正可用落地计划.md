# 应用真正可用落地计划

日期：2026-06-11

本文是内部工程计划，用来说明 `agent蜂群` 从当前本地 MVP-0.4 稳定态走到“真正能帮助用户开发项目”的路线。它不开放真实 Runner，不开放真实模型调用，也不引入云同步或完整权限系统；具体功能必须在后续独立阶段中重新设计、验证和审批。

## 当前判断

- MVP-0.3 已完成：项目计划审批 -> Agent 自动分工 -> 只读 Runner request queue。
- MVP-0.4 已完成：execution request 审查视图、生命周期流转和 runtime events 审计闭环。
- 当前阶段：MVP-0.4 已验收，阶段 2 真实模型调用准入设计已收口。
- 阶段 2 已完成四小步：`model_calls` 结构草案、Model Gateway 未来正式入口 / provider 配置 / Key 存储边界、provider config resolver / redaction helper / 禁用态 route 草案验证、helper-only `model_calls` 写入 / 迁移草案 scaffold；阶段 23 已在新 Tauri/Rust 架构中建立 `model_calls` SQLite 审计表。当前仍不写 `model_calls` 业务记录、不导入 SDK、不读取或返回 raw key、不发 provider 请求。
- 真实 Runner 执行仍关闭。
- 真实模型调用仍关闭。
- 云同步和完整权限系统仍关闭。
- Model Gateway 仍是禁用态说明和边界验证，不是正式模型调用骨架。

## 真正可用的定义

真正可用不是仅能演示 UI，也不是直接让 Runner 自动写代码。当前建议的可用定义是：

```text
用户可以在本地绑定一个真实代码仓库，输入项目需求；
系统通过后端固定形态的模型调用生成计划、拆分任务并形成 Agent Run 记录；
用户审查执行计划、影响文件、diff preview、测试命令和风险；
用户批准后，本地 Runner 在锁定范围内创建 Git checkpoint、修改文件、运行白名单测试；
Web 控制台展示结果、日志、diff 摘要、测试结果、失败原因和 runtime events。
```

## 编码前五问

后续任何进入实现的批次，都必须先回答这五个问题：

1. 真正要解决的问题是什么？
2. 谁创建、谁调用、谁消费，调用链是否闭合？
3. 现有系统里有没有已经承担同一职责的资源？
4. 更简单或更保守的替代方案是什么，为什么不用或怎么降级使用？
5. 最大回归风险是什么，用哪些验证挡住？

没有回答清楚前，不进入代码实现。

- 本地启动、停止、状态查看脚本稳定。
- UI 能演示项目计划审批、任务拆分、只读 Runner request、生命周期和 runtime events。

不做：

- 不接真实 Runner。

## 阶段 2：真实模型调用的最小闭环

目的：让系统从本地确定性模板走向后端受控模型调用，但仍不允许前端自由拼请求，也不允许泄漏密钥。

进入该阶段前，必须先通过 [真实模型接入准入规格](./真实模型接入准入规格.md)。准入规格通过不等于真实调用已开启；真实 provider 请求必须在后续单独实现批次中完成。

当前进度：

- 已完成 `docs/data-model-draft.md` 中的 `model_calls` 结构草案。
- 已在 [真实模型接入准入规格](./真实模型接入准入规格.md) 中补充模型调用记录边界。
- 已完成 [Model Gateway 正式入口设计](./Model%20Gateway正式入口设计.md)，固定候选 route、内部调用链、server env 配置来源和 Key 存储边界。
- 已完成 provider config resolver helper、redaction / response limiter helper、安全 `model_calls` 记录草稿和禁用态 route 草案验证。
- `model_calls` SQLite 审计表已由阶段 23 migration 003 建立；helper-only 草案保持 `canWrite=false`，后续若要进入真实写入，必须单独开实现批次。
- 当前仍不写 `model_calls` 业务记录、不写 runtime events、不导入 SDK、不读取或返回 raw key、不发 provider 请求。

交付物：

- 后端 Model Gateway 正式入口设计。
- Provider 配置只在后端保存和读取。
- API Key 加密或本地安全存储方案。
- 固定请求类型，例如 `project_plan_generation`、`task_breakdown`、`review_summary`。
- 模型调用记录 provider、model、token usage、成本估算、状态和错误。
- 日志脱敏和错误脱敏。

安全闸口：

- 前端不得接触完整 API Key。
- 禁止用户自由提交 base URL、headers、prompt body 或 provider request body。
- 禁止把 raw secret、完整 prompt 或敏感输出写入公开日志。
- 先接一个 provider，验证通过后再扩展。

验收：

- 无 Key 时返回明确 blocked 状态。
- 错 Key 时失败可见且不泄漏密钥。
- 成功时能生成真实 project plan。
- 验证脚本覆盖成功、失败、脱敏和禁越界。

## 阶段 3：Agent Run 真实流程

目的：让 Architect、Scheduler、Specialist 和 Reviewer 不再只是 UI 概念，而是可追踪、可失败、可审查的运行链。

交付物：

- `agent_runs` 数据结构和 API 草案。
- Architect 生成总体方案。
- Scheduler 拆分任务。
- Frontend / Backend / QA / Docs Agent 生成各自建议。
- Reviewer 汇总风险和执行建议。
- 每个 Agent Run 记录输入摘要、输出摘要、状态、模型、token、成本、错误和 parent run。

不做：

- Agent 不直接写文件。
- 子 Agent 不执行代码。
- Agent 不自批。
- 失败后不自动扩大权限。

验收：

- 一个真实需求可以完整跑出 Agent 链。
- 失败停在失败节点，错误原因可见。
- UI 能看懂每个 Agent 做了什么。

## 阶段 4：本地 Runner 连接但不写文件

目的：建立 Web App 与本地 Runner 的真实连接和能力声明，但仍不执行本地写入。

交付物：

- 本地 Runner 服务或 CLI。
- Runner 注册、心跳和离线状态。
- workspace 绑定和本地项目路径别名。
- Runner 上报 Git 状态、能力声明、脏工作区状态和可用验证命令。
- Web UI 显示 Runner 在线状态和安全边界。

不做：

- 不写文件。
- 不执行命令。
- 不修改 Git。
- 不发任意网络请求。

验收：

- Runner 在线 / 离线可见。
- 工作区路径不泄漏敏感信息。
- 脏工作区能阻止后续执行计划。
- 断线后状态可恢复。

## 阶段 5：执行计划与 diff preview

目的：让 Executor Agent 生成可审查的代码修改计划，但仍不落盘。

交付物：

- `code_execution_request` 草案。
- change plan。
- affected files。
- diff preview。
- test command preview。
- risk level。
- second confirm。
- approval 绑定。
- Git checkpoint 要求。

安全闸口：

- 没有 approval 不能进入执行队列。
- 没有 affected files 不能执行。
- 没有 diff preview 不能执行。
- 高风险必须二次确认。
- 计划外文件不得进入执行范围。

验收：

- 用户能看到将改哪些文件、为什么改、怎么测。
- 不允许空白授权。
- 不允许自由命令执行。

## 阶段 6：受限真实 Runner 执行

目的：第一次允许 Runner 真实修改代码，但范围必须极窄，并且所有行为可审计。

第一版只允许：

- 创建 Git checkpoint。
- 修改审批列出的文件。
- 运行白名单测试命令。
- 上传 diff 摘要、测试结果和失败原因。

命令白名单示例：

```text
node --check <known-file>
npm test
powershell -ExecutionPolicy Bypass -File scripts\verify-*.ps1
git status --short
git diff --stat
```

禁止：

- 任意 shell。
- 删除文件。
- 网络请求。
- Git push。
- `git reset --hard`。
- 触碰 `_internal/`、`design/image2/`、`data/local/`、`logs/`、`.playwright-cli/`、`data/mock/runtime-state.json`。

验收：

- 执行前 checkpoint 存在。
- 执行后 diff 文件集合是审批文件集合的子集。
- 测试结果可见。
- 失败即停止。
- 回滚必须重新审批。

## 阶段 7：单人本地可用版

目的：让一个用户可以在本机真实使用系统辅助开发项目。

交付标准：

- 能绑定一个真实 repo。
- 能输入需求。
- 能生成 Agent 方案。
- 能拆任务。
- 能生成修改计划。
- 能审查 diff。
- 能批准 Runner 执行。
- 能看到测试结果。
- 能看到完整日志和审计。
- 能失败后人工处理。
- 能安全停止。

## 阶段 8：团队和云端版

目的：在单人本地可用版稳定后，再推进团队协作和云端产品化。

交付物：

- 登录。
- Workspace。
- 多用户。
- 邀请制。
- Supabase / PostgreSQL 或等价后端存储。
- RBAC + ABAC。
- API Key 加密管理。
- 云端 Agent Run 历史。
- 多项目管理。
- Runner 连接授权。
- 审计日志。

前置条件：

- 单人本地可用版已经稳定。
- Runner 安全执行链已经通过验收。
- 密钥、日志、导出和权限边界已经经过专项验证。

## 总顺序

```text
稳定当前基座
-> 一个 provider 的真实模型调用
-> Agent Run 真实链路
-> Runner 在线连接
-> 执行计划和 diff preview
-> 受限文件写入
-> 受限测试命令
-> 回滚审批
-> 单人本地可用版
-> 团队云端版
```

Runner 永远最后开；真实执行之前，必须先完成模型调用链、Agent Run 记录链和执行计划审查链。

## 每阶段记录要求

每个阶段结束时必须同步：

- 相关 `dev-docs/` 内部计划或交接文档。
- 相关 `docs/` 对外 API / 验收文档，仅当用户可见行为或正式契约变化时更新。
- 相关验证脚本说明。
- 一个清晰 commit。

不得只改代码不回写文档。
