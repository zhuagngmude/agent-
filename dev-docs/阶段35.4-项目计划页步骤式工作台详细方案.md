# 阶段 35.4 项目计划页步骤式工作台详细方案

## 1. 背景

当前 `agent-swarm` 已完成阶段 0-35.3：

- 阶段 25/35：真实模型调用、受控模型目录和模型选择已落地。
- 阶段 26-29：真实模型草案可保存、审批、生成任务和只读 Runner 请求，并可查看执行预览。
- 阶段 30-34：执行前审查、执行许可、只读预演、执行范围锁、阶段 34 极小范围真实 Runner 已落地。
- 阶段 35.1：前端中文化做过一轮收口。
- 阶段 35.3：桌面端外壳、浏览器预览提示、总览页和设置页已收口。

但 `packages/ui/src/pages/ProjectPlanPage.tsx` 已经从早期验证页长成了完整控制台，一页内堆叠了草案、模型、审批、模板、任务、Runner 请求、preflight、gate、dry-run、lock、minimal run 等所有入口。用户会看到一长串卡片，不容易判断自己当前在哪一步。

本阶段目标不是重写业务链路，而是把现有项目计划页整理成步骤式工作台，让用户按流程操作，同时清理该页残留的乱码、英文工程词和阶段边界冲突文案。

## 2. 本阶段目标

把 `ProjectPlanPage` 收口成 5 步工作台：

```text
1. 草案
2. 审批
3. 任务与队列
4. 执行准备
5. 最小执行
```

用户进入页面后应能一眼看懂：

1. 现在处于项目计划流程的哪一步。
2. 当前步骤会不会调用真实模型。
3. 当前步骤会不会执行 Runner。
4. 哪些按钮只是生成记录，哪些按钮会进入阶段 34 最小执行。
5. 浏览器预览和桌面真实模式的写入边界仍然一致。

## 3. 非目标

本阶段禁止扩大范围：

- 不改 Rust 后端。
- 不改 Tauri command。
- 不改数据库 migration。
- 不改 `packages/shared` 类型，除非现有前端类型缺失导致无法通过 typecheck。
- 不改模型调用逻辑。
- 不改 Runner 权限、命令白名单、沙箱路径或安全闸门。
- 不新增依赖。
- 不删除历史阶段文档。
- 不提交 Git。

本阶段只做前端信息架构、页面分组、可见文案和轻量样式收口。

## 4. 建议修改文件

优先只改：

```text
packages/ui/src/pages/ProjectPlanPage.tsx
```

如确实需要补充步骤式布局样式，可小范围修改：

```text
packages/ui/src/theme/global.css
```

不建议改其他文件。若低级智能体认为必须修改其他文件，必须先说明原因，并保证不涉及后端业务链路。

## 5. 页面结构方案

建议使用 Ant Design `Tabs` 做步骤式工作台。

原因：

- 当前页面卡片很多，`Tabs` 比 `Steps` 更适合把已有卡片按区域搬进去。
- 可以保留现有 handler、state、table columns、load 函数，不需要重写业务。
- 用户可以直接切换查看历史结果，不会被强制线性推进。

建议顶部结构：

```tsx
<Space direction="vertical" size={16} style={{ width: "100%" }}>
  <PageHeaderLikeArea />
  <Alert ... />
  <Tabs
    items={[
      { key: "draft", label: "1 草案", children: ... },
      { key: "approval", label: "2 审批", children: ... },
      { key: "tasks", label: "3 任务与队列", children: ... },
      { key: "prepare", label: "4 执行准备", children: ... },
      { key: "run", label: "5 最小执行", children: ... },
    ]}
  />
</Space>
```

不要为了步骤式工作台重写所有业务函数。优先把现有卡片移动到不同 tab 内。

## 6. 五个步骤的具体内容

### 6.1 步骤一：草案

放入这些现有区域：

- `生成项目计划草案`
- `真实模型草案预览`
- `保存为项目计划草案`
- `任务角色模板`

建议标题：

```text
草案
```

建议说明：

```text
先生成本地草案或真实模型草案。真实模型只能从模型目录选择，并通过 Model Gateway 人工触发。
```

真实模型文案必须保持：

```text
真实模型草案只作为预览，不会直接创建任务、审批或 Runner 请求。
前端不接收 key、base URL、prompt、header 或 provider body。
```

注意：

- 模型下拉框仍只能来自后端模型目录。
- 不允许新增自由输入模型名、key、base URL、provider URL。
- `audit_record_id` 不要展示明细 ID，只可显示“已写入安全审计记录”。

### 6.2 步骤二：审批

放入这些现有区域：

- `计划草案`
- `批准生成任务`

建议标题：

```text
审批
```

建议说明：

```text
选择草案并二次确认后，系统会创建任务和只读 Runner 请求。此步骤不会执行 Runner、不会写文件、不会改 Git。
```

要求：

- 草案来源显示中文：
  - `本地模板`
  - `真实模型`
- 草案状态显示中文，不要裸露后端状态枚举。
- 审批按钮仍必须依赖 `canWrite`、选中草案、二次确认。
- 二次确认文案保持明确：

```text
我确认只生成任务和只读队列，不执行 Runner
```

### 6.3 步骤三：任务与队列

放入这些现有区域：

- `刚生成的内存预览`
- `已生成任务和只读 Runner 请求`
- `只读 Runner 请求队列`

建议标题：

```text
任务与队列
```

建议说明：

```text
查看审批后真实落库的任务和只读 Runner 请求。这里只展示队列，不执行 Runner。
```

要求：

- `Runner request` 用户可见文案统一为 `Runner 请求`。
- `runner_requests`、`task_id`、`approval_id` 这类内部字段不要作为标题裸露。
- 表格列名中文化：
  - `任务`
  - `角色`
  - `状态`
  - `优先级`
  - `Runner 请求`
  - `只读`
- 如果需要显示 ID，只在详情或小号文本里显示，并明确是记录 ID。

### 6.4 步骤四：执行准备

放入这些现有区域：

- `执行前审查`
- `执行许可`
- `只读预演`
- `执行范围锁`

建议标题：

```text
执行准备
```

建议说明：

```text
执行准备会创建审查、许可、只读预演和范围锁记录。当前步骤不会执行 Runner、不会写文件、不会改 Git。
```

阶段口径必须改准确：

- 不要写：

```text
当前 Runner 已被系统关闭
当前 Runner 仍被系统边界锁定
```

- 改为：

```text
当前步骤不会执行 Runner。阶段 34 之前的准备记录只用于审查和锁定范围。
```

或：

```text
执行准备只生成记录和预演结果，不执行命令、不写文件、不改 Git。
```

各子区域建议文案：

执行前审查：

```text
创建执行前审查记录，并生成待审批项；不会执行 Runner。
```

执行许可：

```text
在执行前审查通过后创建执行许可记录；该记录本身不执行 Runner。
```

只读预演：

```text
生成计划、命令清单和影响文件清单；不会真正执行命令。
```

执行范围锁：

```text
锁定允许文件和禁止路径；不会创建 Git 检查点，不会写文件。
```

### 6.5 步骤五：最小执行

放入这些现有区域：

- `阶段 34 最小 Runner 执行`
- `最小执行记录`

建议标题：

```text
最小执行
```

建议说明：

```text
阶段 34 会真实执行极小范围 Runner：只允许临时沙箱路径、固定 Git 只读命令和人工二次确认。
```

要求：

- 这是全页唯一可以明确提示“会真实执行”的区域。
- 必须保留危险色或醒目 warning。
- 按钮文案建议：

```text
执行阶段 34 最小 Runner
```

- 二次确认文案保持：

```text
我确认执行阶段34最小Runner，只允许沙箱范围
```

- 不要暗示会自动提交 Git。
- 不要暗示会写用户项目源码。
- 明确只写临时沙箱产物。

## 7. 中文化和乱码收口

`ProjectPlanPage.tsx` 必须清理用户可见乱码。

重点检查：

- 常量里的确认文本。
- `statusLabel()`。
- `reasonLabel()`。
- `checkpointStrategyLabel()`。
- `optionalTextLabel()`。
- 所有 `Card title`。
- 所有 `Alert message/description`。
- 所有 `Table title/locale/columns.title`。
- 所有按钮文案。
- 所有 `message.success/error/warning`。

要求：

- 用户可见文案默认中文。
- 后端枚举、数据库字段名、内部状态不能直接裸露给用户。
- 英文只保留在模型 ID、命令、环境变量、Provider、路径、数据库 ID 等技术标识中。
- 未识别的后端枚举不要原样展示，建议显示：

```text
未识别状态
未识别原因
```

允许保留的英文示例：

```text
gpt-5.4-mini
openai_compat
git status --short
AGENT_SWARM_OPENAI_COMPAT_API_KEY
runner_request_id
```

但这些英文不应出现在大标题和操作说明中，除非是在技术详情里。

## 8. 状态和原因映射要求

若现有映射不完整，补全常见值。

状态建议：

```text
pending -> 待处理
queued -> 排队中
approved -> 已批准
rejected -> 已拒绝
draft_ready -> 草案已生成
success -> 成功
failed -> 失败
blocked -> 已阻断
blocked_by_stage_boundary -> 已被阶段边界阻断
revoked -> 已撤销
locked -> 已锁定
succeeded -> 成功
running -> 运行中
idle -> 空闲
unknown -> 未知
```

阻断/失败原因建议：

```text
runner_execution_disabled_by_stage_boundary -> 当前阶段边界禁止执行
runner_execution_disabled_until_gate_approved -> 执行许可未通过
runner_execution_not_allowed_by_stage -> 当前阶段不允许执行
command_failed -> 命令执行失败
scope_violation -> 超出允许范围
audit_write_failed -> 审计写入失败
provider_error -> Provider 返回错误
provider_config_error -> Provider 配置错误
response_too_large -> 响应过大
invalid_request -> 请求无效
invalid_state -> 状态无效
feature_disabled -> 功能开关未开启
```

检查点策略建议：

```text
manual_checkpoint_required_before_stage34 -> 阶段 34 前需要人工检查点
manual_checkpoint_required -> 需要人工检查点
not_required_for_readonly_preview -> 只读预演不需要检查点
```

## 9. 交互和刷新要求

本阶段必须保留现有行为：

- 页面初始加载仍加载草案、模板、模型、preflight、gate、dry-run、lock、minimal run 等数据。
- 创建、撤销、审批成功后仍刷新对应列表。
- 浏览器预览模式下写入按钮仍不可用。
- `canWrite` 判断不能被移除。
- 所有 handler 依赖数组要包含实际使用的状态和函数，避免闭包拿旧值。
- 不要把后端错误原文直接展示给用户。

建议操作后刷新：

- 保存真实模型草案后：刷新草案列表。
- 审批草案后：刷新草案、执行预览、Runner 请求。
- 创建 preflight 后：刷新 preflight。
- 创建 gate 后：刷新 gate。
- 创建 dry-run 后：刷新 dry-run。
- 创建 lock 后：刷新 lock。
- 创建 minimal run 后：刷新 minimal run。

4. 用户可见文案默认中文。

7. Runner 边界不变：只有阶段 34 最小执行区提示会真实执行。
8. 执行准备区明确不会执行 Runner、不会写文件、不会改 Git。
9. 浏览器预览模式写入按钮仍不可用。
10. 不改 Rust、数据库、模型调用、Runner 权限。
11. `npm run typecheck` 通过。
12. `npm run build` 通过。
13. `git diff --check` 通过。
14. 乱码和旧口径扫描无用户可见命中。
