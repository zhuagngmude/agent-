# 真实模型接入准入规格

日期：2026-06-11

本文是内部工程准入规格，用来说明 `agent蜂群` 在 MVP-0.4 稳定化之后，如何进入真实模型调用的最小闭环设计。本文不启用真实模型调用，不导入 provider SDK，不发起 provider 网络请求，不开放 Runner 执行，也不引入云同步或完整权限系统。

## 编码前五问

### 1. 真正要解决的问题是什么？

当前要解决的不是“马上调用真实模型”，而是把真实模型接入的准入条件、调用链、禁区和验收方式写清楚。只有准入规格通过后，后续才能单独进入实现批次。

### 2. 谁创建、谁调用、谁消费？

- 用户在 Web UI 中创建项目计划请求或后续 Agent Run 请求。
- API 只能通过后端 Model Gateway 创建固定形态的模型请求。
- Provider adapter 只能由 Model Gateway 调用，不允许 UI、Agent、Runner 或任务状态机直接调用 provider。
- 项目计划、Agent Run、任务拆分和审查摘要消费模型结果的结构化摘要。
- UI 只能消费脱敏后的状态、摘要、成本、错误和审计信息。

调用链必须闭合为：

```text
Web UI intent
-> API route
-> Model Gateway request builder
-> provider adapter
-> redaction / response limiter
-> model call record
-> structured consumer
```

任何绕过 Model Gateway 的 provider 调用都禁止。

### 3. 现有系统里有没有已经承担同一职责的资源？

已有资源：

- `services/api/model-gateway.js`：禁用态 provider 元数据、preflight、dry-run 和 connectivity-test 边界。
- `services/api/model-gateway-adapters.js`：禁用态 provider adapter registry 和 OpenAI-compatible relay interface。
- `scripts/verify-model-gateway.ps1`：验证 Model Gateway 仍然 disabled、无副作用、无真实 provider 请求。
- `docs/relay-provider-info-checklist.md`：OpenAI-compatible relay 非秘密信息收集清单。
- `docs/cheng-relay-manual-ping-spec.md`：cheng.pink relay 固定 minimal ping 规格。
- `docs/deepseek-provider-info-checklist.md`：DeepSeek provider 非秘密信息清单。
- `dev-docs/应用真正可用落地计划.md`：真实可用应用的阶段路线。

后续实现必须扩展这些现有边界，不能另起一个绕过它们的新模型调用入口。

### 4. 更简单或更保守的替代方案是什么？

更保守方案是继续保持本地确定性模板，不接真实模型。

本阶段采用该保守方案的变体：只写准入规格和验收要求，不实现 SDK、不打开 feature flag、不发网络请求。这样可以先消除路线误判和安全缺口。

### 5. 最大回归风险是什么？怎么挡住？

最大风险：

- 前端接触完整 API Key。
- 接受自由 prompt、headers、base URL 或 provider body。
- 把 raw prompt、raw provider response、raw error 或 key fragment 写进日志、数据库或 UI。
- 真实模型调用触发 Agent、任务、审批或 Runner 执行。
- 复用 disabled connectivity-test flag 直接打开真实业务调用。

验证方式：

- Model Gateway 验证脚本必须继续覆盖 disabled/no-side-effect 边界。
- 新增真实调用前必须先有无 Key、错 Key、feature disabled、unsupported provider/model、timeout、provider error、response limit 和 redaction 的测试。
- 任何真实 provider 请求都必须在单独提交中实现，并且默认 feature flag 关闭。

## 当前准入状态

- MVP-0.3 项目计划审批闭环已完成。
- MVP-0.4 execution request 生命周期和 runtime events 审计闭环已完成。
- Model Gateway 当前仍是禁用态。
- `AGENT_SWARM_ENABLE_MODEL_CONNECTIVITY_TEST` 当前只报告请求标志，不允许真实 provider 请求。
- `services/api/model-gateway-project-plan.js` 已提供 helper-only 的 `project_plan_generation` 请求准入构造器，默认仍返回 `feature_disabled`，不发 provider 请求。
- 真实模型调用尚未开启实现。

## 第一条真实模型链路

第一条真实模型链路只允许选择一个用途：

```text
project_plan_generation
```

原因：

- 它已有本地确定性模板作为回退。
- 结果可以先进入 `project_plan` 审批草案，而不是直接创建任务或 Runner request。
- 用户审批前不会触发 Agent、Runner、Git、文件写入或本地命令。

暂不做：

- `task_breakdown`。
- `review_summary`。
- 自由聊天。
- 任意 prompt playground。
- Agent 自动连续调用。
- Runner 执行建议自动落地。

## Provider 顺序

第一版只接一个 provider。

候选优先级：

1. `openai_compat` relay：已有 cheng.pink minimal ping 规格和 base URL 规范化草案。
2. `deepseek`：已有公开文档信息清单，但尚未进入 adapter registry。
3. 官方 OpenAI / Anthropic / Google：当前只保留 disabled metadata，不作为第一条真实链路。

最终选择必须在实现前写入本文件或单独 provider 准入文档。

## Feature Flag

真实业务模型调用必须使用独立 feature flag，不得复用 `AGENT_SWARM_ENABLE_MODEL_CONNECTIVITY_TEST`。

建议名称：

```text
AGENT_SWARM_ENABLE_REAL_MODEL_PROJECT_PLAN=false
```

默认值必须为 false。

要求：

- feature flag 关闭时，所有真实业务模型调用返回 blocked / feature_disabled。
- feature flag 开启但缺少 key 时，返回 missing_key。
- feature flag 开启但 provider 未准入时，返回 unsupported_provider。
- feature flag 开启但请求用途不是固定用途时，返回 invalid_request。
- feature flag 开启也不得触发 Runner、写项目文件、改 Git 或创建可执行 Runner job。

## 请求形态

第一条真实业务请求只能由后端构造。

本地 API 可以接收：

```json
{
  "projectId": "project_agent_swarm",
  "purpose": "project_plan_generation",
  "idea": "Build a local customer lead tracker",
  "constraints": "Mock/SQLite first; no Runner execution",
  "requestedBy": "local_user"
}
```

禁止客户端提交：

- API key。
- base URL。
- headers。
- provider request body。
- system prompt。
- free-form prompt。
- stream 设置。
- tool / function call 设置。
- files。
- Runner job id。
- arbitrary provider options。

## Provider 请求构造规则

Provider 请求必须由 Model Gateway 的 request builder 固定构造。

要求：

- 固定 purpose。
- 固定 provider。
- 固定 model 或后端配置白名单 model。
- 固定 timeout。
- 固定 response body limit。
- 不接受前端 headers。
- 不接受前端 base URL。
- 不接受前端 API key。
- 不接受前端 prompt 模板。
- 不允许工具调用。
- 不允许文件输入。
- 不允许 streaming 作为第一版。

## 结果处理

第一版允许消费结构化计划摘要，但不得保存或展示 raw provider response。

允许保存：

- `modelCallId`
- `provider`
- `model`
- `purpose`
- `status`
- `durationMs`
- `tokenUsage`，如果 provider 返回且已安全解析
- `costEstimate`，如果本地能安全估算
- `redactionApplied`
- `structuredSummary`
- `errorCategory`
- `createdAt`

禁止保存或返回：

- raw API key。
- key suffix / masked key fragment。
- raw request headers。
- raw response headers。
- raw provider body。
- raw provider error。
- raw prompt。
- full prompt template。
- model reasoning text。
- 未经过结构化和脱敏的模型输出。

## 数据写入边界

真实模型调用可以在后续实现中写入模型调用记录，但不得在第一步直接改项目状态。

允许的后续写入必须单独设计：

- `model_calls` 或等价 SQLite / Mock 记录。
- runtime event，记录 model call 状态变化。
- `project_plan` 审批草案中的结构化计划摘要。

禁止：

- 直接创建任务。
- 直接创建 Runner job。
- 直接触发 Agent。
- 直接改 Agent 配置。
- 直接写项目文件。
- 修改 Git。
- 执行 Runner。

## 错误分类

第一版错误分类必须粗粒度：

```text
feature_disabled
missing_key
invalid_request
unsupported_provider
unsupported_model
timeout
provider_unavailable
network_error
response_body_limit
redaction_failed
unknown
```

禁止把 provider 原始错误体、原始堆栈、请求头、响应头或 secret 值传给前端。

## 验收矩阵

实现前必须补齐验证脚本，至少覆盖：

- feature disabled。
- missing key。
- unsupported provider。
- unsupported model。
- invalid purpose。
- timeout simulation。
- provider error simulation。
- response body limit。
- redaction applied。
- no raw key in response。
- no raw prompt in response。
- no provider body stored。
- no Runner job created。
- no task created before approval。
- no Agent triggered。
- no Git mutation。
- no project file write。

## 实现顺序

1. 保持现有 disabled Model Gateway 通过验证。
2. 增加真实模型业务调用的 request builder helper，但仍返回 feature_disabled。（已落地：`model-gateway-project-plan.js`）
3. 增加验证脚本覆盖准入矩阵，不发真实网络请求。（已落地：`scripts/verify-real-model-admission.ps1`）
4. 增加模型调用记录结构草案和数据模型文档。
5. 在单独提交中接入一个 provider 的真实调用，默认 feature flag 关闭。
6. 只在用户明确批准并提供本地环境配置后，运行一次固定用途真实调用。
7. 将真实调用结果只进入 `project_plan` 审批草案，不直接创建任务或 Runner request。

## 当前不做

- 不导入 provider SDK。
- 不发真实 provider 网络请求。
- 不读取或展示原始 API Key。
- 不记录 raw prompt 或 raw provider response。
- 不开放自由 prompt。
- 不开放任意 provider URL / headers / body。
- 不触发 Agent。
- 不执行 Runner。
- 不写项目文件。
- 不改 Git。
- 不做云同步。
- 不做完整权限系统。

## 文档同步要求

当真实模型调用进入实现时，必须同步：

- `docs/api-draft.md`：正式 API 契约变化。
- `docs/data-model-draft.md`：模型调用记录或 runtime event 结构。
- `scripts/README.md`：新增验证脚本。
- `dev-docs/下一步开发路线.md`：阶段状态和下一步。
- `dev-docs/AI开发维护手册.md`：后续 AI 操作边界。

没有这些同步，不允许提交实现。
