# 阶段 17：长期分层边界设计

日期：2026-06-14

本文是阶段 17 的设计文档，定义 `packages/ui`、`packages/shared`、`packages/agent-core`、Rust `services/`、Rust `db/`、Rust `commands/` 六层之间的职责边界、调用方向和约束。本文只产出边界定义和路线更新，不写代码。

## 一、当前问题

阶段 16 完成了独立页面拆分，UI、Tauri、SQLite、写入 commands 已全部跑通。但架构图上的 `packages/shared` 和 `packages/agent-core` 目前只是空目录（各有一个 README），类型全部定义在 `packages/ui/src/utils/desktopHost.ts` 中，跨端共享类型实际上不存在。

如果不现在定死分层边界，后续低级智能体很容易把业务规则一会儿写到前端、一会儿写到 Rust、一会儿又塞进共享包，导致项目回到旧原型时期的混乱状态。

## 二、六层架构定义

```
packages/ui              界面 + Tauri invoke 调用
packages/shared          跨端共享类型、常量、DTO
packages/agent-core      Agent/Task/Approval 规则模型、状态定义、流程描述
apps/desktop/src-tauri/
  commands/              Tauri 命令入口（薄封装）
  services/              Rust 业务校验和状态流转
  db/                    SQLite 读写（纯数据访问）
```

### 2.1 packages/ui — 界面层

**允许：**
- React 组件、页面、布局
- Ant Design 组件组合
- 通过 Tauri `invoke` 调用 Rust commands
- 从 `packages/shared` 导入类型、常量
- 从 `packages/agent-core` 导入规则定义（状态枚举、转换映射）
- 前端专用工具函数（如 `isTauriHost()` 环境检测）
- 浏览器 fallback 数据和降级逻辑
- 国际化文案（中文标签、提示文本）

**禁止：**
- 直接读写 SQLite
- 直接操作文件系统
- 直接调用外部 API/Provider
- 定义业务规则（状态机、校验逻辑）
- 定义跨端共享类型（应放到 `packages/shared`）
- 执行 Git 命令
- 启动/停止 Runner

**当前待迁移项（阶段性合理，`packages/shared` 和 `packages/agent-core` 建好后迁出）：**
- `desktopHost.ts` 中的跨端类型（`TaskStatus`、`ProjectSummary`、`AgentSummary`、`TaskSummary`、`ApprovalSummary`）→ 应迁至 `packages/shared`
- `desktopHost.ts` 中的输入类型（`CreateTaskInput`、`UpdateTaskStatusInput`）→ 应迁至 `packages/shared`
- `TasksPage.tsx` 中的 `statusTransitions` 状态转换映射 → 应迁至 `packages/agent-core`
- 字段长度常量（`TITLE_MAX=120`、`DESC_MAX=2000` 等）→ 应迁至 `packages/shared`

### 2.2 packages/shared — 跨端共享层

**允许：**
- TypeScript 类型定义（`ProjectSummary`、`TaskSummary`、`AgentSummary`、`ApprovalSummary`）
- 输入/输出类型（`CreateTaskInput`、`UpdateTaskStatusInput`）
- 状态枚举（`TaskStatus`、`ApprovalStatus`）
- 共享常量（字段长度限制、允许值列表）
- API 响应类型（DTO）

**禁止：**
- 业务执行逻辑
- 数据库操作
- 文件 I/O
- React 组件或 hooks
- 任何运行时副作用

**包名：** `@agent-swarm/shared`
**目录结构建议：**
```
packages/shared/src/
  types/
    project.ts       # ProjectSummary
    task.ts          # TaskSummary, TaskStatus, CreateTaskInput, UpdateTaskStatusInput
    agent.ts         # AgentSummary
    approval.ts      # ApprovalSummary, ApprovalStatus
    host.ts          # DesktopHostOverviewData, DesktopHostOverviewState
  constants/
    limits.ts        # 字段长度限制 (TITLE_MAX=120, DESC_MAX=2000, REASON_MAX=2000)
    enums.ts         # PRIORITY_VALUES, RISK_LEVEL_VALUES, TARGET_SERVICE_VALUES
  index.ts           # 统一导出
```

### 2.3 packages/agent-core — 规则模型层

**允许：**
- 状态转换映射（`statusTransitions`）和终态判断
- 领域规则纯函数（如 `isValidTransition(current, next): boolean`、`isTerminalStatus(status): boolean`）
- 角色定义和权限清单
- 领域模型描述（Agent 角色、调度策略、Workflow 节点规则）
- 流程描述（文本/结构化定义，不是执行引擎）

**禁止：**
- 直接读写 SQLite
- 直接调用 Tauri invoke
- React 组件或 hooks
- 文件 I/O
- 网络请求
- 真实 Agent 执行（不触发模型调用、不写文件）
- 引入 Rust 或任何非 JS/TS 依赖

**包名：** `@agent-swarm/agent-core`
**目录结构建议：**
```
packages/agent-core/src/
  models/
    task.ts           # TaskStatus, statusTransitions, isValidTaskTransition, isTerminalTaskStatus
    approval.ts       # ApprovalStatus, approvalTransitions, isValidApprovalTransition
    agent.ts          # AgentRole, rolePermissions
    project.ts        # ProjectPhase, phaseConstraints
  index.ts            # 统一导出
```

**依赖关系：**
- `packages/agent-core` 可以依赖 `packages/shared`（引用类型和常量）
- `packages/agent-core` 不可以依赖 `packages/ui`
- `packages/agent-core` 不可以依赖任何 Rust 包

### 2.4 Rust commands/ — Tauri 命令入口

**允许：**
- 从 Tauri state 提取数据库连接
- 调用 services 层函数
- 返回 services 层的结果
- 基础参数类型转换（String → &str）

**禁止：**
- 任何业务校验（字段验证、状态机判断）
- 直接执行 SQL
- 调用外部 API
- 文件 I/O

**当前状态：** 已经干净。所有 command 都是单行委托。

### 2.5 Rust services/ — 业务逻辑层

**允许：**
- 字段验证和标准化（normalize）
- 状态转换校验（状态机）
- 跨实体引用检查（agent 是否存在、task 是否属于当前项目）
- 业务规则执行
- 调用 db/ 层读写数据
- 与 `packages/agent-core` 中的规则保持语义一致（各自实现，不共享代码）

**禁止：**
- 直接暴露 Tauri state（`tauri::State` 只应在 commands 层使用）
- 向前端暴露内部错误细节（敏感信息脱敏）
- 调用外部 API/Provider（当前阶段；后续 Model Gateway 会是独立 service）

**当前需要改进：**
- `tasks.rs` 和 `approvals.rs` 有 8 个重复的 normalize 辅助函数 → 可提取到 `services/common.rs`
- 状态枚举值散落在 normalize 函数中 → 可提取为模块级常量

### 2.6 Rust db/ — 数据访问层

**允许：**
- 数据库连接管理
- Schema 迁移（DDL）
- 种子数据
- 基础 CRUD 查询

**禁止：**
- 字段验证
- 状态机判断
- 跨实体引用检查（这属于 services 层）
- 业务规则

**当前状态：** 已经干净。所有业务逻辑都在 services 层。

## 三、调用方向（单向依赖）

```
packages/ui ──→ packages/agent-core ──→ packages/shared
     │
     └──→ Tauri invoke ──→ Rust commands ──→ Rust services ──→ Rust db
```

**不允许的反向依赖：**
- `packages/shared` 不能引用 `packages/agent-core` 或 `packages/ui`
- `packages/agent-core` 不能引用 `packages/ui`
- Rust 层不引用任何 `packages/` 下的 TypeScript 包
- `commands/` 不引用 `services/` 以外的内部模块
- `db/` 不引用 `services/` 或 `commands/`
- `services/` 不引用 `commands/`

## 四、规则一致性约定

TypeScript 端（`packages/agent-core`）和 Rust 端（`services/`）各自维护业务规则，但必须保持语义一致。这不是代码共享，而是契约一致。

**一致性要求：**
- 状态枚举值：TS 的 `TaskStatus` union 必须与 Rust `normalize_task_status` 允许的列表一致
- 状态转换：TS 的 `statusTransitions` 必须与 Rust `ensure_task_transition_allowed` 的规则一致
- 字段限制：TS 的 `limits.ts` 必须与 Rust `normalize_required_text` 等函数的长度限制一致
- 允许值列表：TS 的 `enums.ts` 必须与 Rust `normalize_enum` 允许的值列表一致

**不一致时的处理：**
- Rust services 是权威源（运行时校验以此为准）
- TS agent-core 是 UI 辅助（前端表单校验、状态按钮显隐）
- 验证脚本应定期检查两端一致性

## 五、旧模块冻结条件

### 5.1 Runner

**当前状态：** 全局禁用。旧原型中有 `runner/status`、`runner/jobs`、`runner/approval` 等路由。

**解冻条件（全部满足才开）：**
1. 审批链设计完成：至少覆盖 `create→approve→execute→verify` 四个环节
2. Git checkpoint 设计完成：执行前自动创建、失败可回滚
3. 补丁预览设计完成：用户能看到将修改哪些文件、具体 diff
4. 安全回滚设计完成：失败后回滚必须重新审批，不能自动扩大权限
5. 命令白名单设计完成：只允许指定命令，禁止任意 shell
6. 保护路径检查设计完成：`_internal/`、`design/image2/`、`data/local/`、`logs/`、`.playwright-cli/` 不可触碰

### 5.2 真实模型调用

**当前状态：** 阶段 2 设计已收口，但未建表、未接 SDK、不发请求。

**解冻条件（全部满足才开）：**
1. 通过 `dev-docs/真实模型接入准入规格.md` 全部准入项
2. Provider 配置只在后端，前端不可见完整 API Key
3. 日志脱敏验证通过
4. 先接一个 provider，验证通过后再扩展
5. 禁止前端自由拼 prompt body 或 provider request body

### 5.3 Agent Run 真实执行

**当前状态：** Mock/SQLite 记录链已完成。不触发真实 Agent。

**部分解冻（可以先做）：**
- Agent Run 记录视图：在 UI 中展示 Mock/SQLite 中的 `agent_runs` 记录 → 允许，不涉及真实执行

**完全解冻条件（全部满足才开）：**
1. 真实模型调用已开放
2. Agent 角色定义和执行边界已确认
3. 子 Agent 不自批（不能自己审批自己的输出）
4. 失败停在失败节点，不自动重试或扩权
5. Agent 不直接写文件（只生成建议，由 Runner 在审批后执行）

### 5.4 Git 执行

**当前状态：** 全局禁止修改 Git。UI 中有 "Git 保存点" 按钮但为 disabled 占位。

**解冻条件（全部满足才开）：**
1. Runner 已开放且审批链通过验收
2. 第一版只允许只读命令：`git status --short`、`git diff --stat`、`git diff --name-only`。禁止所有写入命令（`git stash`、`git stash pop` 等有冲突风险，后续独立设计）
3. 每个 Git 操作必须有对应的审批记录
4. 禁止 `git push`、`git reset --hard`、`git stash`、force push 到任何分支

| 监控与日志 | 全部未接入 | 暂不做 |
| 费用与用量 | 全部未接入 | 等真实模型调用开放后再接入 |

## 六、阶段 18 最小实现目标

阶段 17 通过后，阶段 18 做最小骨架实现，目标是把共享类型的"家"建起来：

### 6.1 新建 packages/shared

- 创建 `packages/shared/package.json`（包名 `@agent-swarm/shared`）
- 创建 `packages/shared/tsconfig.json`
- 从 `packages/ui/src/utils/desktopHost.ts` 迁出以下类型：
  - `TaskStatus`、`TaskSummary`、`CreateTaskInput`、`UpdateTaskStatusInput`
  - `AgentSummary`
  - `ApprovalSummary`
  - `ProjectSummary`
  - `DesktopHostOverviewData`、`DesktopHostOverviewState`
- 补充常量：`TITLE_MAX_LENGTH = 120`、`DESC_MAX_LENGTH = 2000`、`REASON_MAX_LENGTH = 2000`
- 补充枚举值列表：`PRIORITY_VALUES`、`RISK_LEVEL_VALUES`、`TARGET_SERVICE_VALUES`
- `packages/ui` 改为从 `packages/shared` 导入类型（使用相对路径或 tsconfig paths，方法待阶段 18 单独讨论）

### 6.2 新建 packages/agent-core

- 创建 `packages/agent-core/package.json`（包名 `@agent-swarm/agent-core`）
- 创建 `packages/agent-core/tsconfig.json`
- 从 `packages/ui` 迁出：
  - `statusTransitions`（从 `TasksPage.tsx` 提取）
  - 状态校验纯函数（`isValidTransition`、`isTerminalStatus`）
- 从 `packages/shared` 引用字段长度常量，不重复定义

### 6.3 跨包导入方式

仓库根目录当前没有 `package.json` 和 monorepo workspace 配置。阶段 18 的跨包导入需要先确认方法：

**方案 A（保守）：** TypeScript path aliases —— 在 `packages/ui/tsconfig.json` 中配置 `paths` 映射，直接引用 `packages/shared/src/` 和 `packages/agent-core/src/`。不需要 workspace，构建时 TypeScript 会直接编译引用文件。

**方案 B（标准）：** 建立 pnpm/npm workspace —— 在仓库根目录创建 `package.json` 和 `pnpm-workspace.yaml`（或等效配置），各包通过 `@agent-swarm/shared` 包名互相引用。

建议阶段 18 先用方案 A 跑通，不新增根目录 workspace 配置。是否切换到方案 B 后续单独讨论。
