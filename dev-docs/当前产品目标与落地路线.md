# 当前产品目标与落地路线

日期：2026-06-18

本文件用于校准当前方向。后续做数据库、后端、模型网关、Runner 或 UI 调整时，先按这里判断是否跑偏。

## 当前结论

这次前端讨论没有偏离项目主线。它把原文档里的长期目标具体化成了更清楚的产品形态：

```text
本地桌面端 AI 项目总控台
-> 用户只对总控 Agent 描述项目目标
-> 总控判断项目类型、技术栈、风险和阶段
-> 总控选择固定员工和项目专家
-> 每个 Agent 绑定执行器、模型和职责边界
-> 模型调用经过模型网关
-> 写文件和高风险动作经过 Runner、审批和运行记录
-> 项目产物归档到 workspace/generated
```

不需要更换前端。`packages/ui` 仍是桌面端和预览共用的 UI 真源。浏览器 `127.0.0.1:5173` 只是预览模式；真实读写 SQLite、Runner 和模型网关能力必须通过 Tauri 桌面端调用 Rust 后端。

## 当前 UI 探索形成的新准绳

### 主控台

主控台是用户和总控 Agent 对话的入口。

它应该展示：

- 用户目标输入。
- 总控理解和任务拆解。
- 模块画布：开始、管理器、智能体、槽位、条件、汇总等基础流程模块。
- 画布支持右键加模块、拖动、缩放、连线、删除模块。
- 底部对话栏用于解释概念、补充需求和后续接真实总控模型。

主控台不是自由执行入口。真正写文件或调用外部能力时，仍必须进入受控服务链路。

### 任务拆解

任务页的核心不是产物列表，而是：

```text
一个项目任务
  -> 下面分配的 AI 员工
  -> 每个员工负责的任务、状态、进度、产物路径
```

任务页必须能显示：

- 项目任务标题。
- 已分配 AI 员工。
- 哪些员工正在工作。
- 每个员工负责什么。
- 整体进度。
- 输出目录。
- 继续做、打开文件夹、删除任务记录。

产物删除不在当前页作为主功能。真实文件删除属于高风险动作，后续必须经过审批或明确确认。

### 流程蓝图

流程蓝图用于解释总控如何调度，而不是堆很长的节点说明。

它应该显示：

- 总控调度阶段。
- 每个阶段涉及哪些 AI 员工。
- 默认压缩展示员工标签。
- 当前活跃阶段才展开职责详情。
- 阶段与 Runner、审批、模型网关的关系。

流程蓝图不应成为第二个任务页，也不应展示过多假入口。

### AI 员工

AI 员工页是全局员工池和执行器配置入口。

它应包含：

- 固定员工池。
- 项目专家推荐。
- 当前项目成员。
- 执行器配置树。
- 每个员工的职责边界、技术栈、默认执行器和模型。

执行器配置必须能表达：

- Provider。
- Base URL。
- API Key / Token。
- 模型目录。
- 绑定智能体。
- Skill 配置。

当前前端可先本地保存非敏感配置用于预览。API Key 不得进入 localStorage、SQLite、日志或文档。真实保存必须走系统设置、安全存储或当前桌面进程环境变量。

### 系统设置

系统设置里的模型服务是系统默认模型服务，也可以理解为“总控默认大脑”。

它用于：

- 总控 Agent 默认模型调用。
- 未单独配置执行器的员工默认模型调用。
- 模型连接测试。

AI 员工页里的执行器配置可以在后续覆盖系统默认配置。

## 总控调度和越界防护

必须记录并实现：总控可以分配所有 AI，但不能只靠提示词保证 Agent 不越界。

真正的越界防护必须分三层：

1. 总控分配层：总控根据项目类型、技术栈、风险和阶段，把任务拆到前端、后端、数据库、桌面端、安全、测试、文档等模块。跨模块任务必须拆分或转派。
2. Agent 职责边界层：每个 Agent 有固定职责、允许任务类型、允许路径和禁止动作。Agent 发现任务超出职责时，应退回总控。
3. Runner / 后端强校验层：任何写文件、运行命令、模型调用、审批放行、打开产物、删除任务记录等动作，都要由后端根据 `agent_id`、`task_type`、`module_scope`、`target_path` 和风险等级判断允许、拒绝或进入审批。

示例规则：

```text
前端 Agent 不能修改数据库 migration。
数据库 Agent 不能修改页面样式。
文档 Agent 不能改变产品行为。
安全 Agent 不能直接放行未经记录的高风险动作。
任何 Agent 都不能自由 shell、Git commit/push、删除文件或写保护路径。
```

这条规则是后端接入阶段的核心验收项。如果只做 UI 提示或 prompt 约束，不算完成。

### P1：任务拆解绑定真实 Agent

任务不应只是一组泛泛子任务，而应该绑定到具体 Agent：

- 项目任务。
- 子任务。
- assigned_agent_id。
- executor_key。
- model_id。
- output_folder。
- status。
- run/progress 信息。

任务页要从 SQLite 读取真实任务结构，而不是浏览器示例数据。

### P2：总控分派规则

总控应按以下顺序分派：

```text
用户目标
-> 项目类型
-> 技术栈
-> 风险
-> 必选核心员工
-> 推荐专家
-> 任务拆解
-> Agent 绑定
-> Runner / 审批 / 运行记录
```

跨模块任务必须拆分或转派。角色不能自行越权。

### P3：运行输出和审计闭环

运行输出应按 Agent 维度展示：

- Agent。
- 任务。
- 使用模型/执行器。
- 输入摘要。
- 输出摘要。
- 文件改动。
- 产物路径。
- 错误和重试。
- 审批来源。

## 不要做

- 不要更换前端栈。
- 不要把浏览器预览当成真实桌面端能力。
- 不要绕过 Tauri/Rust/SQLite 直接在前端伪造真实写入。
- 不要让 UI 选择模型后直接修改真实执行配置，真实配置必须走受控服务。
- 不要把 API Key、Token、raw prompt、raw response、raw provider error 写入数据库、日志或文档。
- 不要让 Agent 自由 shell、自动 Git commit/push、删除文件或写保护路径。
- 不要为了角色数量好看默认激活一堆专家。
