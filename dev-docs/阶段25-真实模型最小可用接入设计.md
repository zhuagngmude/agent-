# 阶段 25：真实模型最小可用接入设计

日期：2026-06-15

## 1. 阶段结论

阶段 25 只开放一条真实模型链路：

```text
用户手动触发 project_plan_generation
-> Tauri command
-> Rust Model Gateway
-> openai_compat provider
-> 脱敏 / 限长 / 结构化摘要
-> 返回前端预览
```

本阶段不开放真实 Runner。Runner 仍保持只读 `runner_requests` 队列，不执行命令、不写文件、不改 Git、不发网络请求。

## 2. 为什么不能模型和 Runner 一起开

真实模型调用的主要风险是网络请求、密钥读取、prompt / response 泄漏。

真实 Runner 的主要风险是本地文件写入、命令执行、Git 变更和不可逆副作用。

两者如果同阶段一起打开，回归面会同时覆盖网络、密钥、数据库、UI、文件系统和 Git，无法满足《Agent 宪法》的“最稳妥方案”和“可验证证明”。因此阶段 25 只接真实模型；Runner 后置到模型链、Agent Run 真实链和执行计划审查链稳定之后。

## 3. 编码前五问

### 3.1 真正要解决的问题是什么？

让应用第一次从本地确定性模板走向真实模型输出，但模型结果只能作为人工可见的项目计划草案预览，不得直接创建任务、审批、Runner request 或写文件。

### 3.2 谁创建、谁调用、谁消费？

- 用户在 `ProjectPlanPage` 手动输入项目想法并二次确认。
- 前端只调用 `request_project_plan_model_draft`，不得传 key、base URL、provider、model、prompt、headers 或 provider body。
- Rust command 只做 Tauri invoke 入口。
- Rust `services/model_gateway` 负责校验、读取后端 env、构造固定请求、调用 provider adapter、脱敏和返回结构化摘要。
- 前端只展示脱敏摘要。

### 3.3 现有系统里有没有承担同一职责的资源？

已有资源：

- `commands/model_gateway.rs`
- `services/model_gateway/mod.rs`
- `services/model_gateway/project_plan.rs`
- `services/model_gateway/provider_config.rs`
- `services/model_gateway/redaction.rs`
- `services/model_gateway/model_calls.rs`
- `model_calls` SQLite 表（阶段 23）
- `ProjectPlanPage` 本地确定性计划入口（阶段 24）
- `dev-docs/真实模型接入准入规格.md`
- `dev-docs/Model Gateway正式入口设计.md`

阶段 25 必须扩展这些资源，不另开自由聊天、自由 prompt、任意 provider 或 Runner 入口。

### 3.4 更保守的替代方案是什么？

继续只使用阶段 24 的本地确定性模板。阶段 25 的保守落法是：即使 feature flag 打开，真实模型结果也只返回内存预览，暂不落 `project_plan_drafts`、不写 `model_calls` 业务记录、不创建 `runtime_events`。

### 3.5 最大回归风险是什么？怎么挡住？

最大风险：

- 前端传入密钥、base URL、headers、prompt 或 provider body。
- Rust 错误消息泄漏 raw key、base URL、raw prompt 或 provider 原始响应。
- 模型结果绕过审批直接创建任务或 Runner request。
- Feature flag 被误用导致页面加载自动发请求。

防线：

- `#[serde(deny_unknown_fields)]` 保持开启。
- 二次确认必须为 `true`，确认文本必须等于 `我确认发起真实模型调用`。
- `AGENT_SWARM_ENABLE_REAL_MODEL_PROJECT_PLAN` 默认关闭。
- 真实请求只允许 `openai_compat` + `gpt-5.4-mini` + `project_plan_generation`。
- provider adapter 只在内部读取 env，不把 raw key / base URL 放进可序列化返回结构。
- 所有错误只返回粗粒度 `error_category`。
- 返回结构只允许 `status / error_category / summary / warnings` 等脱敏字段。

## 4. 依赖申请

真实 HTTP 请求需要新增一个 Rust HTTP 客户端 crate。候选：

### 方案 A：`ureq`

- 优点：同步阻塞 API 简单，适合当前 Tauri command 的最小闭环；依赖面比异步栈小。
- 缺点：后续如果需要 streaming 或高并发，需要再评估。

### 方案 B：`reqwest`

- 优点：生态最常见，功能完整。
- 缺点：默认依赖面更大；异步运行时和 Tauri command 的接入要更谨慎。

建议阶段 25 采用方案 A：`ureq`，只用于一次固定用途 POST 请求，并设置超时和响应体大小上限。

在用户批准前，不修改 `Cargo.toml`。

## 5. Provider 请求形态

第一版只支持 OpenAI-compatible Chat Completions：

```text
POST {base_url}/chat/completions
Authorization: Bearer <env key>
Content-Type: application/json
```

请求体由 Rust 固定构造：

```json
{
  "model": "gpt-5.4-mini",
  "messages": [
    {
      "role": "system",
      "content": "你是项目计划助手，只输出简洁的中文项目计划摘要。"
    },
    {
      "role": "user",
      "content": "根据项目想法和约束生成项目计划草案。"
    }
  ],
  "temperature": 0.2,
  "stream": false
}
```

禁止：

- 前端传 system prompt。
- 前端传 model。
- 前端传 provider。
- 前端传 headers。
- 前端传 base URL。
- 前端传 raw prompt。
- streaming。
- tools / function calls。
- 文件输入。

## 6. 返回与落库策略

阶段 25 第一版真实调用成功后：

- 返回 `status=draft_ready`。
- 返回脱敏、限长后的 `summary`。
- `warnings` 可包含粗粒度提示，例如“真实模型结果未落库”。
- 不写 `project_plan_drafts`。
- 不写 `tasks`。
- 不写 `approvals`。
- 不写 `runner_requests`。
- 不写用户项目文件。
- 不执行 Runner。
- 不改 Git。

`model_calls` 写入可作为阶段 25.2 单独批次：只有阶段 25.1 真实调用链稳定后，才允许把脱敏摘要和粗粒度 token usage 写入 SQLite。

### 25.3 model_calls 安全审计落库（已完成 2026-06-15）

阶段 25.1 真实模型调用链、阶段 25.2 前端真实模型草案预览入口均已完成。25.3 将真实调用审计记录写入 `model_calls` 表，但只保存安全字段。

**写入时机：**
- 只有通过 feature flag、二次确认和 provider config 校验后，进入 `provider.send(...)` 的真实调用阶段才写入 `model_calls`。
- 成功调用：写入 `status=success`，保存脱敏、截断后的 `structured_summary`。
- 失败调用（timeout/network_error/provider_error/response_too_large/invalid response）：写入 `status=failed`，保存粗粒度 `error_category`。
- provider 成功但审计写入失败 → `status=audit_write_failed`，`error_category=audit_write_failed`，`summary=None`
- provider 错误且审计写入失败 → 同上，返回 `audit_write_failed`，不伪装成已正常审计
- `audit_write_failed` 是接口响应 error_category，**不入库** `model_calls` 表（审计写入失败意味着记录未成功写入）

> **阶段 26** 已实现：真实模型草案可手动保存为待审批项目计划草案，见 `dev-docs/阶段26-真实模型草案进入项目计划审批流设计.md`。

**不写 model_calls 的情况：**
- feature flag 关闭
- 二次确认失败
- provider config 缺 key / 缺 base URL / base URL 无效
- 输入校验失败

原因：这些没有真正进入 provider 调用阶段，避免把未调用请求当作真实调用审计。

**仍不写的：**
- `runtime_events`（25.3 只做 model_calls）
- raw key / key fragment / raw base URL
- raw prompt / raw provider request / raw provider response / raw provider error
- 不新增前端自由模型配置入口

**前端：**
- 成功时显示"已写入安全审计记录"
- 不展示记录详情，不新增审计列表页面

**实现文件：**
- `apps/desktop/src-tauri/src/services/model_gateway/model_calls.rs`：新增 `SafeModelCallRecordInput`、`insert_safe_model_call()`、`compute_request_hash()`
- `apps/desktop/src-tauri/src/services/model_gateway/mod.rs`：`create_project_plan_draft_core` 新增 `db_connection` 参数，在 provider 调用处写入 `model_calls`
- `apps/desktop/src-tauri/src/commands/model_gateway.rs`：Tauri command 接入 `DbState`，传递连接给 service
- `packages/shared/src/types/project-plan.ts`：`ProjectPlanModelDraftResponse` 新增 `audit_record_id` 字段
- `packages/ui/src/pages/ProjectPlanPage.tsx`：成功/失败时显示审计记录提示

**测试覆盖（151 个 Rust 测试全部通过）：**
- feature disabled / invalid request / config error 不写 model_calls
- provider 成功写 1 条 success 记录，`audit_record_id` 有值
- provider 错误写 1 条 failed 记录
- 所有 error_category 映射正确
- 敏感内容不入库（sk-xxx、bearer token、api_key、token、password 均被脱敏）
- `request_hash` 不含 raw idea/constraints
- 不创建 task/approval/runner_requests
- 不写 runtime_events
- `audit_record_id` 在非 provider 路径为 null

## 8. 测试计划

Rust 单测：

1. feature flag 关闭返回 `feature_disabled`。
2. feature flag 开启但缺 key 返回 `missing_key`。
3. feature flag 开启但缺 base URL 返回 `missing_base_url`。
4. base URL 非 https 返回 `invalid_base_url`。
5. 未二次确认返回 `invalid_request`。
6. 确认文本错误返回 `invalid_request`。
7. forbidden value pattern 仍拦截 key / token / password。
8. provider error 不返回 raw provider body。
9. response body 超限返回 `response_too_large`。
10. 成功响应只返回脱敏摘要。

前端验证：

1. 浏览器预览模式按钮禁用。
2. Tauri 模式未确认时按钮不可提交。
3. 错误状态用 Alert 显示粗粒度错误。
4. 成功状态只显示摘要，不显示 raw response。
