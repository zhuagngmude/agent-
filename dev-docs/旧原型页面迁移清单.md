# 旧原型页面迁移清单

日期：2026-06-14

本文盘点旧原型（`apps/web/`、`design/index.html`、`services/api/`）的全部页面和模块，按"迁移 / 冻结 / 暂不做"三类归档，并确认新 `packages/ui` 的页面拆分方案。

## 一、旧原型盘点

### 1.1 旧 Web 前端（`apps/web/`，13 个页面）

旧前端是原生 HTML/CSS/JS 单页应用，通过 `section.page-view` 切换视图，已冻结。

| # | 页面 ID | 导航标题 | 当前状态 |
|---|---------|----------|----------|
| 1 | `overview` | 项目概览 | 仪表盘：指标卡、任务队列、Agent 状态、审批、Git、知识库、设置 |
| 2 | `agents` | 智能体管理 | Agent 资源池、模型分配、变更请求、配置版本历史 |
| 3 | `tasks` | 任务管理 | 任务队列 + 详情面板 + 状态操作 |
| 4 | `workflow` | 工作流编排 | 项目计划审批（project_plan） |
| 5 | `agentRuns` | Agent Run 记录 | Agent Run 链记录 + 详情 + Mock 生成 |
| 6 | `knowledge` | 知识库管理 | 人类说明书、AI 开发维护手册 |
| 7 | `code` | 代码与知识 | 简易文件列表 |
| 8 | `runtime` | 运行与调度 | Runner 状态、Job 队列、Runtime Events |
| 9 | `approval` | 审批与确认 | Runner 审批列表 + 审查详情 |
| 10 | `monitor` | 监控与日志 | CPU/内存/网络指标 + 日志占位 |
| 11 | `billing` | 费用与用量 | Token 消耗/成本占位 |
| 12 | `integrations` | 集成与扩展 | GitHub/Runner/Provider 集成占位 |

### 1.3 旧 API 路由（`services/api/server.js`，30+ 端点）

旧后端是 Node.js 原生 HTTP，路由通过 `method + pathname` 链式匹配，已冻结。

核心业务模块：

| 模块 | 路由举例 | 说明 |
|------|----------|------|
| Dashboard | `GET /api/projects/:id/dashboard` | 仪表盘聚合数据 |
| Agents | `GET /api/projects/:id/agents` | Agent 列表 + 配置管理 |
| Tasks | `GET/POST /api/projects/:id/tasks`、`/api/tasks/:id/:action` | 任务 CRUD + 状态操作 |
| Approvals | `GET/POST /api/projects/:id/approvals`、`/api/approvals/:id/:action` | 审批读写 + 终态操作 |
| Runner | `GET /api/projects/:id/runner/status/jobs`、`POST .../:action` | Runner 状态 + Job 操作 |
| Workflows | `GET /api/projects/:id/workflows` | 工作流查询 |
| Project Plan | `POST /api/projects/:id/project-plan-requests` | 项目计划审批 |
| Agent Runs | `GET /api/projects/:id/agent-runs`、`POST .../agent-run-requests` | Agent Run 链 |
| Agent Config | `POST /api/agents/:id/change-requests`、`.../:id/apply` 等 | Agent 配置变更、apply、回滚、版本历史 |
| Model Gateway | `GET /api/model-gateway/status`、`POST .../dry-run` | 模型网关状态、干运行 |
| Git | `GET /api/projects/:id/git/checkpoints` | Git 检查点 |
| Knowledge | `GET /api/projects/:id/knowledge/updates` | 知识库更新 |
| Runtime | `GET /api/runtime-state`、`POST .../reset` | 运行时状态读写 |
| Settings | `GET /api/settings` | 设置数据 |
| Usage | `GET /api/usage` | 用量数据 |

| 旧模块 | 新架构对应 | 状态 |

### 2.2 建议迁移（下一步拆独立页面）

这些模块在新架构中已有数据基础（SQLite 表 + Tauri commands），只需拆成独立页面。

| 旧模块 | 新页面 | 迁移内容 | 优先级 |
|--------|--------|----------|--------|
| 任务管理 | `TasksPage` | 从 Overview 拆出任务表格 + 操作列 + 新建按钮 | 高 |
| Agent 管理 | `AgentsPage` | 从 Overview 拆出 Agent 表格（只读；配置变更后续另走审批设计） | 高 |
| 审批管理 | `ApprovalsPage` | 从 Overview 拆出 SQLite 审批记录表格 + 终态操作按钮；不迁移 Runner diff / Runner job / 文件审批详情 | 高 |
| 系统设置 | `SettingsPage` | 占位页。第一版只读展示当前项目信息；模型配置仅做禁用态/占位 | 中 |

### 2.3 暂时冻结（后续阶段迁移）

这些模块有完整旧实现，但依赖 Runner、Agent Run、Model Gateway 等当前禁用的能力，现在不迁。

| 旧模块 | 依赖 | 冻结原因 |
|--------|------|----------|
| 工作流编排（project_plan） | 批量任务生成、Runner request | 阶段 24 已迁移到 `ProjectPlanPage` + `approve_project_plan`；仍不执行 Runner |
| Agent Run 记录 | 真实 Agent 执行 | 阶段 3 已收口为本地 Mock/SQLite，暂无真实 Agent |
| 运行与调度（Runner） | 真实 Runner 执行 | 全局禁用 Runner |
| 审批与确认（Runner 审查） | Runner 执行、文件 diff | 全局禁用 Runner |
| Git 历史 | Git 操作 | 全局禁止 Git 修改 |

### 2.4 暂不做（未接入或纯占位）

这些模块在旧原型中也只是 UI 占位，从未接真实数据，新架构中暂不处理。

| 旧模块 | 说明 |
|--------|------|
| 监控与日志 | CPU/内存/网络指标全部未接入 |
| 费用与用量 | Token 消耗/成本全部未接入 |
| 集成与扩展 | GitHub/Runner/Provider 集成全部未连接 |
| 代码与知识 | 简易文件列表，无真实功能 |

### 2.5 仅保留设计参考（design/index.html 中的页面）

设计原型的视觉风格（深色侧边栏 `#071a34`、浅色内容区 `#f7f7fb`、紫色强调 `#7367f0`）已在新 `packages/ui` 的 global.css + uiTheme.ts 中落地。原型 HTML 本身不再迁移。

| 设计页面 | 处置 |
|----------|------|
| 项目总览 | 视觉已落地，数据已在 OverviewPage |
| Agent 编排 | 视觉参考保留，功能拆到 AgentsPage |
| 任务拆解 | 视觉参考保留，功能拆到 TasksPage |
| 执行流水线 | 暂不做 |
| Runner 确认 | 暂不做（Runner 禁用） |
| Git 历史 | 暂不做（Git 禁用） |
| 知识库 | 暂不做 |
| 系统设置 | 视觉参考保留，占位到 SettingsPage |

## 三、新页面结构

### 3.1 当前结构

```
AppShell
  └── OverviewPage（所有数据集中在一个页面）
```

### 3.2 目标结构

```
AppShell
  ├── OverviewPage        总览仪表盘（指标卡 + 连接状态 + 快速状态摘要）
  ├── ProjectPlanPage     项目计划草案 + 二次确认 + 只读 Runner request
  ├── TasksPage           任务列表 + 新建 + 状态操作
  ├── AgentsPage           Agent 列表（只读，后续加配置）
  ├── ApprovalsPage        审批列表 + 通过/拒绝/仅补丁操作
  └── SettingsPage         占位页（当前项目信息；模型配置仅保留禁用态/占位）
```

### 3.3 各页面职责

| 页面 | Route | 数据源 | 写入能力 |
|------|-------|--------|----------|
| `OverviewPage` | `/` 或默认 | `get_project` + `list_tasks` + `list_agents` + `list_approvals` | 无（只展示统计摘要） |
| `ProjectPlanPage` | `/project-plan` | `list_project_plan_drafts` + `list_runner_requests` | `create_project_plan_draft`、`approve_project_plan` |
| `TasksPage` | `/tasks` | `list_tasks` | `create_task`、`update_task_status` |
| `AgentsPage` | `/agents` | `list_agents` | 第一版只读；Agent 配置变更后续另走审批设计 |
| `ApprovalsPage` | `/approvals` | `list_approvals` | `approve_approval`、`reject_approval`、`patch_only_approval` |
| `SettingsPage` | `/settings` | `get_project` | 第一版只读；真实 provider、raw key 和 Model Gateway 实现必须另走准入 |

说明：上表中的 `Route` 只是页面 key / 目标入口名。阶段 16 第一版建议用 `useState + activePage` 做页面切换，不引入 React Router，也不承诺真实 URL 路由。

### 3.4 导航方案

AppShell 侧边栏从当前"单一入口"扩展为多页面导航：

```text
导航项：
  - 项目总览 (OverviewPage)
  - 项目计划 (ProjectPlanPage)
  - 任务管理 (TasksPage)
  - Agent 编排 (AgentsPage)
  - 审批确认 (ApprovalsPage)
  - 系统设置 (SettingsPage)
```

实现方式：
- 阶段 16 第一版只用简单状态切换（`useState` + `activePage`），不新增路由依赖
- 后续页面增多再评估是否需要 React Router；如需新增依赖，必须先说明理由并获批准

## 四、迁移顺序建议

### 第一批：分拆 Overview 三张表（阶段 16）

1. 从 OverviewPage 拆出 TasksPage
2. 从 OverviewPage 拆出 AgentsPage
3. 从 OverviewPage 拆出 ApprovalsPage
4. OverviewPage 精简为总览仪表盘

### 第二批：占位页面（阶段 17）

1. SettingsPage 占位
2. AppShell 侧边栏导航扩展

### 第三批：冻结模块解冻（后续阶段，条件触发）

- 项目计划审批（project_plan）：阶段 24 已迁移为本地确定性草案、二次确认和只读 Runner request 队列
- Agent Run 记录：等待真实 Agent 执行
- Runner 状态：等待 Runner 开放
- 模型配置 / provider key：等待真实模型准入规格重新确认，禁止读取、返回或记录 raw key

## 五、旧原型清理原则

当前阶段不删除旧文件。清理条件（来自 `dev-docs/旧原型归档方案.md`）：

- 新架构页面覆盖对应功能后，旧页面标记为"已迁移"
- 新架构通过验收后，只能提出删除建议；实际删除必须人工确认、已有 Git 存档，并逐项说明回滚方式
- 全部旧页面迁移完成后，`apps/web/` 和 `services/api/` 只能进入"可讨论移除"状态，不允许 AI 自动整体删除

当前可以做的：
- 新建 `TasksPage`、`AgentsPage`、`ApprovalsPage` 时，不参考旧原型的 JS 实现
- 只参考旧原型的业务语义（字段、状态枚举、操作类型）
- 新页面使用 Ant Design 组件，不抄旧原生 HTML/CSS

## 六、与旧 API 的关系

新架构不再通过 HTTP 调用旧 API。已确认：

- 前端只通过 Tauri `invoke` 调用 Rust commands
- Rust commands 直接操作 SQLite
- 旧 `services/api/server.js` 不启动、不调用、不引用
- 旧 API 路由作为"业务语义参考"保留，不直接搬迁代码

新 Rust commands 与旧 API 的语义对照已记录在 `docs/write-commands-design.md` 第九节。
