# 目录架构确认

日期：2026-06-13

## 一、确认结论

这个项目按单人、自用、本地桌面工具来做：

- 桌面端是主入口
- Web 只做辅助预览 / 浏览器入口
- 共享 UI 只保留一套源码
- 不单独起网络后端服务

## 二、建议目录

```text
apps/
  desktop/        # 桌面主入口，后续承载 Tauri 2 + Rust
  web/            # 浏览器预览入口，共享同一套 UI

packages/
  ui/             # 唯一 UI 源：组件、布局、页面、主题
  agent-core/     # 业务模型、编排规则、流程用例
  shared/         # 公共类型、常量、工具、DTO
```

## 三、现有目录定位

- `apps/desktop/`：桌面主入口预留目录，后续承载 Tauri 宿主和本地能力层。
- `apps/desktop/src-tauri/src/`：Rust 宿主建议分层位置，建议再细分为 `commands/`、`db/`、`services/`，分别放 Tauri 命令入口、rusqlite 访问和本地业务服务。
- `apps/web/`：浏览器预览入口预留目录；当前原型只作为过渡参考，不继续手搓新 UI。
- `packages/ui/`：唯一 UI 源，桌面和 Web 都从这里消费。
- `packages/agent-core/`：纯业务模型和编排规则层，不放页面渲染，也不放具体 I/O / Rust 执行代码。
- `packages/shared/`：跨端共享类型和工具。
- `services/`：旧 Mock / API 原型区，保留作迁移参考，不作为正式后端方向继续扩展。
- `design/`：视觉参考和交接材料，不作为正式 UI 源。
- `data/`：seed、迁移、运行态数据；`mock/` 保留旧原型验证状态，`migrations/` 作为未来 SQLite 迁移脚本来源，受保护路径继续保持只读边界。

## 四、收口原则

1. UI 只保留一套源码。
2. 桌面端和 Web 端只保留不同运行入口，不再各写各的 UI。
3. Rust / Tauri 只负责本地能力，不重新画 UI。
4. Rust 宿主内部建议按 `commands/`、`db/`、`services/` 分层，避免命令、存储和业务逻辑混在一起。
5. 不再把 `services/api/server.js` 继续当成正式后端扩展。
6. 旧原型先归档，等新架构稳定后再清理。

## 五、下一步

1. 确认共享 UI 的组件边界。
2. 确认桌面宿主要暴露哪些本地能力。
3. 确认旧原型归档清单。
4. 再进入新工程初始化。
