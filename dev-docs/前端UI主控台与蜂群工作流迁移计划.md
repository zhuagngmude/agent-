# 前端 UI 主控台与蜂群工作流迁移计划

## 目标

把当前偏表单、偏后台的桌面端 UI，渐进迁移为“Agent 蜂群对话式主控台 + 受控自动化工作流”。迁移只改前端体验、页面组织和用户可见文案，不改变已有 MVP 的真实模型、审批链、Runner、安全闸门、SQLite 和审计逻辑。

本阶段默认不删除旧页面、不绕过旧 API、不放宽 Runner 权限。

## 核心原则

1. 中文优先：用户可见的标题、按钮、标签、说明、状态默认全部中文。
2. 技术名内收：英文技术名只保留在代码、接口、日志、高级详情或调试视图里。
3. 新旧并行：新增主控台/蜂群工作流入口，旧项目计划页完整保留。
4. 只读先行：新工作流页第一版只读展示，不保存、不执行、不创建任务。
5. 安全不变：所有真实动作继续走现有审批链和 Runner 安全链。
6. 小步迁移：每次只迁移一个页面或一个能力面板，验证后再继续。

## 用户可见术语中文化映射

| 内部/技术术语 | 用户可见中文 |
| --- | --- |
| Runner preflight | 执行前审查 |
| Runner gate | 执行许可闸门 |
| Runner dry-run | 只读预演 |
| Runner lock | 执行锁定 |
| minimal-run | 最小范围执行 |
| model_calls audit | 模型调用审计 |
| project seed | 项目种子 |
| plan draft | 计划草案 |
| pending approval | 等待审批 |
| Runner request | 执行请求 |
| Git checkpoint | Git 保存点 |
| Git commit / push | Git 提交 / 推送 |
| API | 接口 |
| debug | 调试 |

规则：

- 面向普通用户的页面不要直接展示 `preflight`、`gate`、`dry-run`、`lock`、`model_calls`。
- 如果必须保留英文，放在“高级详情 / 技术标识 / 日志原文”里。
- 状态解释优先用自然语言，例如“等待你确认”“已完成只读预演”“执行许可已锁定”。

## 页面迁移结构

本计划初期导航建议：

```text
主控台
蜂群工作流（新）
项目计划（旧）
想法引导
Agent 队列
审批闸门
运行记录
系统设置
```

其中：

- `主控台`：电脑端 Agent 蜂群指挥室，一打开应用首先看到。
- `蜂群工作流（新）`：新增只读计划页，展示阶段轨道、工作流画布、自动化规则和安全边界。
- `项目计划（旧）`：保留旧 MVP 项目计划页，作为稳定入口和回退入口。

## 主控台页面方向

主控台负责整体态势，不承担复杂配置。

布局：

```text
左侧：Agent 队列 / 角色进度条
中间：对话式主控区 / 项目种子 / 下一步建议
右侧：安全仪表 / 审批状态 / Runner 边界
顶部：薄状态栏 / 本地模式 / Runner 阶段 / 高风险动作数
```

验收标准：

- 用户第一眼能理解当前项目状态、Agent 状态、安全状态。
- 中间对话区是视觉主角。
- 不出现后台式大表格总览作为默认入口。
- 不出现用户看不懂的英文技术标签。

## 蜂群工作流页面方向

蜂群工作流不是传统项目计划表，而是“受控自动化编排台”。

布局：

```text
左侧：阶段轨道
中间：工作流画布
右侧：自动化规则 + 安全闸门
底部：对话式计划输入框
```

阶段轨道建议：

```text
想法澄清
项目种子
需求拆解
计划草案
Agent 分配
执行前审查
执行许可闸门
只读预演
人工确认
最小范围执行
验收复盘
下一轮计划
```

工作流节点需要展示：

- 负责 Agent
- 当前状态
- 是否可自动推进
- 是否需要审批
- 风险级别
- 产出物
- 与旧 MVP 数据的对应关系

## 新工作流页只读数据映射

第一版只读取已有数据，并映射为节点状态。

示例映射：

| 已有数据 | 工作流节点状态 |
| --- | --- |
| 有项目种子 | 项目种子节点完成 |
| 有计划草案 | 计划草案节点完成 |
| 有等待审批记录 | 人工闸门节点等待确认 |
| 有执行前审查记录 | 执行前审查节点完成 |
| 有执行许可记录 | 执行许可闸门节点完成 |
| 有只读预演记录 | 执行预演节点完成 |
| 有执行锁定记录 | 执行锁定状态开启 |
| 有最小范围执行记录 | 最小范围执行节点完成 |
| 有模型调用审计记录 | 模型调用审计可追踪 |

第一版禁止写入：

- 不创建任务
- 不保存自动化规则
- 不触发模型调用
- 不触发 Runner
- 不修改 Git
- 不删除文件
- 不发网络请求

## 自动化规则展示

右侧自动化规则第一版只做展示，不真正保存配置。

可以显示为开启：

- 自动生成计划草案
- 自动拆分任务模板
- 创建审批请求
- 生成只读预演
- 汇总结果和下一步建议

必须显示为关闭：

- 自动执行命令
- 自动写入文件
- 自动删除文件
- 自动 Git 提交 / 推送
- 绕过人工审批
- 批量自动 Runner

失败策略文案：

```text
任一节点失败或风险升级时，自动化暂停，回到主控对话询问用户，不继续向后执行。
```

## 后续动作接入顺序

只有在只读页稳定后，才逐个接入动作按钮。

建议顺序：

1. 生成计划草案
2. 保存计划草案
3. 创建审批请求
4. 查看执行前审查
5. 查看只读预演
6. 撤销执行许可
7. 解除执行锁定
8. 查看最小范围执行结果

每个动作必须复用现有旧 API 和旧审批链。

不得新增直连危险能力：

- 自动执行命令
- 自动写文件
- 自动删除文件
- 自动 Git commit / push
- 自动网络请求
- 跳过二次确认

## 验收清单

第一步：主控台外壳

- 主入口是对话式主控台。
- 左侧有 Agent 进度卡。
- 右侧有安全仪表。
- 顶部状态栏不抢占对话区注意力。
- 类型检查通过。

第二步：蜂群工作流只读页

- 新增页面，不替换旧项目计划页。
- 能看到阶段轨道、工作流画布、自动化规则。
- 自动化开关只展示，不保存。
- 所有用户可见英文技术术语已中文化。
- 旧项目计划页仍可进入。
- 类型检查通过。

第三步：动作按钮逐步接入

- 每次只接一个动作。
- 每个动作都复用旧 API 和审批链。
- 高风险动作仍要求人工确认。
- 不出现绕过 Runner 安全链的路径。

## 暂不做事项

- 不删除旧项目计划页。
- 不把工作流自动化配置写入数据库。
- 不开放自由命令。
- 不开放批量自动执行。
- 不开放自动 Git 提交 / 推送。
- 不开放自动删除文件。
- 不绕过审批链。

## 当前建议下一步

优先做第二步：

```text
新增“蜂群工作流”只读页面
保留旧“项目计划”页面
把用户可见英文术语全部中文化
从已有项目计划、审批、Runner 记录映射工作流节点状态
```

这样最安全，也最符合当前 MVP 已经跑起来的状态。
