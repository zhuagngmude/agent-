# 阶段 19：冻结模块解冻条件评估

日期：2026-06-14

本文盘点所有冻结模块的现有资产、依赖关系和实现成本，对每个模块给出分级建议，并选出最安全的下一阶段实现目标。本文只做讨论和文档，不写功能代码。

## 一、背景

阶段 16-18 完成了新架构地基：页面拆分、六层边界定义、shared/agent-core 骨架。地基已稳，接下来需要决定：后面先恢复哪个旧模块，恢复到什么程度。

## 二、冻结模块盘点

### 2.1 Agent Run 记录

**旧资产（完整）：**
- `services/api/agent-runs.js`（471行）— 7角色目录、链生成、token/cost估算、失败注入
- `services/api/server.js` — `GET /api/projects/:projectId/agent-runs` 和 `POST /api/projects/:projectId/agent-run-requests` 两个路由
- `services/api/mock-data.js` — 种子链（7条run）、runtime events
- `apps/web/index.html` + `app.js` — 完整旧UI页面（链列表 + 详情面板 + 生成按钮）
- `scripts/sqlite/sqlite_write.py` — 完整 `agent_runs` 表schema（24字段、4索引）

**新架构缺失：**
- 无 Rust Tauri command（`list_agent_runs` 等）
- 无 Rust service 层
- `001_initial_sqlite.sql` 不包含 `agent_runs` 表（需独立 migration）
- 无 React UI 页面、导航项、类型定义

**依赖：** 无。不触发真实Agent、不调模型、不写文件、不操作Git。

**数据来源：** 旧原型通过 Python 脚本在 SQLite 中建表和种子数据。阶段 20 必须优先走 Rust migration（新建 `002_add_agent_runs.sql` 在 `db/mod.rs` 中按序执行），旧 Python SQLite schema 只能作为字段设计参考，不能作为新架构的运行依赖。

## 三、分级

### 分级标准

| 级别 | 含义 | 前置条件 |
|------|------|----------|
| 🔴 继续冻结 | 现在不动，条件满足后再评估 | 解冻条件全部未满足 |
| 🟡 只读迁移 | 在新架构中展示已有数据，不新增写入 | 旧数据可用，无危险依赖 |
| 🟢 可做最小实现 | 在新架构中实现最小可用闭环 | 依赖已满足或可通过 Mock 绕过 |
| ⏸️ 必须等 Runner/模型后置 | 核心依赖未开放，现在做是空中楼阁 | Runner 或模型必须先开放 |

### 分级结果

| 模块 | 级别 | 理由 |
|------|------|------|
| **Agent Run 记录** | 🟡 只读迁移 | 旧资产最完整，零危险依赖（不触发真实Agent、不调模型、不写文件），阶段17已批准部分解冻 |
| 项目计划/Workflow | 🟢 可做最小实现 | 确定性模板生成无外部依赖，但输出 RunnerJob 部分不可用，需裁剪为只生成 Task |
| **Runner** | 🔴 继续冻结 | 6 个解冻条件全部未满足 |
| **真实模型调用** | 🔴 继续冻结 | 5 个准入条件全部未通过，SDK 未导入 |
| **Git 保存点** | ⏸️ 必须等 Runner 后置 | Git 解冻的前置条件是 Runner 开放 |
| 知识库/文件列表 | 🔴 继续冻结 | 旧资产本身也只是占位，复用价值低 |
| 日志监控 | 🟡 只读迁移 | Runtime events 可随 Agent Run 一起展示 |

## 四、建议：下一阶段优先做 Agent Run 只读迁移

### 4.1 选择理由

1. **风险最低。** Agent Run 只读视图不触发真实 Agent、不调用模型、不写文件、不操作 Git。只是把旧项目中已有的本地记录链数据展示到新 UI 架构里。

2. **旧资产最完整。** 旧原型有 471 行业务逻辑代码、完整 API 路由、种子数据、Python SQLite schema。迁移时有明确的语义参考（字段含义、状态枚举、角色定义），不需要从零猜测。

3. **已有批准。** 阶段17文档（5.3节）明确写了"Agent Run 记录视图：在 UI 中展示 Mock/SQLite 中的 agent_runs 记录 → 允许，不涉及真实执行"。

4. **展示新架构能力。** Agent Run 是用户能看到的"AI 做了什么"的直接证据。把它迁到新 UI 后，5 个导航页面都有了实质性内容（不再是空占位），能证明新架构确实能承载旧原型的全部信息。

5. **后续扩展基础。** 一旦真实模型调用开放，Agent Run 记录表可以直接对接真实执行记录，不需要重新设计 UI。

### 4.2 实现范围（建议作为阶段 20）

**做：**
- Rust：新建 migration `002_add_agent_runs.sql`，在 `db/mod.rs` 中按序执行
- Rust：`services/agent_runs.rs` — `list_agent_runs`（只读查询）、`list_runtime_events`（按 agent_run_id 筛选）
- Rust：`commands/agent_runs.rs` — `list_agent_runs`、`list_runtime_events` Tauri command
- 2 个 Tauri command 委托 service → db
- TS：`packages/shared` 新增 `AgentRunSummary`、`RuntimeEventSummary` 类型
- TS：`packages/agent-core` 新增 agent run 状态定义（`agentRunStatusValues`）
- UI：新建 `AgentRunsPage.tsx` — 链列表 + 详情面板（只读表格 + 展开行）
- UI：`mainNavItems.ts` 新增 `agentRuns` 导航项
- UI：`AppShell.tsx` 侧边栏扩展，`App.tsx` 路由分支

**不做：**
- 不建 `POST /agent-run-requests`（不新增写入 command）
- 不实现链生成（`createAgentRunChain`）
- 不接模型（不导入 SDK、不发 provider 请求）
- 不接 Runner（不改文件、不执行命令）
- 失败注入留给后续阶段

### 4.3 数据来源策略

当前 `agent_runs` 表不在 Rust migration 中，但 Python 脚本 (`scripts/sqlite/`) 可以动态创建和填充。阶段 20 有两种数据来源方案：

**方案 A：Rust 迁移 + 种子数据。** 新建 `002_add_agent_runs.sql` 创建表结构，在 `db/mod.rs` 的 seed 流程中从旧种子 JSON 加载 agent run 数据。与现有 4 张表的模式完全一致。

**方案 B：复用 Python 脚本已有数据。** 如果开发机 SQLite 中已有 Python 脚本创建的 `agent_runs` 表，Rust 侧直接读取（不重建）。但这种方式依赖外部脚本的执行顺序。

建议阶段 20 用方案 A，与现有架构一致。
