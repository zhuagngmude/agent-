# 阶段 36：角色模型绑定设计

## 目标

在阶段 35 模型目录基础上，让不同任务角色绑定不同模型。目标不是让模型自由调度，而是让系统在受控目录中为每个角色选择合适模型。

示例目标：

```text
frontend -> gpt-4.1-mini
backend -> gpt-5-mini
qa -> gpt-4o-mini
docs -> 便宜模型
reviewer -> 强审查模型
security -> 强安全模型
```

## 当前边界

本阶段只做“角色到模型的配置关系”和 UI 展示。

仍然禁止：

- 角色自己决定 provider、base URL、key。
- 前端自由输入模型名。
- 模型自由创建任务结构。
- 模型自由写 prompt、header、body。
- 自动执行 Runner。
- 写用户项目文件。
- 改 Git。
- 发起批量模型调用。
- 把 raw prompt / raw response / raw provider error 落库或展示。

## 与阶段 35 的关系

阶段 36 必须依赖阶段 35：

- 模型必须来自 `model_catalog`。
- 只有 `enabled=1` 的模型可绑定。
- 第一版只允许 `provider=openai_compat`。
- 第一版只服务 `project_plan_generation`。

如果阶段 35 未完成，不要实现阶段 36。

## 数据模型

新增 migration：

```text
data/migrations/013_add_role_model_bindings.sql
```

新增表建议：

```sql
CREATE TABLE IF NOT EXISTS role_model_bindings (
  id TEXT PRIMARY KEY,
  project_id TEXT NOT NULL,
  role TEXT NOT NULL,
  purpose TEXT NOT NULL,
  model_record_id TEXT NOT NULL,
  enabled INTEGER NOT NULL DEFAULT 1 CHECK (enabled IN (0, 1)),
  is_builtin INTEGER NOT NULL DEFAULT 0 CHECK (is_builtin IN (0, 1)),
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  FOREIGN KEY (project_id) REFERENCES projects(id),
  FOREIGN KEY (model_record_id) REFERENCES model_catalog(id)
);

CREATE UNIQUE INDEX IF NOT EXISTS idx_role_model_bindings_project_role_purpose
  ON role_model_bindings(project_id, role, purpose);
```

## 允许角色

第一版只允许这些角色：

```text
frontend
backend
qa
docs
reviewer
security
devops
ux
data
```

角色必须来自既有 `project_plan_task_templates` 角色集合。不要允许任意角色字符串。

## Seed 策略

DB 初始化时为每个内置角色 seed 一个绑定：

- 优先绑定当前默认启用模型。
- 如果没有 enabled 模型，绑定记录可以不创建，或创建为 disabled。
- 使用 `INSERT OR IGNORE` 保持幂等。

注意：不要因为没有模型绑定而自动调用模型。

```text
apps/desktop/src-tauri/src/services/role_model_binding.rs

```

新增 command：

```text
list_role_model_bindings
update_role_model_binding
```

`list_role_model_bindings`：

- 只返回当前项目。
- 合并返回绑定模型的 `model_id`、`display_name`、`provider`。
- 不返回 key、base URL、prompt、raw config。

`update_role_model_binding`：

- 需要二次确认。
- 校验 role 白名单。
- 校验 purpose 第一版只允许 `project_plan_generation`。
- 校验 `model_record_id` 当前项目、enabled=1、provider=openai_compat。
- 不调用模型。
- 不写 model_calls。
- 不写 runtime_events。
- 不创建 tasks / approvals / runner_requests。

## 调用链行为

阶段 36 第一版不要求一次项目计划生成时多模型并发调用。推荐先只把“角色绑定”作为配置落库和 UI 展示。

如果要在生成方案时使用绑定模型，必须单独进入后续阶段，例如阶段 39“按角色多模型草案生成”。不要在阶段 36 偷偷引入多次 provider 调用。

阶段 36 的安全目标是：

- 用户能看到每个角色准备使用哪个模型。
- 用户能手动修改绑定。
- 真实调用仍保持阶段 35 的单模型调用路径。

## 前端

在 `ProjectPlanPage` 或设置页新增“角色模型绑定”卡片：

- 表格列：
  - role
  - purpose
  - 当前模型 display_name / model_id
  - provider
  - enabled 状态
  - 操作：修改模型
- 修改模型使用 Select，下拉项来自 enabled 的 model_catalog。
- 修改需要二次确认。
- 不允许输入自由模型名。
- 不展示 key/base URL。
