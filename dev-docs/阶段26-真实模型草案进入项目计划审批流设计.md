# 阶段 26：真实模型草案保存为待审批项目计划草案

目标：把阶段 25.3 已经生成并审计落库的真实模型草案，手动保存为 `project_plan_drafts` + `approvals`，进入人工审批流。

这一阶段只让模型结果进入“待审批草案”，仍然不创建任务、不创建 Runner 队列、不执行命令、不写用户项目文件、不改 Git。

## 一、阶段边界

允许做：

- 用户在 `ProjectPlanPage` 生成真实模型草案后，手动点击“保存为项目计划草案”。
- 后端根据已写入 `model_calls` 的安全审计记录创建一条 `project_plan_drafts`。
- 同时创建一条 `approvals`，状态为 `pending`，等待人工审批。
- `project_plan_drafts.summary` 使用阶段 25.3 保存的脱敏、截断后的 `structured_summary`。
- `project_plan_drafts.generated_by` 使用新值，例如 `real_model_preview`。
- `project_plan_drafts.idea` 和 `constraints` 可以使用用户当前表单输入，但必须走现有长度校验和敏感值拦截。

禁止做：

- 不自动创建任务。
- 不自动批准审批。
- 不创建 `runner_requests`。
- 不写 `runtime_events`。
- 不执行 Runner。
- 不执行命令。
- 不写用户项目文件。
- 不改 Git。
- 不读取、返回或保存 raw key / raw base URL。
- 不保存 raw prompt / raw provider request / raw provider response / raw provider error。
- 不让前端传 key / base URL / provider / model / prompt / headers / provider body。
- 不新增前端自由模型配置入口。
- 不触碰保护路径：`design/image2/`、`_internal/`、`data/mock/runtime-state.json`、`data/local/`、`logs/`、`.playwright-cli/`。

## 二、为什么阶段 26 不直接创建任务

阶段 25 已经打开真实模型调用，阶段 25.3 已经补上 `model_calls` 安全审计。下一步最稳的是把模型摘要纳入现有审批体系，让人先看见、保存、再决定是否进入后续流程。

如果现在直接让模型结果创建任务或 Runner 队列，风险会同时覆盖模型输出质量、任务拆分、审批、执行计划和 Runner 边界，审查面过大。

因此阶段 26 的唯一新增能力是：

```text
真实模型草案预览 -> 手动保存为待审批项目计划草案
```

不进入执行。

## 三、推荐交互

在 `ProjectPlanPage` 的“真实模型草案预览”区域增加按钮：

```text
保存为项目计划草案
```

按钮启用条件：

- 当前是 Tauri 桌面宿主，可写。
- 已有真实模型草案结果。
- `modelDraftResult.status === "draft_ready"`。
- `modelDraftResult.summary` 非空。
- `modelDraftResult.audit_record_id` 非空。
- 项目想法输入框非空。
- 不在保存中。

点击后调用新的 Tauri command，例如：

```text
save_project_plan_model_draft
```

成功后：

- 显示“真实模型草案已保存为待审批项目计划草案”。
- 刷新 `project_plan_drafts` 列表。
- 刷新 overview / approvals。
- 不清空用户输入也可以；如果清空，要保持和现有本地草案入口风格一致。

失败后：

- 前端只显示粗粒度错误文案。
- 不显示 SQLite 原始错误。
- 不显示 `model_calls` 原始字段。
- 不显示 raw prompt / raw response。

## 四、后端建议实现

```rust
pub struct SaveProjectPlanModelDraftInput {

pub fn save_project_plan_model_draft(
```

Tauri command 建议放在：


## 五、保存前必须校验

后端必须校验：

1. `idea` 非空，长度沿用现有项目计划草案限制。
2. `constraints` 长度沿用现有项目计划草案限制。
3. `idea` / `constraints` 继续走敏感值拦截，禁止夹带 `sk-...`、`Authorization: Bearer ...`、`api_key=`、`token=`、`password=`。
4. `second_confirm == true`。
5. `confirm_text` 必须等于：

```text
我确认保存真实模型草案
```

6. `audit_record_id` 必须非空。
7. `audit_record_id` 必须存在于 `model_calls`。
8. 该 `model_calls` 记录必须满足：
   - `project_id` 等于当前项目 ID。
   - `purpose = project_plan_generation`。
   - `provider = openai_compat`。
   - `model = gpt-5.4-mini`。
   - `status = success`。
   - `structured_summary` 非空。
   - `error_category IS NULL`。
9. 如果已经有 `project_plan_drafts` 关联了同一个 `audit_record_id`，应该幂等返回已有草案；如果当前表没有字段保存该关联，见下一节。

## 六、是否需要改表

推荐新增 migration：

```text
005_add_project_plan_model_audit_link.sql
```

给 `project_plan_drafts` 增加可选字段：

```sql
ALTER TABLE project_plan_drafts ADD COLUMN model_call_id TEXT;
CREATE INDEX IF NOT EXISTS idx_project_plan_drafts_model_call_id ON project_plan_drafts(model_call_id);
```

原因：

- 阶段 26 需要知道某个待审批项目计划草案来自哪一次真实模型调用。
- 可以防止重复保存同一个真实模型结果。
- 不需要把 raw prompt / raw response 放进 `project_plan_drafts`。

注意：

- SQLite `ALTER TABLE ADD COLUMN` 不能重复执行。migration runner 如果没有“只执行一次”机制，要先检查现有 migration 写法，不要盲目加会重复失败的 SQL。
- 如果当前 migration runner 只是 `execute_batch` 每次启动都跑，不能直接使用无保护的 `ALTER TABLE`。必须先确认 `db/mod.rs` 的迁移机制。
- 如果现有机制不适合新增列，本阶段也可以先不加字段，但必须用其他安全方式保证不重复保存；若做不到，先提出问题，不要硬写。

如果新增字段可行，`project_plan_drafts.model_call_id` 只保存 `model_calls.id`，不保存任何模型原文。

## 七、写入事务

保存真实模型草案必须使用一个事务：

1. 读取当前项目 ID。
2. 校验 `model_calls` 记录属于当前项目且安全。
3. 检查是否已有同 `model_call_id` 草案。
4. 插入 `approvals` pending 记录。
5. 插入 `project_plan_drafts` draft 记录。
6. 提交事务。

事务内只允许写：

- `approvals`
- `project_plan_drafts`

事务内禁止写：

- `tasks`
- `runner_requests`
- `runtime_events`
- 用户项目目录文件
- Git

## 八、草案字段建议

`project_plan_drafts`：

- `id`: 可以用 `project_plan_model_{slug}` 或基于 `audit_record_id` 的安全 ID。
- `project_id`: 当前项目 ID。
- `approval_id`: 新建 approval ID。
- `idea`: 用户当前输入的 idea，经过长度校验和敏感值拦截。
- `constraints`: 用户当前输入的 constraints，经过校验。
- `summary`: 使用 `model_calls.structured_summary`。
- `status`: `draft`。
- `generated_by`: `real_model_preview`。
- `requested_by`: `local_user` 或现有规范值。
- `model_call_id`: 如果 migration 支持新增字段，则写入 `audit_record_id`。

`approvals`：

- `target_service`: `project_plan`。
- `status`: `pending`。
- `risk_level`: 建议 `medium`。
- `operation_types`: 复用现有项目计划审批 operation types。
- `reason`: 必须是简短脱敏文本，例如：

```text
请确认真实模型生成的项目计划草案。
```

不要把完整模型摘要塞进 `reason`。

## 九、返回结构

可以复用现有 `CreateProjectPlanDraftResponse`：

```text
draft
approval
planned_tasks
planned_runner_requests
side_effects
```

但注意：

- `planned_tasks` 可以继续作为内存预览，但不能落入 `tasks` 表。
- `planned_runner_requests` 可以继续作为内存预览，但不能落入 `runner_requests` 表。
- 如果担心误解，也可以阶段 26 返回空数组，并在 UI 上说明“保存草案不生成任务预览”。选择一种并写清楚测试。

`side_effects` 必须是：

```text
writes_project_files = false
modifies_git = false
executes_runner = false
calls_real_model = false
reads_raw_secrets = false
makes_network_requests = false
triggers_agents = false
creates_tasks = false
creates_runner_requests = false
```

注意：保存动作本身不再调用真实模型，所以 `calls_real_model=false`。真实模型调用已经发生在阶段 25 的预览按钮里。

## 十一、测试要求

Rust 必须补测试：

1. 保存成功：
   - 先插入一条安全 `model_calls` success 记录。
   - 调用 `save_project_plan_model_draft`。
   - 写入 1 条 `project_plan_drafts`。
   - 写入 1 条 `approvals` pending。
   - `summary` 等于 `model_calls.structured_summary`。
   - `generated_by = real_model_preview`。
   - 不写 `tasks`。
   - 不写 `runner_requests`。
   - 不写 `runtime_events`。

2. 二次确认失败：
   - 不写 `project_plan_drafts`。
   - 不写 `approvals`。

3. 确认文本错误：
   - 不写任何业务表。

4. `audit_record_id` 不存在：
   - 返回粗粒度错误。
   - 不写草案或审批。

5. `model_calls.status != success`：
   - 不允许保存。

6. `model_calls.structured_summary IS NULL` 或空白：
   - 不允许保存。

7. `model_calls.project_id` 不等于当前项目：
   - 不允许跨项目保存。

8. 敏感输入拦截：
   - idea / constraints 中包含 `sk-...`、`Authorization: Bearer ...`、`api_key=`、`token=`、`password=` 时拒绝。

9. 幂等：
   - 如果加了 `model_call_id` 字段，同一个 `audit_record_id` 重复保存应返回已有草案，不重复创建 approval。
   - 如果本阶段不加字段，必须明确不支持幂等，并说明为什么；但推荐实现幂等。

10. 不产生副作用：
    - 保存成功后 `tasks` 数量不变。
    - `runner_requests` 数量不变。
    - `runtime_events` 数量不变。

- `npm run typecheck`
- `npm run build`

- `cargo check`
